# 자율 E2E 테스트 파이프라인

## 1. 파이프라인의 목적

사용자 관점의 통합 테스트인 E2E(End-to-End) 테스트는 소프트웨어의 품질을 보증하는 가장 현실적인 검증 방법 중 하나입니다. 
그러나 전통적인 E2E 테스트는 테스트 코드 작성에 필요한 개발 역량과 높은 초기 구축 비용을 요구하며, 더 나아가 UI 변경에 따라 테스트 코드가 쉽게 실패하는 '깨지기 쉬운 테스트(Brittle Test)' 문제로 인해 유지보수 비용이 과도하게 발생하는 고질적인 문제를 안고 있습니다.

자율 E2E 테스트 파이프라인은 AI를 활용하여 이러한 문제를 해결하는 것을 목적으로 합니다. 
테스트 시나리오 정의를 위해 별도의 코드를 작성하는 대신 자연어를 사용하고, UI 변경에 강건한 '자동 복원(Self-Healing)' 메커니즘을 제공함으로써, E2E 테스트의 생성 및 유지보수 비용을 획기적으로 절감하고 테스트의 지속 가능성을 확보합니다.

---

## 2. 동작 흐름

```
[사용자 지시 (User Instruction)]
- 자연어 (Natural Language)
- 기획서 (Specification Document)
- 동작 녹화 (Action Recording)
      │
      ▼ (지시 → 변환)
[AI 에이전트]
- 지시문을 9가지 표준 동작으로 변환
- (페이지 이동, 클릭, 입력 등)
      │
      ▼ (변환 → 실행)
[테스트 브라우저 (Test Browser)]
- 변환된 동작 목록을 순서대로 수행
- 단계별 스크린샷 증적 저장
      │
      ▼ (실행 중 오류 발생 시)
[자동 적응 (Self-Healing)]
- 의미 기반으로 변경된 UI 요소 자동 탐색
- AI가 대체 요소를 제안 (필요시)
      │
      ▼
[테스트 결과 (Test Result)]
- 성공/실패 기록
- 적응 기록 포함
```

이 파이프라인은 **"시나리오 정의 → 동작 변환 → 자동 실행 → 동적 적응"** 의 4단계로 동작합니다.

### 단계 1: 테스트 시나리오 정의

사용자는 다음 세 가지 방식 중 하나로 테스트할 시나리오를 시스템에 전달합니다.

1.  **자연어 기반 정의:** "로그인 페이지에서 아이디와 비밀번호를 입력하고 로그인 버튼을 클릭합니다. 대시보드 화면이 나타나는지 확인합니다." 와 같이, 검증하고자 하는 사용자 시나리오를 순차적인 자연어 문장으로 기술합니다.
2.  **요구사항 문서 기반 정의:** 요구사항 정의서나 기능 명세서(PDF, Word 등)를 시스템에 업로드하면, AI가 문서의 내용을 분석하여 테스트 시나리오를 자동으로 추출합니다.
3.  **사용자 직접 시연 기반 정의:** 시스템이 제공하는 녹화용 브라우저에서 실제 사용자가 웹사이트를 조작하면, 해당 동작(클릭, 입력 등)이 순서대로 기록되어 시나리오로 활용됩니다.

### 단계 2: 표준화된 동작으로 변환

AI 에이전트는 사용자가 정의한 시나리오(자연어, 문서, 또는 녹화된 동작)를 분석하여, 브라우저 자동화 도구가 수행할 수 있는 9가지 유형의 표준 동작 목록으로 변환합니다. 
표준 동작은 페이지 이동, 클릭, 텍스트 입력, 키보드 입력, 드롭다운 선택, 체크박스 선택, 마우스 오버, 대기, 그리고 특정 요소의 존재 여부 확인으로 구성됩니다. 
모든 동작은 이 표준 집합 내에서만 생성되므로, 예측 불가능한 동작의 발생이 원천적으로 차단됩니다.

### 단계 3: 테스트 자동 실행 및 증적 기록

변환된 동작 목록에 따라, 시스템은 실제 브라우저를 구동하여 각 동작을 순서대로 수행합니다. 
각 동작의 수행 전후 화면은 스크린샷으로 자동 캡처되어, 테스트의 성공 또는 실패에 대한 명확한 시각적 증적으로 보존됩니다.

### 단계 4: 동적 적응 (Self-Healing)

테스트 실행 중, 기존에 정의된 UI 요소를 찾지 못할 경우(예: 버튼의 CSS 선택자 변경), 시스템은 테스트를 즉시 실패 처리하지 않고 자동으로 적응을 시도합니다.

- **1차 적응 (의미 기반 탐색):** 일반적인 테스트 도구가 사용하는 위치 기반(XPath, CSS Selector) 탐색과 달리, 본 시스템은 "로그인 버튼"과 같이 요소의 의미적 속성(접근성 역할, 텍스트, 라벨 등)을 기반으로 대상을 탐색합니다. 이 방식은 UI 구조나 디자인의 일부 변경에 대해 높은 수준의 강건성을 제공합니다.
- **2차 적응 (AI 기반 제안):** 의미 기반 탐색으로도 요소를 찾지 못하면, 현재 화면의 DOM 구조와 오류 정보를 AI에게 전달하여 가장 가능성이 높은 대체 요소를 제안받아 테스트를 계속 수행합니다.

이러한 동적 적응 매커니즘을 통해, 빈번한 UI 변경에도 기존 테스트 시나리오의 재사용성이 극대화되어 유지보수 부담이 크게 감소합니다.

## 3. 사용 방법

### 시나리오 유형별 실행 방법

- **자연어 기반 시나리오 실행**
  1.  Jenkins에서 `DSCORE-ZeroTouch-QA` 작업을 선택합니다.
  2.  테스트 대상 웹사이트 주소와 자연어로 기술된 시나리오를 파라미터로 입력하고 작업을 실행합니다.
  3.  실행 완료 후, 빌드 결과물(Artifacts)에서 단계별 스크린샷과 실행 로그를 확인합니다.

- **기획서 기반 시나리오 실행**
  1.  Jenkins에서 동일한 작업을 선택합니다.
  2.  테스트 시나리오 추출의 기반이 될 기획서 또는 명세서 파일을 업로드하고 작업을 실행합니다.

- **사용자 동작 녹화 기반 시나리오 생성**
  1.  제공된 별도의 스크립트를 터미널에서 실행하여 녹화용 브라우저를 엽니다.
  2.  평소와 같이 웹사이트를 사용한 후 브라우저를 닫으면, 실행한 동작이 시나리오 파일로 자동 저장됩니다.

---

## 4. 주요 기대 효과 및 활용 시나리오

본 파이프라인의 실행 결과로, 각 테스트 단계별 스크린샷, 성공/실패 여부가 포함된 실행 로그, 그리고 자동 적응이 발생한 경우 상세 내역이 포함된 산출물이 생성됩니다.

- **QA 팀의 회귀 테스트 자동화:** QA 담당자는 테스트 코드를 작성하는 대신, 주요 기능의 핵심 시나리오를 자연어로 관리하며 매 배포 시마다 자동으로 회귀 테스트를 수행할 수 있습니다.
- **기획 단계에서의 요구사항 검증:** 기획자나 PO는 작성된 요구사항 문서를 파이프라인에 직접 입력하여, 해당 기능 명세가 E2E 테스트로써 동작하는지 사전에 검증할 수 있습니다.
- **개발 단계에서의 스모크 테스트:** 개발자는 신규 배포 전, 로그인, 검색, 결제 등 시스템의 핵심 기능에 대한 시나리오를 실행하여 배포의 안정성을 신속하게 확인할 수 있습니다.

---

## 5. 참고 사항

- 본 파이프라인은 로컬 환경의 실제 브라우저에서 실행되며, 테스트 진행 과정을 화면에서 직접 관찰할 수 있습니다.
- 테스트 시나리오의 자연어 지시문이 구체적이고 명확할수록 AI가 생성하는 자동화 동작의 정확도가 향상됩니다.
- 모든 데이터 처리는 로컬 서버에서 이루어지므로, 테스트 대상 웹사이트의 정보가 외부에 노출되지 않습니다.
