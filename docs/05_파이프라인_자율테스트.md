# 자율 E2E 테스트 파이프라인 (Job #6)

## 1. 개요

자율 E2E 테스트 파이프라인(Zero-Touch QA)은 자연어 요구사항 또는 기획서만 입력하면 AI가 브라우저 E2E 테스트를 자동으로 생성하고 실행하는 기능입니다.

테스트 코드를 직접 작성할 필요가 없으며, 대상 웹사이트의 화면이 변경되더라도 시스템이 자동으로 적응하여 테스트를 계속 수행합니다.

---

## 2. 기능 설명

### 2.1. 핵심 구조

이 시스템은 지능 계층과 실행 계층이 분리되어 있습니다.

| 계층 | 역할 | 담당 |
| --- | --- | --- |
| 지능 계층 (Dify Brain) | 사용자의 지시를 분석하여 테스트 시나리오를 생성합니다. 실행 중 오류가 발생하면 원인을 판단하고 복구 방안을 제시합니다. | Dify Chatflow (Planner/Healer LLM) |
| 실행 계층 (Python Executor) | 생성된 시나리오에 따라 실제 브라우저를 조작합니다. 요소를 찾지 못할 경우 자체적인 탐색 로직과 로컬 복구를 먼저 시도합니다. | mac_local_executor.py + Playwright |

이 분리 덕분에 지능 계층은 Dify 프롬프트 수정만으로 업그레이드할 수 있고, 실행 계층은 코드 수정 없이도 대부분의 UI 변경에 자동 적응합니다.

### 2.2. 9대 표준 액션

AI와 실행 엔진은 다음 9가지 약속된 동작으로만 소통합니다. AI가 이 범위를 벗어나는 동작을 생성하면 실행 엔진이 즉시 오류를 발생시켜 예측 불가능한 동작을 방지합니다.

| 동작 | 설명 |
| --- | --- |
| navigate | 지정된 URL로 페이지를 이동합니다 |
| click | 화면의 요소를 클릭합니다 |
| fill | 입력란에 텍스트를 입력합니다 |
| press | 키보드 키를 누릅니다 (예: Enter, Tab) |
| select | 드롭다운 목록에서 항목을 선택합니다 |
| check | 체크박스를 체크하거나 해제합니다 |
| hover | 요소 위에 마우스를 올립니다 |
| wait | 지정된 시간 동안 대기합니다 |
| verify | 특정 요소가 존재하거나 텍스트가 표시되는지 확인합니다 |

### 2.3. 7단계 시맨틱 탐색

일반적인 테스트 도구는 HTML 구조상의 위치(CSS 셀렉터)로 요소를 찾기 때문에, 디자인이 변경되면 테스트가 실패합니다. 이 시스템은 요소의 의미(역할, 텍스트, 라벨 등)를 기준으로 7단계에 걸쳐 탐색하며, 상위 단계일수록 UI 변경에 강합니다.

| 단계 | 탐색 기준 | UI 변경 내성 |
| --- | --- | --- |
| 1 | 접근성 역할과 이름 (예: "로그인 버튼") | 가장 강함 |
| 2 | 화면에 표시된 텍스트 | 강함 |
| 3 | 입력란에 연결된 라벨 | 강함 |
| 4 | 입력란의 힌트 텍스트(placeholder) | 보통 |
| 5 | 개발팀이 부여한 테스트 ID 속성 | 보통 |
| 6 | CSS 셀렉터 또는 XPath | 약함 |
| 7 | 위 6단계 모두 실패 시 자가 치유로 전환 | — |

### 2.4. 하이브리드 자가 치유 (Self-Healing)

7단계 탐색이 모두 실패하면 두 단계의 자동 복구가 작동합니다.

| 단계 | 방식 | 설명 |
| --- | --- | --- |
| 1단계 | Candidate Search (로컬) | 현재 페이지의 요소를 스캔하여, 실패한 대상과 가장 유사한 요소를 자동으로 찾습니다. AI 호출 없이 즉시 수행됩니다. |
| 2단계 | Dify Healer LLM | 1단계 실패 시, 페이지 구조와 오류 정보를 AI에 전송하여 대체 요소를 제안받습니다. |

로컬 복구를 우선 시도하므로 대부분의 경미한 UI 변경은 AI 호출 없이 즉시 복구됩니다.

## 3. 테스트 생성 방법

테스트는 세 가지 방법으로 생성할 수 있습니다.

### 3.1. Chat 모드: 자연어로 지시

자연어로 테스트 요구사항을 작성하면 AI가 즉시 실행 가능한 시나리오로 변환합니다.

지시문은 단계별로 번호를 매기고, 각 단계에 하나의 동작만 포함하며, 마지막에 확인 조건을 명시하는 것이 효과적입니다.

지시문 예시:

```text
1. 네이버 메인 페이지에 접속합니다.
2. 검색창에 'DSCORE'를 입력합니다.
3. 엔터 키를 누릅니다.
4. 검색 결과 페이지가 표시되는지 확인합니다.
```

Jenkins에서 `DSCORE-ZeroTouch-QA` Job을 선택한 후, `RUN_MODE`를 `chat`으로 설정하고 `SRS_TEXT`에 위와 같은 지시문을 입력하면 됩니다. `TARGET_URL`에는 테스트 대상 웹사이트의 주소를 입력합니다.

### 3.2. Doc 모드: 기획서 업로드

기획서(PDF/Word)를 업로드하면 AI가 문서를 분석하여 테스트 시나리오를 자동 생성합니다.

Jenkins에서 `RUN_MODE`를 `doc`으로 설정하고, `DOC_FILE` 파라미터에 기획서 파일을 업로드합니다.

이 기능은 Jenkins에 File Parameter 플러그인이 설치되어 있어야 합니다. 플러그인이 없는 경우, 기획서의 핵심 내용을 `SRS_TEXT`에 직접 입력하여 Chat 모드로 대체할 수 있습니다.

### 3.3. Record 모드: 브라우저 녹화

브라우저를 직접 조작하면 시스템이 클릭, 텍스트 입력, 드롭다운 선택 등의 동작을 자동으로 캡처하여 테스트 시나리오로 변환합니다. 조작 시 해당 요소에 붉은 테두리가 표시되어 어떤 동작이 기록되고 있는지 실시간으로 확인할 수 있습니다.

이 모드는 맥북 터미널에서 직접 실행합니다.

```text
python3 mac_local_executor.py --mode record --url https://대상사이트주소
```

브라우저 창을 닫으면 녹화가 종료되고, 캡처된 데이터가 AI를 통해 정제된 시나리오로 변환됩니다.

## 4. 실행 결과

실행이 완료되면 Jenkins 빌드 페이지의 `Artifacts`에서 다음 산출물을 확인할 수 있습니다.

| 산출물 | 설명 |
| --- | --- |
| scenario.json | AI가 생성한 원본 테스트 시나리오입니다 |
| scenario.healed.json | 자가 치유가 적용된 최종 시나리오입니다. 다음 실행 시 기준 시나리오로 재활용할 수 있습니다 |
| run_log.jsonl | 각 단계의 성공(PASS), 적응 성공(HEALED), 실패(FAIL) 기록입니다 |
| step_N_pass.png | 각 단계 성공 시 캡처된 브라우저 스크린샷입니다 |
| error_final.png | 최종 실패 시 캡처된 브라우저 화면입니다 |

자가 치유로 성공한 단계(HEALED)가 있는 경우, `scenario.healed.json`을 다운로드하여 다음 실행의 기준 시나리오로 활용하면 불필요한 적응 과정을 생략하여 실행 속도를 높일 수 있습니다.

---

## 5. 활용 범위

| 활용 시나리오 | 설명 |
| --- | --- |
| 코드를 작성하지 않는 회귀 테스트 | QA 담당자가 자연어로 주요 시나리오를 정의하여 정기적으로 실행할 수 있습니다 |
| 기획서 기반 기능 검증 | 기획자가 요구사항 문서를 업로드하면 해당 기능의 E2E 테스트가 자동 생성됩니다 |
| 배포 전 스모크 테스트 | 핵심 사용자 시나리오(로그인, 검색, 결제 등)를 자연어로 정의하여 배포 전 자동 검증합니다 |
| 복잡한 UI 흐름의 자산화 | Record 모드로 복잡한 조작을 녹화하여 재사용 가능한 테스트 시나리오로 보존합니다 |

---

## 6. 참고 사항

- 이 파이프라인은 Mac Local Agent에서 실행되며, 실제 브라우저가 화면에 표시된 상태로 테스트가 진행됩니다.
- 지시문은 구체적일수록 정확한 시나리오가 생성됩니다. "테스트해줘"보다 "검색창에 DSCORE를 입력하고 엔터를 누릅니다"가 효과적입니다.
- 모든 데이터 처리는 로컬 서버에서 이루어지며, 외부로 전송되지 않습니다.
