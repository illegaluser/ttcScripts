# 자율 E2E 테스트 파이프라인 사용 가이드 (Job #6)

---

## 1. 이 기능으로 할 수 있는 것

자율 테스트 시스템은 테스트 코드를 직접 작성하지 않아도, 사람이 일상적으로 사용하는 말로 지시하면 컴퓨터가 알아서 웹 브라우저를 조작하고 그 결과를 보고해 주는 기능입니다.

테스트 코드를 작성할 줄 몰라도, 자연어로 지시하면 시스템이 알아서 브라우저를 조작하고 결과를 보고합니다. 개발 지식이 없는 기획자, 디자이너, 품질 담당자도 바로 사용할 수 있습니다.

아래 세 가지 대표적인 사용 시나리오를 통해 자율 테스트 시스템이 어떤 일을 하는지 확인하십시오.

---

### 시나리오 1: 자연어로 지시하여 즉시 테스트하기

회사의 웹사이트에 검색 기능이 올바르게 동작하는지 확인하고 싶다고 가정합니다. 이때 다음과 같이 지시하면 됩니다.

```
1. 검색창에 'DSCORE'를 입력합니다.
2. 엔터 키를 누릅니다.
3. 검색 결과 목록이 화면에 표시되는지 확인합니다.
```

이 지시문을 시스템에 전달하면, 시스템이 자동으로 브라우저를 열고 검색창을 찾아 'DSCORE'를 입력한 뒤 엔터 키를 누릅니다. 그리고 검색 결과가 화면에 나타나는지 확인한 후, 각 단계의 화면 캡처와 성공 여부를 보고합니다.

사람이 직접 마우스와 키보드로 하는 작업을 시스템이 대신 수행하는 것입니다. 지시문을 작성하는 데 걸리는 시간은 1분 이내이며, 실행까지 포함해도 수 분 안에 결과를 확인할 수 있습니다.

---

### 시나리오 2: 기획서 파일을 업로드하여 대량 테스트 생성하기

신규 기능에 대한 기획서가 이미 작성되어 있는 경우, 해당 파일을 시스템에 업로드하면 시스템이 기획서를 읽고 그 안에 담긴 요구사항을 분석하여 테스트 시나리오를 자동으로 만들어 냅니다.

예를 들어, 회원 가입 기능에 대한 기획서를 업로드하면 시스템은 다음과 같은 테스트 시나리오를 스스로 만들어 실행합니다.

- 정상적인 정보를 입력하여 가입이 완료되는지 확인
- 필수 항목을 비워 두었을 때 오류 메시지가 표시되는지 확인
- 이미 가입된 이메일로 가입을 시도했을 때 안내 메시지가 표시되는지 확인

기획서 한 장으로 여러 테스트 시나리오가 동시에 만들어지므로, 수동으로 하나씩 작성하는 것보다 훨씬 빠르고 누락이 적습니다.

---

### 시나리오 3: 브라우저에서 직접 시연하여 테스트 녹화하기

복잡한 화면 조작을 말로 설명하기 어려울 때, 직접 브라우저에서 동작을 시연하면 시스템이 해당 동작을 기록하여 재사용 가능한 테스트 시나리오로 변환합니다.

예를 들어, 쇼핑몰에서 상품을 장바구니에 담고 결제 페이지까지 이동하는 과정을 직접 시연하면, 시스템은 클릭한 위치, 입력한 텍스트, 선택한 항목을 모두 기록합니다. 시연이 끝나면 이 기록을 자동으로 정리하여 언제든 다시 실행할 수 있는 테스트 시나리오로 만들어 줍니다.

이렇게 만들어진 시나리오는 이후 동일한 기능에 변경이 발생할 때마다 반복 실행하여 문제가 없는지 확인하는 데 활용할 수 있습니다.

---

### 핵심 장점: 화면 변경에 자동으로 적응합니다

자율 테스트 시스템의 가장 큰 장점 중 하나는 화면 변경에 자동으로 적응하는 능력입니다.

일반적인 테스트 도구는 화면의 특정 위치나 이름에 의존하여 동작하기 때문에, 버튼의 위치가 바뀌거나 이름이 변경되면 테스트가 실패합니다. 그래서 화면이 조금만 바뀌어도 테스트를 처음부터 다시 만들어야 하는 번거로움이 있었습니다.

자율 테스트 시스템은 이 문제를 다음과 같이 해결합니다.

- 테스트 대상 웹사이트의 화면이 일부 변경되어도, 시스템이 변경된 부분을 자동으로 감지하고 적응하여 테스트를 계속 실행합니다.
- 버튼 이름이 바뀌거나 위치가 옮겨져도 시스템이 스스로 찾아갑니다.
- 예를 들어, '로그인' 버튼이 '로그인하기'로 이름이 바뀌더라도, 시스템은 해당 버튼을 자동으로 인식하고 테스트를 정상적으로 수행합니다.
- 이 기능 덕분에 화면 디자인이 변경될 때마다 테스트를 다시 만들 필요가 없습니다.

이러한 자동 적응 기능이 작동한 경우, 결과 보고서에 '적응 완료'라는 표시가 나타나므로 어느 부분이 변경되었는지도 파악할 수 있습니다.

---

## 2. 테스트를 시작하는 세 가지 방법

자율 테스트 시스템은 테스트를 시작하는 세 가지 방법을 제공합니다. 상황에 맞는 방법을 선택하십시오.

### 방법 1: 자연어로 지시하기 (Chat 모드)

테스트하고 싶은 내용을 단계별로 글로 작성하여 전달하는 방법입니다. 새로운 기능을 빠르게 확인하거나, 특정 동작이 올바르게 작동하는지 즉석에서 검증할 때 적합합니다. 별도의 문서나 파일 없이 바로 시작할 수 있어 가장 간편합니다.

### 방법 2: 기획서 파일로 지시하기 (Doc 모드)

이미 작성된 기획서나 요구사항 문서를 업로드하여 테스트를 생성하는 방법입니다. 기획 단계에서 작성한 문서를 그대로 활용할 수 있으므로, 별도의 테스트 지시문을 작성할 필요가 없습니다. 여러 기능에 대한 테스트를 한 번에 만들어야 할 때 효율적입니다.

### 방법 3: 직접 시연하여 녹화하기 (Record 모드)

브라우저에서 직접 동작을 수행하면 시스템이 해당 동작을 기록하여 테스트 시나리오로 변환하는 방법입니다. 복잡한 화면 조작을 글로 설명하기 어려울 때, 또는 기존 업무 절차를 그대로 테스트로 만들고 싶을 때 유용합니다.

## 3. 시작하기 전에 확인할 사항

테스트를 실행하기 전에 아래 네 가지 항목을 반드시 확인하십시오. 하나라도 준비되지 않으면 테스트가 정상적으로 실행되지 않습니다.

---

### 확인 1: Jenkins에 접속할 수 있는지 확인합니다

Jenkins는 테스트를 실행하고 결과를 확인하는 관리 화면입니다. 다음 절차로 접속 가능 여부를 확인하십시오.

1. 웹 브라우저를 열고 주소창에 Jenkins 주소를 입력합니다.
   - 일반적으로 `http://localhost:8080` 또는 팀에서 안내받은 주소입니다.
2. Jenkins 로그인 화면이 나타나면 아이디와 비밀번호를 입력하여 로그인합니다.
3. 대시보드 화면이 정상적으로 표시되면 접속에 성공한 것입니다.

접속이 되지 않는 경우 다음을 확인하십시오.

- Jenkins 서버가 실행 중인지 확인합니다. 서버 관리 담당자에게 문의하십시오.
- 브라우저에서 주소를 정확하게 입력했는지 확인합니다.
- 같은 네트워크(사내망)에 연결되어 있는지 확인합니다.

---

### 확인 2: 테스트 전용 에이전트가 연결되어 있는지 확인합니다

자율 테스트 시스템은 테스트를 실행하기 위한 전용 컴퓨터(에이전트)가 필요합니다. 이 에이전트가 Jenkins에 정상적으로 연결되어 있어야 합니다.

1. Jenkins에 로그인한 후, 상단 메뉴에서 'Jenkins 관리'를 클릭합니다.
2. 'Nodes' 또는 '노드 관리' 항목을 클릭합니다.
3. 목록에서 'mac-local-agent'를 찾습니다.
4. 해당 에이전트의 상태가 'Connected'(연결됨)로 표시되어 있는지 확인합니다.

상태가 'Connected'가 아닌 경우 다음을 확인하십시오.

- 에이전트 역할을 하는 컴퓨터가 켜져 있는지 확인합니다.
- 해당 컴퓨터에서 에이전트 프로그램이 실행 중인지 확인합니다.
- 네트워크 연결에 문제가 없는지 확인합니다.
- 위 항목을 모두 확인해도 연결되지 않으면 시스템 관리 담당자에게 문의하십시오.

---

### 확인 3: 테스트 생성 서비스가 가동 중인지 확인합니다

자율 테스트 시스템이 테스트 시나리오를 자동으로 생성하려면, 테스트 생성 서비스가 가동 중이어야 합니다. 다음 절차로 확인하십시오.

1. 시스템 관리 담당자에게 테스트 생성 서비스의 상태를 문의합니다.
2. 또는 팀에서 안내받은 서비스 상태 확인 페이지에 접속하여 '정상 가동' 상태인지 확인합니다.

서비스가 가동 중이 아닌 경우 다음을 확인하십시오.

- 서비스가 실행되는 서버가 정상 동작 중인지 확인합니다.
- 서비스에 필요한 인증 정보가 올바르게 설정되어 있는지 확인합니다.
- 시스템 관리 담당자에게 서비스 재시작을 요청합니다.

---

### 확인 4: 테스트 대상 웹사이트에 접속할 수 있는지 확인합니다

테스트를 실행하려면 대상 웹사이트에 접속이 가능해야 합니다.

1. 웹 브라우저를 열고 테스트 대상 웹사이트의 주소를 입력합니다.
2. 웹사이트가 정상적으로 화면에 표시되는지 확인합니다.
3. 로그인이 필요한 페이지를 테스트할 경우, 테스트용 계정으로 로그인이 가능한지도 확인합니다.

접속이 되지 않는 경우 다음을 확인하십시오.

- 웹사이트 주소가 올바른지 확인합니다.
- 해당 웹사이트 서버가 정상 동작 중인지 확인합니다.
- 사내망에서만 접속 가능한 경우, 현재 사내망에 연결되어 있는지 확인합니다.
- 웹사이트 관리 담당자에게 서비스 상태를 문의합니다.

---

### 사전 확인 체크리스트 요약

| 확인 항목 | 확인 방법 | 정상 상태 |
| --- | --- | --- |
| Jenkins 접속 | 브라우저에서 Jenkins 주소 입력 | 대시보드 화면이 표시됩니다 |
| 테스트 전용 에이전트 연결 | Jenkins 관리 화면의 Nodes 목록 확인 | mac-local-agent가 Connected 상태입니다 |
| 테스트 생성 서비스 가동 | 서비스 상태 확인 페이지 또는 담당자 문의 | 정상 가동 상태입니다 |
| 테스트 대상 웹사이트 접속 | 브라우저에서 대상 주소 입력 | 화면이 정상적으로 표시됩니다 |

위 네 가지 항목이 모두 확인되면 테스트를 시작할 준비가 완료된 것입니다.

## 4. 방법 1: 자연어로 테스트 지시하기 (Chat 모드)

### 4.1. 언제 이 방법을 사용합니까

Chat 모드는 다음과 같은 상황에서 사용하기에 가장 적합합니다.

**신규 기능 빠른 검증**

새로 개발된 기능이 기본적으로 동작하는지 빠르게 확인하고 싶을 때 사용합니다. 예를 들어, 방금 배포된 로그인 기능이 정상적으로 작동하는지 30초 만에 지시문을 작성하여 확인할 수 있습니다.

**특정 문제 재현**

사용자가 보고한 특정 문제를 동일하게 재현해 보고 싶을 때 사용합니다. 문제가 발생한 절차를 그대로 지시문으로 작성하면, 시스템이 해당 절차를 그대로 수행하여 문제가 실제로 재현되는지 확인합니다.

**즉석 검증**

회의 중에 "이 기능이 지금 정상인가요?"라는 질문이 나왔을 때, 그 자리에서 바로 지시문을 작성하여 확인할 수 있습니다. 별도의 준비 없이 즉시 시작할 수 있다는 것이 Chat 모드의 가장 큰 장점입니다.

**단순 반복 확인**

매일 아침 주요 페이지가 정상적으로 열리는지, 로그인이 되는지 등 간단한 확인을 반복적으로 수행할 때 사용합니다.

---

### 4.2. 테스트 지시문 작성 요령

테스트 지시문은 자율 테스트 시스템에 "무엇을 해 달라"고 전달하는 글입니다. 지시문을 잘 작성하면 시스템이 정확하게 동작하고, 모호하게 작성하면 엉뚱한 결과가 나올 수 있습니다. 아래 요령을 참고하여 지시문을 작성하십시오.

---

#### 좋은 지시문의 네 가지 원칙

**원칙 1: 번호를 매겨 순서대로 작성합니다**

시스템은 지시문의 번호 순서대로 동작을 수행합니다. 번호가 없으면 어떤 동작을 먼저 수행해야 하는지 혼란이 생길 수 있으므로, 반드시 번호를 매기십시오.

**원칙 2: 각 단계는 하나의 동작만 포함합니다**

하나의 단계에 여러 동작을 넣으면 시스템이 정확하게 해석하지 못할 수 있습니다. "아이디를 입력하고 로그인 버튼을 클릭한다"가 아니라, "아이디를 입력한다"와 "로그인 버튼을 클릭한다"를 별도의 단계로 분리하십시오.

**원칙 3: 대상을 구체적으로 지정합니다**

"입력란에 입력한다"보다 "'아이디' 입력란에 입력한다"가 더 정확합니다. 화면에 여러 입력란이나 버튼이 있을 때, 어느 것을 지칭하는지 명확하게 지정해야 합니다.

**원칙 4: 마지막에 확인 조건을 포함합니다**

테스트의 목적은 "올바른 결과가 나오는지 확인하는 것"입니다. 지시문의 마지막에 "화면에 ~가 표시되는지 확인합니다"를 추가하면 시스템이 자동으로 결과를 검증합니다.

---

#### 좋은 예시 1: 네이버 검색 테스트 (간단한 테스트)

```
1. 네이버 메인 페이지(https://www.naver.com)에 접속합니다.
2. 검색창에 'DSCORE'를 입력합니다.
3. 엔터 키를 누릅니다.
4. 검색 결과 페이지에 'DSCORE'라는 텍스트가 표시되는지 확인합니다.
```

이 예시가 좋은 이유:
- 접속할 주소가 명확합니다.
- 각 단계가 하나의 동작만 포함합니다.
- 입력할 텍스트가 작은따옴표로 감싸져 있어 명확합니다.
- 마지막에 확인 조건이 있습니다.

---

#### 좋은 예시 2: 로그인 테스트 (양식 입력 테스트)

```
1. 로그인 페이지(https://my-app.com/login)에 접속합니다.
2. '이메일' 입력란에 'testuser@example.com'을 입력합니다.
3. '비밀번호' 입력란에 'Test1234!'를 입력합니다.
4. '로그인' 버튼을 클릭합니다.
5. 화면에 '환영합니다'라는 텍스트가 표시되는지 확인합니다.
```

이 예시가 좋은 이유:
- 각 입력란의 이름이 구체적으로 지정되어 있습니다 ('이메일' 입력란, '비밀번호' 입력란).
- 입력할 값이 정확하게 명시되어 있습니다.
- 클릭할 버튼이 명확합니다 ('로그인' 버튼).
- 성공 기준이 분명합니다 ('환영합니다'라는 텍스트 표시).

---

#### 좋은 예시 3: 쇼핑몰 장바구니 테스트 (여러 단계를 거치는 테스트)

```
1. 쇼핑몰 메인 페이지(https://shop.example.com)에 접속합니다.
2. '신상품' 메뉴를 클릭합니다.
3. 2초 대기합니다.
4. 첫 번째 상품의 이름을 클릭합니다.
5. 2초 대기합니다.
6. '장바구니 담기' 버튼을 클릭합니다.
7. 2초 대기합니다.
8. '장바구니로 이동' 버튼을 클릭합니다.
9. 장바구니 페이지에 방금 선택한 상품이 표시되는지 확인합니다.
10. '총 금액'이 0원이 아닌지 확인합니다.
```

이 예시가 좋은 이유:
- 페이지가 전환되는 시점마다 '2초 대기합니다'를 추가하여 화면이 완전히 로딩될 시간을 확보합니다.
- 여러 페이지를 순서대로 이동하는 흐름이 명확합니다.
- 확인 조건이 두 가지여서 더 꼼꼼한 검증이 이루어집니다.

---

#### 좋은 예시 4: 게시판 글쓰기 테스트 (복잡한 양식 테스트)

```
1. 게시판 페이지(https://my-app.com/board)에 접속합니다.
2. '글쓰기' 버튼을 클릭합니다.
3. 2초 대기합니다.
4. '제목' 입력란에 '테스트 게시물입니다'를 입력합니다.
5. '카테고리' 선택 목록에서 '공지사항'을 선택합니다.
6. '내용' 입력란에 '자율 테스트 시스템으로 작성한 테스트 게시물입니다. 이 게시물은 테스트 완료 후 삭제될 예정입니다.'를 입력합니다.
7. '공개 여부'의 '공개' 항목을 선택합니다.
8. '등록' 버튼을 클릭합니다.
9. 3초 대기합니다.
10. 화면에 '게시물이 등록되었습니다'라는 텍스트가 표시되는지 확인합니다.
11. 게시물 목록에 '테스트 게시물입니다'라는 제목이 표시되는지 확인합니다.
```

이 예시가 좋은 이유:
- 입력란, 선택 목록, 선택 항목 등 다양한 유형의 양식 요소를 다루고 있습니다.
- 각 양식 요소의 이름이 구체적입니다.
- 등록 후 충분한 대기 시간을 확보합니다.
- 확인 조건이 구체적이고 두 가지입니다.

---

#### 좋은 예시 5: 드롭다운 선택과 체크박스 테스트

```
1. 회원 정보 수정 페이지(https://my-app.com/profile/edit)에 접속합니다.
2. '국가' 선택 목록에서 '대한민국'을 선택합니다.
3. '도시' 선택 목록에서 '서울'을 선택합니다.
4. '마케팅 수신 동의' 체크박스를 체크합니다.
5. '이메일 수신' 체크박스를 체크합니다.
6. '저장' 버튼을 클릭합니다.
7. 2초 대기합니다.
8. 화면에 '저장되었습니다'라는 텍스트가 표시되는지 확인합니다.
```

이 예시가 좋은 이유:
- 선택 목록에서 선택할 항목이 정확하게 지정되어 있습니다.
- 체크박스를 '체크합니다'라고 명확하게 기술하고 있습니다.
- 각 양식 요소의 이름이 화면에 표시되는 레이블과 일치합니다.

---

#### 나쁜 예시 1: "로그인 테스트 해줘"

```
로그인 테스트 해줘
```

이 지시문이 좋지 않은 이유:
- 어떤 웹사이트에 접속해야 하는지 알 수 없습니다.
- 어떤 아이디와 비밀번호를 입력해야 하는지 모릅니다.
- 어떤 결과가 나와야 성공인지 기준이 없습니다.
- 시스템이 임의로 추측하여 동작하게 되므로 의도와 다른 결과가 나올 수 있습니다.

---

#### 나쁜 예시 2: "사이트가 잘 동작하는지 확인"

```
사이트가 잘 동작하는지 확인해 주세요
```

이 지시문이 좋지 않은 이유:
- '잘 동작한다'는 기준이 모호합니다.
- 어떤 페이지를, 어떤 순서로, 어떻게 확인해야 하는지 전혀 알 수 없습니다.
- 구체적인 동작 단계가 없으므로 시스템이 실행할 수 없습니다.

---

#### 나쁜 예시 3: "모든 페이지를 테스트해줘"

```
모든 페이지를 테스트해 주세요
```

이 지시문이 좋지 않은 이유:
- 웹사이트의 모든 페이지를 한 번에 테스트하는 것은 범위가 너무 넓습니다.
- 각 페이지에서 어떤 동작을 수행하고 어떤 결과를 확인해야 하는지 알 수 없습니다.
- 이런 경우에는 기능 단위로 지시문을 나누어 여러 번 실행하는 것이 효과적입니다.

---

#### 지시문 작성 팁 모음

다음 팁을 참고하면 더 정확한 테스트 결과를 얻을 수 있습니다.

| 팁 | 설명 | 예시 |
| --- | --- | --- |
| 입력할 텍스트는 작은따옴표로 감쌉니다 | 시스템이 입력할 텍스트를 정확하게 인식합니다 | 검색창에 'DSCORE'를 입력합니다 |
| 동작을 명확하게 기술합니다 | '~를 클릭합니다', '~를 입력합니다' 형태로 작성합니다 | '로그인' 버튼을 클릭합니다 |
| 대기가 필요한 경우 명시합니다 | 페이지 전환이나 데이터 로딩 시 대기 시간을 추가합니다 | 2초 대기합니다 |
| 확인 조건을 명확하게 작성합니다 | '~가 화면에 표시되는지 확인합니다' 형태로 작성합니다 | '검색 결과'가 화면에 표시되는지 확인합니다 |
| 한 단계에 하나의 동작만 넣습니다 | 여러 동작을 하나의 단계에 넣지 않습니다 | (O) 1. 아이디를 입력합니다 / 2. 로그인을 클릭합니다 |
| 웹사이트 주소를 정확하게 입력합니다 | 접속할 페이지의 전체 주소를 포함합니다 | https://www.example.com/login |
| 대상 요소의 이름을 화면에 보이는 그대로 씁니다 | 화면에 '회원가입'이라고 표시되어 있으면 그대로 사용합니다 | '회원가입' 버튼을 클릭합니다 |
| 긴 텍스트 입력도 가능합니다 | 게시판 본문 등 긴 텍스트도 그대로 작성합니다 | '내용' 입력란에 '안녕하세요...'를 입력합니다 |

### 4.3. Jenkins에서 테스트를 실행합니다

아래 절차를 순서대로 따라하십시오.

---

**1단계: Jenkins에 접속합니다**

웹 브라우저를 열고 Jenkins 주소를 입력하여 로그인합니다. 로그인에 성공하면 대시보드 화면이 표시됩니다.

---

**2단계: 테스트 실행 항목을 찾습니다**

대시보드에 표시된 목록에서 'DSCORE-ZeroTouch-QA'를 찾아 클릭합니다. 이 항목이 자율 테스트 시스템의 실행 항목입니다.

만약 목록에서 찾기 어려운 경우, 대시보드 상단의 검색 기능을 이용하여 'ZeroTouch'를 검색하면 빠르게 찾을 수 있습니다.

---

**3단계: 매개변수 입력 화면으로 이동합니다**

화면 좌측 메뉴에서 'Build with Parameters' (매개변수와 함께 빌드)를 클릭합니다. 테스트 실행에 필요한 정보를 입력하는 화면이 나타납니다.

---

**4단계: 실행 방식을 선택합니다**

'RUN_MODE' 항목의 드롭다운 목록에서 'chat'을 선택합니다. 이것은 자연어 지시 방식으로 테스트를 실행하겠다는 뜻입니다.

---

**5단계: 테스트 대상 웹사이트 주소를 입력합니다**

'TARGET_URL' 항목에 테스트할 웹사이트의 주소를 입력합니다.

예시:
- `https://www.naver.com`
- `https://my-app.com/login`
- `https://shop.example.com`

주소를 입력할 때 'https://'를 포함하여 전체 주소를 입력하십시오.

---

**6단계: 테스트 지시문을 입력합니다**

'SRS_TEXT' 항목에 앞서 작성한 테스트 지시문을 붙여넣습니다. 4.2절에서 안내한 작성 요령에 따라 번호를 매기고, 각 단계를 구체적으로 작성한 지시문을 입력하십시오.

예시:

```
1. 메인 페이지에 접속합니다.
2. 검색창에 'DSCORE'를 입력합니다.
3. 엔터 키를 누릅니다.
4. 검색 결과가 화면에 표시되는지 확인합니다.
```

---

**7단계: 테스트를 실행합니다**

모든 정보를 입력한 후 화면 하단의 '빌드' 버튼을 클릭합니다. 테스트 실행이 시작됩니다.

---

**8단계: 실행 중인 상태를 확인합니다**

'빌드' 버튼을 클릭하면 화면 좌측 하단의 '빌드 히스토리' 영역에 새로운 빌드 항목이 나타납니다. 이 항목 옆에 표시되는 아이콘의 의미는 다음과 같습니다.

| 아이콘 상태 | 의미 |
| --- | --- |
| 파란색 원이 깜빡이는 상태 | 테스트가 현재 실행 중입니다. 완료될 때까지 기다리십시오. |
| 파란색 원 (정지) | 테스트가 성공적으로 완료되었습니다. |
| 빨간색 원 | 테스트 실행 중 문제가 발생하여 실패했습니다. |
| 회색 원 | 테스트가 취소되었거나 실행되지 않았습니다. |

---

**테스트 실행 중에 일어나는 일:**

테스트가 실행되면 에이전트 컴퓨터에서 다음과 같은 일이 자동으로 일어납니다.

1. 시스템이 입력한 지시문을 분석하여 테스트 시나리오를 생성합니다.
2. 브라우저가 자동으로 열리고 지정한 웹사이트에 접속합니다.
3. 지시문에 작성한 순서대로 클릭, 텍스트 입력, 키보드 누르기 등의 동작을 수행합니다.
4. 각 단계를 수행할 때마다 화면을 캡처합니다.
5. 마지막으로 확인 조건을 검증하고 결과를 기록합니다.
6. 모든 단계가 완료되면 브라우저가 닫히고 결과 파일이 생성됩니다.

---

**소요 시간 안내:**

일반적인 테스트의 소요 시간은 다음과 같습니다.

| 테스트 규모 | 예상 소요 시간 |
| --- | --- |
| 간단한 테스트 (3~5단계) | 약 1~2분 |
| 보통 테스트 (6~10단계) | 약 2~4분 |
| 복잡한 테스트 (11~20단계) | 약 4~8분 |

시나리오 생성에 걸리는 시간이 포함되어 있으므로, 실제 브라우저 조작 시간보다 전체 소요 시간이 길 수 있습니다.

---

### 4.4. 테스트 결과를 확인합니다

테스트가 완료되면 Jenkins에서 결과를 확인할 수 있습니다. 아래 절차를 따라 결과를 확인하십시오.

---

**1단계: 빌드 결과 페이지로 이동합니다**

좌측의 '빌드 히스토리'에서 방금 실행한 빌드 번호(예: #15)를 클릭합니다. 해당 빌드의 상세 정보 페이지가 표시됩니다.

---

**2단계: 결과 파일 목록을 확인합니다**

빌드 상세 정보 페이지에서 '산출물' 또는 'Artifacts' 링크를 클릭합니다. 테스트 실행 중에 생성된 결과 파일 목록이 표시됩니다.

---

**3단계: 화면 캡처 이미지를 확인합니다**

결과 파일 목록에서 'step_1_pass.png', 'step_2_pass.png'와 같은 이미지 파일을 클릭하면, 각 단계에서 캡처된 실제 화면을 확인할 수 있습니다.

파일 이름의 의미는 다음과 같습니다.

| 파일 이름 형식 | 의미 |
| --- | --- |
| step_N_pass.png | N번째 단계가 성공했을 때의 화면입니다 |
| step_N_healed.png | N번째 단계에서 화면 변경을 감지하고 자동 적응한 후 성공했을 때의 화면입니다 |
| step_N_fail.png | N번째 단계에서 실패했을 때의 화면입니다 |
| error_final.png | 최종 실패 시점의 화면입니다 |

각 이미지를 클릭하여 열어 보면, 시스템이 해당 단계에서 실제로 어떤 화면을 보고 있었는지 확인할 수 있습니다. 예상한 화면과 다른 경우, 지시문을 수정하거나 웹사이트의 상태를 확인해야 합니다.

---

**4단계: 실행 기록을 확인합니다**

결과 파일 목록에서 실행 기록 파일(run_log)을 클릭하면 각 단계의 상세한 실행 결과를 확인할 수 있습니다.

실행 기록에서 각 단계의 상태는 다음과 같이 표시됩니다.

| 상태 표시 | 의미 | 설명 |
| --- | --- | --- |
| PASS | 성공 | 해당 단계가 정상적으로 완료되었습니다. 지시한 대로 동작이 수행되었습니다. |
| HEALED | 적응 후 성공 | 화면이 이전과 달라졌지만, 시스템이 자동으로 적응하여 동작을 완료했습니다. 이 경우 화면의 어떤 부분이 변경되었는지 확인하는 것이 좋습니다. |
| FAIL | 실패 | 해당 단계를 완료하지 못했습니다. 화면에서 대상을 찾지 못했거나, 예상한 결과가 나타나지 않았습니다. |

---

**5단계: 결과를 종합적으로 판단합니다**

모든 단계가 PASS인 경우:
- 테스트가 성공한 것입니다. 지시한 모든 동작이 정상적으로 수행되었고, 확인 조건도 충족되었습니다.

HEALED가 포함된 경우:
- 테스트는 성공했지만, 화면에 변경된 부분이 있습니다. 해당 단계의 캡처 이미지를 확인하여 변경 내용이 의도된 것인지 확인하십시오.

FAIL이 포함된 경우:
- 테스트가 실패한 것입니다. 실패한 단계의 캡처 이미지를 확인하고, 아래 4.5절의 대처 방법을 참고하십시오.

---

### 4.5. 테스트가 실패했을 때 대처 방법

Chat 모드에서 테스트가 실패한 경우, 다음 항목을 순서대로 확인하십시오.

---

**문제 1: 특정 단계에서 대상을 찾지 못했다는 오류가 발생합니다**

원인: 지시문에 작성한 대상의 이름이 실제 화면에 표시되는 이름과 다를 수 있습니다.

해결 방법:
1. 실패한 단계의 캡처 이미지를 확인합니다.
2. 실제 화면에서 해당 요소가 어떤 이름으로 표시되어 있는지 확인합니다.
3. 지시문의 대상 이름을 실제 화면에 표시되는 이름과 동일하게 수정합니다.
4. 수정된 지시문으로 다시 실행합니다.

예시:
- 지시문에 "'로그인' 버튼을 클릭합니다"라고 썼는데, 실제 화면에는 '로그인하기'라고 표시되어 있다면, "'로그인하기' 버튼을 클릭합니다"로 수정합니다.

---

**문제 2: 페이지가 완전히 로딩되기 전에 동작을 수행하여 실패합니다**

원인: 이전 단계에서 페이지 전환이 발생했는데, 다음 단계의 동작이 페이지 로딩보다 빨리 실행되었습니다.

해결 방법:
1. 페이지가 전환되는 단계 다음에 '2초 대기합니다' 또는 '3초 대기합니다'를 추가합니다.
2. 데이터가 많은 페이지는 대기 시간을 더 길게 설정합니다 (예: '5초 대기합니다').

---

**문제 3: 확인 조건이 충족되지 않아 실패합니다**

원인: 지시문에 작성한 확인 조건의 텍스트가 실제 화면에 표시되는 텍스트와 정확하게 일치하지 않을 수 있습니다.

해결 방법:
1. 실패한 단계의 캡처 이미지를 확인합니다.
2. 실제 화면에 표시되는 텍스트를 정확하게 확인합니다.
3. 확인 조건의 텍스트를 실제 화면에 표시되는 텍스트와 동일하게 수정합니다.

예시:
- "'로그인 성공'이 표시되는지 확인합니다"라고 썼는데, 실제로는 '로그인에 성공하였습니다'라고 표시된다면, 해당 텍스트로 수정합니다.

---

**문제 4: 테스트 실행 자체가 시작되지 않습니다**

원인: 사전 준비 사항이 충족되지 않았을 수 있습니다.

해결 방법:
1. 3절의 '시작하기 전에 확인할 사항'을 다시 확인합니다.
2. 특히 에이전트 연결 상태와 테스트 생성 서비스 가동 상태를 확인합니다.

---

**문제 5: 시나리오가 생성되었지만 내용이 이상합니다**

원인: 지시문이 모호하게 작성되어 시스템이 의도와 다르게 해석했을 수 있습니다.

해결 방법:
1. 4.2절의 '좋은 지시문의 네 가지 원칙'에 따라 지시문을 수정합니다.
2. 특히 각 단계가 하나의 동작만 포함하는지 확인합니다.
3. 대상 요소의 이름이 구체적인지 확인합니다.

---

**문제 6: 특정 동작은 성공하지만 이후 동작이 계속 실패합니다**

원인: 이전 단계에서 예상과 다른 페이지로 이동했거나, 팝업 창이 나타나 이후 동작을 방해할 수 있습니다.

해결 방법:
1. 실패하기 직전 단계의 캡처 이미지를 확인합니다.
2. 예상하지 못한 팝업이나 알림 창이 표시되어 있다면, 해당 팝업을 닫는 단계를 지시문에 추가합니다.
3. 예상과 다른 페이지에 있다면, 해당 페이지에 맞게 이후 지시문을 수정합니다.

---

## 5. 방법 2: 기획서 파일로 테스트 지시하기 (Doc 모드)

### 5.1. 언제 이 방법을 사용합니까

Doc 모드는 다음과 같은 상황에서 사용하기에 적합합니다.

**기획 문서가 이미 작성되어 있을 때**

기획 단계에서 이미 요구사항 정의서, 기능 명세서, 화면 설계서 등을 작성해 둔 경우, 해당 문서를 그대로 활용하여 테스트를 생성할 수 있습니다. 별도의 테스트 지시문을 다시 작성할 필요가 없으므로 시간을 절약할 수 있습니다.

**여러 기능을 한 번에 테스트해야 할 때**

기획서에는 보통 여러 기능에 대한 요구사항이 포함되어 있습니다. Doc 모드를 사용하면 문서에 포함된 여러 기능에 대한 테스트 시나리오를 한 번에 자동 생성할 수 있습니다.

**테스트의 근거를 문서로 남기고 싶을 때**

기획서를 기반으로 테스트를 생성하면, "어떤 요구사항에 근거하여 이 테스트를 수행했는가"를 명확하게 추적할 수 있습니다.

---

### 5.2. 사용할 수 있는 파일 형식

자율 테스트 시스템이 인식할 수 있는 파일 형식은 다음과 같습니다.

| 파일 형식 | 설명 | 권장 여부 |
| --- | --- | --- |
| 문서 파일 (.pdf) | 가장 널리 사용되는 형식입니다. 대부분의 기획서가 이 형식으로 작성됩니다. | 권장 |
| 문서 파일 (.docx) | 편집 가능한 문서 형식입니다. | 사용 가능 |
| 텍스트 파일 (.txt) | 순수 텍스트 파일입니다. | 사용 가능 |

파일을 준비할 때 다음 사항을 참고하십시오.

- 문서에 테스트 대상 기능의 동작이 구체적으로 기술되어 있을수록 좋은 테스트가 생성됩니다.
- "로그인 버튼을 클릭하면 로그인 페이지로 이동한다"와 같이 사용자의 동작과 예상 결과가 기술된 문서가 가장 효과적입니다.
- 화면 목업 이미지만 있는 문서보다는, 동작 설명이 텍스트로 기술된 문서가 더 정확한 테스트를 생성합니다.

---

### 5.3. Jenkins에서 테스트를 실행합니다

---

**1단계: Jenkins에 접속합니다**

웹 브라우저를 열고 Jenkins 주소를 입력하여 로그인합니다.

---

**2단계: 테스트 실행 항목을 찾습니다**

대시보드에서 'DSCORE-ZeroTouch-QA'를 클릭합니다.

---

**3단계: 매개변수 입력 화면으로 이동합니다**

화면 좌측 메뉴에서 'Build with Parameters'를 클릭합니다.

---

**4단계: 실행 방식을 선택합니다**

'RUN_MODE' 드롭다운에서 'doc'을 선택합니다.

---

**5단계: 테스트 대상 웹사이트 주소를 입력합니다**

'TARGET_URL'에 테스트할 웹사이트의 주소를 입력합니다.

---

**6단계: 기획서 파일을 업로드합니다**

'DOC_FILE' 항목에서 '파일 선택' 버튼을 클릭하고, 준비한 기획서 파일을 선택합니다.

참고: 'DOC_FILE' 항목이 보이지 않는 경우

일부 Jenkins 환경에서는 파일 업로드 기능이 제공되지 않을 수 있습니다. 이 경우 다음과 같이 대체할 수 있습니다.

1. 'RUN_MODE'를 'doc' 대신 'chat'으로 변경합니다.
2. 기획서의 핵심 내용(요구사항, 기능 설명 등)을 복사합니다.
3. 복사한 내용을 'SRS_TEXT' 입력란에 붙여넣습니다.
4. Chat 모드와 동일한 방식으로 테스트가 실행됩니다.

이 방법은 기획서의 텍스트를 직접 입력하는 것이므로, 파일 업로드와 유사한 결과를 얻을 수 있습니다. 다만 문서의 서식(표, 그림 등)은 반영되지 않으므로, 텍스트 내용 위주로 붙여넣으십시오.

---

**7단계: 테스트를 실행합니다**

'빌드' 버튼을 클릭합니다. 시스템이 기획서를 분석하여 테스트 시나리오를 자동 생성한 후, 생성된 시나리오를 순서대로 실행합니다.

---

**소요 시간 안내:**

Doc 모드는 기획서를 분석하는 과정이 추가되므로, Chat 모드보다 시간이 더 소요됩니다.

| 기획서 분량 | 예상 소요 시간 |
| --- | --- |
| 1~3페이지 | 약 3~5분 |
| 4~10페이지 | 약 5~10분 |
| 11페이지 이상 | 약 10~15분 |

---

### 5.4. 테스트 결과를 확인합니다

Doc 모드의 결과 확인 방법은 Chat 모드와 동일합니다. 4.4절의 '테스트 결과를 확인합니다'를 참고하십시오.

다만 Doc 모드에서는 다음과 같은 추가 사항이 있습니다.

**생성된 시나리오 검토:**

기획서에서 자동 생성된 시나리오가 의도한 내용과 일치하는지 확인하십시오. 결과 파일 중 시나리오 파일을 열어보면 어떤 동작들이 생성되었는지 확인할 수 있습니다.

**누락된 테스트 확인:**

기획서에 포함된 모든 요구사항이 테스트로 변환되었는지 확인하십시오. 누락된 요구사항이 있다면, 해당 내용을 별도의 지시문으로 작성하여 Chat 모드로 추가 실행할 수 있습니다.

---

### 5.5. 테스트가 실패했을 때 대처 방법

Doc 모드에서 실패가 발생한 경우, 4.5절의 Chat 모드 대처 방법에 추가로 다음 사항을 확인하십시오.

---

**문제 1: 기획서 내용을 인식하지 못합니다**

원인: 기획서의 내용이 이미지 위주이거나, 표로만 구성되어 텍스트를 추출하기 어려울 수 있습니다.

해결 방법:
1. 기획서의 핵심 요구사항을 텍스트로 정리합니다.
2. 정리한 텍스트를 'SRS_TEXT' 입력란에 직접 입력합니다.
3. 'RUN_MODE'를 'chat'으로 변경하여 실행합니다.

---

**문제 2: 생성된 시나리오가 너무 단순하거나 부정확합니다**

원인: 기획서의 내용이 기능 동작보다 비즈니스 배경 설명 위주로 작성되어 있을 수 있습니다.

해결 방법:
1. 기획서에서 사용자 동작과 예상 결과가 구체적으로 기술된 부분을 발췌합니다.
2. 발췌한 내용을 기반으로 Chat 모드의 지시문 형식으로 다시 작성합니다.
3. Chat 모드로 실행합니다.

---

**문제 3: 여러 시나리오 중 일부만 실패합니다**

원인: 성공한 시나리오의 실행 결과가 이후 시나리오에 영향을 주었을 수 있습니다. 예를 들어, 이전 시나리오에서 데이터를 삭제하여 다음 시나리오에서 해당 데이터를 찾지 못하는 경우입니다.

해결 방법:
1. 실패한 시나리오만 별도로 추출합니다.
2. Chat 모드로 해당 시나리오만 단독으로 실행하여 문제를 확인합니다.
3. 시나리오 간 의존성이 있다면, 각 시나리오가 독립적으로 실행될 수 있도록 지시문을 수정합니다.

---

## 6. 방법 3: 직접 시연하여 녹화하기 (Record 모드)

### 6.1. 언제 이 방법을 사용합니까

Record 모드는 다음과 같은 상황에서 사용하기에 적합합니다.

**복잡한 화면 조작을 글로 설명하기 어려울 때**

여러 단계의 드래그 앤 드롭, 복잡한 양식 입력, 여러 화면을 오가는 작업 등은 글로 설명하는 것보다 직접 시연하는 것이 빠르고 정확합니다.

**기존 업무 절차를 그대로 테스트로 만들고 싶을 때**

"매일 아침 수행하는 확인 절차"와 같이 이미 손에 익은 작업을 그대로 녹화하면, 동일한 절차를 언제든 자동으로 반복 실행할 수 있는 테스트가 만들어집니다.

**지시문 작성이 익숙하지 않을 때**

Chat 모드의 지시문 작성 요령이 아직 익숙하지 않다면, Record 모드로 시작하는 것이 더 직관적일 수 있습니다. 직접 보여주면 시스템이 이해합니다.

---

### 6.2. 녹화를 시작합니다

Record 모드는 Jenkins가 아닌 컴퓨터의 터미널 프로그램에서 직접 실행합니다. 아래 절차를 순서대로 따라하십시오.

---

**1단계: 터미널 프로그램을 엽니다**

컴퓨터에서 터미널 프로그램을 실행합니다.

- macOS의 경우: 'Spotlight 검색'(화면 우측 상단의 돋보기 아이콘 또는 Command+Space)에서 '터미널'을 검색하여 실행합니다.
- 또는 '응용 프로그램' 폴더 안의 '유틸리티' 폴더에서 '터미널'을 찾아 실행합니다.

---

**2단계: 녹화 명령을 실행합니다**

터미널 프로그램의 입력 창에 다음 내용을 입력하고 Enter 키를 누릅니다.

```
cd ~/Developer/automation/local_qa
```

위 명령은 시스템이 설치된 폴더로 이동하는 것입니다. 팀에서 안내받은 경로가 다른 경우, 해당 경로를 입력하십시오.

이어서 다음 명령을 입력하고 Enter 키를 누릅니다.

```
source venv/bin/activate
```

위 명령은 시스템의 실행 환경을 준비하는 것입니다.

마지막으로 다음 명령을 입력하고 Enter 키를 누릅니다. 아래의 웹사이트 주소 부분은 녹화할 대상 웹사이트의 실제 주소로 변경하십시오.

```
python3 mac_local_executor.py --mode record --url https://테스트할-웹사이트-주소
```

예시:

```
python3 mac_local_executor.py --mode record --url https://www.naver.com
```

---

**3단계: 브라우저가 열리기를 기다립니다**

명령을 실행하면 잠시 후 자동으로 브라우저가 열리고 지정한 웹사이트가 표시됩니다. 브라우저가 열리면 녹화가 시작된 것입니다.

브라우저가 열리지 않는 경우:
- 터미널에 오류 메시지가 표시되었는지 확인합니다.
- 웹사이트 주소가 올바른지 확인합니다.
- 시스템 관리 담당자에게 문의합니다.

---

### 6.3. 브라우저에서 동작을 시연합니다

브라우저가 열리면, 평소와 같이 웹사이트를 사용합니다. 시스템은 사용자의 동작을 자동으로 기록합니다.

---

**기록되는 동작:**

다음과 같은 동작이 자동으로 기록됩니다.

| 동작 | 설명 | 녹화 시 표시 |
| --- | --- | --- |
| 클릭 | 버튼, 링크, 메뉴 등을 클릭할 때 기록됩니다 | 클릭한 요소에 붉은 테두리가 잠시 나타납니다 |
| 텍스트 입력 | 입력란에 텍스트를 타이핑하면 기록됩니다 | 입력란에 붉은 테두리가 나타나고 화면이 캡처됩니다 |
| 드롭다운 선택 | 선택 목록에서 항목을 선택하면 기록됩니다 | 선택 목록에 붉은 테두리가 나타납니다 |
| 체크박스 | 체크박스를 체크하거나 해제하면 기록됩니다 | 체크박스에 붉은 테두리가 나타납니다 |

---

**녹화 중 유의사항:**

1. 각 동작을 천천히 수행하십시오. 너무 빠르게 클릭하면 일부 동작이 기록되지 않을 수 있습니다.

2. 텍스트를 입력한 후 잠시(약 1초) 기다린 후 다음 동작을 수행하십시오. 시스템이 텍스트 입력이 완료되었는지 판단하는 데 짧은 시간이 필요합니다.

3. 동작을 수행할 때마다 클릭한 요소에 붉은 테두리가 나타나는 것을 확인하십시오. 붉은 테두리가 나타나지 않으면 해당 동작이 기록되지 않은 것일 수 있습니다.

4. 각 동작을 수행할 때 화면이 자동으로 캡처됩니다. 이 캡처 이미지는 시나리오 생성 시 참고 자료로 활용됩니다.

5. 스크롤은 자동으로 기록되지 않습니다. 스크롤이 필요한 경우, 스크롤 후 원하는 요소를 클릭하면 해당 클릭 동작이 기록됩니다.

6. 새 탭이나 새 창을 여는 동작은 기록되지 않을 수 있습니다. 가급적 현재 탭 내에서 동작을 수행하십시오.

---

**시연 순서 권장 사항:**

녹화할 때 다음 순서로 시연하면 더 좋은 시나리오가 생성됩니다.

1. 시작 페이지에서 테스트할 기능이 있는 페이지로 이동합니다.
2. 테스트할 기능의 동작을 순서대로 수행합니다.
3. 마지막에 결과를 확인할 수 있는 요소(성공 메시지, 결과 목록 등)를 한 번 클릭합니다.

---

### 6.4. 녹화를 종료하고 결과를 확인합니다

---

**1단계: 브라우저를 닫습니다**

시연이 끝나면 브라우저 창을 닫습니다. 브라우저 창의 닫기 버튼(X)을 클릭하거나, Command+W(macOS)를 누릅니다.

---

**2단계: 시스템이 시나리오를 생성하는 동안 기다립니다**

브라우저를 닫으면 시스템이 자동으로 다음 작업을 수행합니다.

1. 녹화된 동작과 캡처된 화면을 분석합니다.
2. 분석 결과를 바탕으로 재사용 가능한 테스트 시나리오를 생성합니다.

이 과정은 녹화된 동작의 수에 따라 1~5분 정도 소요됩니다. 터미널에 진행 상황이 표시되므로 완료될 때까지 기다리십시오.

---

**3단계: 생성된 시나리오를 확인합니다**

시나리오 생성이 완료되면 터미널에 완료 메시지가 표시됩니다. 생성된 시나리오 파일은 다음 위치에 저장됩니다.

```
artifacts/scenario.json
```

이 파일을 열어보면 시스템이 녹화된 동작을 어떻게 해석하고 시나리오로 변환했는지 확인할 수 있습니다.

---

**4단계: 생성된 시나리오를 재사용합니다**

녹화로 생성된 시나리오는 이후 Jenkins에서 반복 실행할 수 있습니다. 시나리오 파일의 내용을 참고하여 Chat 모드의 지시문으로 변환하거나, 그대로 재실행할 수 있습니다.

---

### 6.5. 자주 발생하는 문제와 해결 방법

---

**문제 1: 명령을 실행했는데 브라우저가 열리지 않습니다**

원인: 시스템 실행 환경이 올바르게 준비되지 않았을 수 있습니다.

해결 방법:
1. 터미널에 표시된 오류 메시지를 확인합니다.
2. 시스템이 설치된 폴더로 올바르게 이동했는지 확인합니다.
3. 실행 환경 준비 명령('source venv/bin/activate')을 빠뜨리지 않았는지 확인합니다.
4. 시스템 관리 담당자에게 설치 상태를 확인 요청합니다.

---

**문제 2: 클릭했는데 붉은 테두리가 나타나지 않습니다**

원인: 클릭한 요소가 녹화 대상이 아니거나, 시스템이 해당 요소를 인식하지 못한 것일 수 있습니다.

해결 방법:
1. 해당 요소를 다시 한 번 천천히 클릭합니다.
2. 요소의 다른 부분(예: 텍스트 부분)을 클릭해 봅니다.
3. 그래도 기록되지 않으면, 해당 동작은 녹화 후 Chat 모드의 지시문으로 별도로 추가합니다.

---

**문제 3: 녹화 종료 후 시나리오가 생성되지 않습니다**

원인: 테스트 생성 서비스와의 연결에 문제가 있을 수 있습니다.

해결 방법:
1. 터미널에 표시된 오류 메시지를 확인합니다.
2. 테스트 생성 서비스가 가동 중인지 확인합니다 (3절의 확인 3 참조).
3. 인터넷 연결 상태를 확인합니다.
4. 시스템 관리 담당자에게 문의합니다.

---

**문제 4: 생성된 시나리오에 일부 동작이 빠져 있습니다**

원인: 너무 빠르게 동작을 수행하여 일부가 기록되지 않았거나, 시스템이 지원하지 않는 동작(예: 드래그 앤 드롭, 우클릭)을 수행한 경우입니다.

해결 방법:
1. 빠진 동작을 Chat 모드의 지시문으로 작성하여 시나리오에 추가합니다.
2. 녹화 시 각 동작 사이에 1~2초의 간격을 두고 다시 녹화합니다.

---

**문제 5: macOS에서 화면 녹화 권한 관련 오류가 발생합니다**

원인: macOS의 보안 설정으로 인해 화면 캡처가 차단되었을 수 있습니다.

해결 방법:
1. macOS의 '시스템 설정'을 엽니다.
2. '개인정보 보호 및 보안' 항목을 선택합니다.
3. '화면 기록' 항목에서 터미널 프로그램에 권한을 부여합니다.
4. '접근성' 항목에서도 동일하게 권한을 부여합니다.
5. 권한을 부여한 후 터미널을 닫았다가 다시 열고, 녹화를 다시 시작합니다.

---

## 7. 테스트 결과 파일 설명

테스트가 완료되면 여러 종류의 결과 파일이 생성됩니다. 각 파일의 역할과 확인 방법은 다음과 같습니다.

---

### 시나리오 원본 파일

**파일 이름:** scenario.json

**설명:** 시스템이 지시문 또는 기획서를 분석하여 생성한 테스트 시나리오의 원본입니다. 이 파일에는 시스템이 수행할 동작이 단계별로 정의되어 있습니다.

**확인 방법:** 이 파일을 열어보면 시스템이 어떤 동작을 어떤 순서로 수행할 것인지 확인할 수 있습니다. 의도와 다른 동작이 포함되어 있다면 지시문을 수정하여 다시 실행하십시오.

**활용 방법:** 이 파일은 이후 동일한 테스트를 반복 실행할 때 기준이 됩니다. 잘 만들어진 시나리오는 보관해 두면 유용합니다.

---

### 적응 반영 시나리오 파일

**파일 이름:** scenario.healed.json

**설명:** 테스트 실행 중에 화면 변경이 감지되어 시스템이 자동 적응한 경우, 그 결과가 반영된 최종 시나리오입니다.

**확인 방법:** 원본 시나리오(scenario.json)와 비교하면, 화면의 어떤 부분이 변경되었고 시스템이 어떻게 적응했는지 파악할 수 있습니다.

**활용 방법 (운영 권장 사항):**

테스트 결과에 "HEALED(적응 성공)" 항목이 있다면, 다음 절차에 따라 적응 결과를 재활용하십시오. 이를 통해 다음 실행 시 불필요한 적응 과정을 건너뛰어 실행 속도가 빨라지고, AI 호출 비용과 시간을 절약할 수 있습니다.

1. Jenkins 빌드 결과의 산출물(Artifacts)에서 `scenario.healed.json` 파일을 다운로드합니다.
2. 파일을 열어 수정된 부분을 확인합니다. 원본(`scenario.json`)과 비교하면 어떤 대상 요소가 변경되었는지 파악할 수 있습니다.
3. 수정된 내용이 올바르다면, 이 파일의 내용을 다음 테스트 실행 시 기준 지시문으로 활용합니다. 예를 들어, 수정된 요소 이름을 SRS_TEXT에 반영하거나, 시스템 관리 담당자에게 이 파일을 기준 시나리오로 등록해 달라고 요청하십시오.
4. 대상 웹사이트의 화면이 다시 변경될 때까지 이 기준 시나리오가 유효하게 유지됩니다.

---

### 실행 기록 파일

**파일 이름:** run_log.jsonl

**설명:** 각 단계의 실행 결과(성공, 적응 후 성공, 실패)가 순서대로 기록된 파일입니다.

**확인 방법:** 이 파일을 열면 각 단계별로 성공 여부와 적응 과정을 확인할 수 있습니다. 자세한 해석 방법은 8절을 참고하십시오.

**활용 방법:** 테스트 결과를 팀원에게 공유하거나, 문제 원인을 분석할 때 활용합니다.

---

### 단계별 화면 캡처 이미지

**파일 이름:** step_1_pass.png, step_2_pass.png, step_3_healed.png 등

**설명:** 각 단계에서 캡처된 실제 화면입니다. 시스템이 해당 단계를 수행했을 때 브라우저에 표시되던 화면 그대로입니다.

**확인 방법:** 이미지를 클릭하여 열면 실제 화면을 확인할 수 있습니다. 예상한 화면과 다른 경우, 지시문이나 웹사이트의 상태를 확인해야 합니다.

**활용 방법:** 테스트 결과를 시각적으로 확인하고, 팀원에게 공유할 때 활용합니다. 특히 실패한 단계의 이미지는 문제의 원인을 파악하는 데 큰 도움이 됩니다.

---

### 최종 오류 화면 이미지

**파일 이름:** error_final.png

**설명:** 테스트가 실패로 종료된 경우, 마지막 시점의 화면 상태입니다. 이 파일은 테스트가 성공한 경우에는 생성되지 않습니다.

**확인 방법:** 이 이미지를 통해 테스트가 실패한 시점에 화면이 어떤 상태였는지 확인할 수 있습니다. 예상하지 못한 팝업, 오류 메시지, 잘못된 페이지 등 실패 원인의 단서를 찾을 수 있습니다.

---

### 결과 파일 요약 표

| 파일 | 내용 | 항상 생성 여부 |
| --- | --- | --- |
| scenario.json | 테스트 시나리오 원본입니다. 어떤 동작을 수행할 것인지 정의되어 있습니다. | 항상 생성됩니다 |

## 8. 실행 로그 해석 방법

실행 기록 파일(run_log.jsonl)은 테스트의 각 단계가 어떻게 실행되었는지 상세하게 기록한 파일입니다. 이 절에서는 실행 기록을 읽고 해석하는 방법을 설명합니다.

---

### 기본 구조

실행 기록은 각 단계마다 한 줄씩 기록됩니다. 각 줄에는 다음 정보가 포함되어 있습니다.

| 항목 | 설명 |
| --- | --- |
| step | 몇 번째 단계인지 나타냅니다 (1부터 시작) |
| action | 해당 단계에서 수행한 동작의 종류입니다 (페이지 이동, 클릭, 텍스트 입력 등) |
| status | 해당 단계의 결과입니다 (성공, 적응 후 성공, 실패) |
| heal_stage | 자동 적응이 발생한 경우, 어떤 방식으로 적응했는지 나타냅니다 |

---

### 상태(status) 값의 의미

**PASS (성공)**

해당 단계가 처음 시도에서 바로 성공했음을 의미합니다. 시스템이 화면에서 대상을 정확하게 찾아 동작을 수행했고, 아무런 문제가 없었습니다.

이것은 가장 이상적인 결과입니다. 모든 단계가 PASS라면 테스트가 완벽하게 성공한 것입니다.

---

**HEALED (적응 후 성공)**

해당 단계에서 처음에는 대상을 찾지 못했지만, 시스템이 자동으로 적응하여 결국 성공했음을 의미합니다.

이것은 성공이지만, 화면에 변경이 있었다는 신호입니다. 예를 들어 다음과 같은 경우에 발생합니다.

- 버튼의 이름이 '확인'에서 '확인하기'로 변경된 경우
- 요소의 위치가 이동한 경우
- 화면의 구조가 부분적으로 변경된 경우

HEALED가 나타나면 해당 단계의 캡처 이미지를 확인하여, 변경된 내용이 의도된 것인지 확인하는 것을 권장합니다.

---

**FAIL (실패)**

해당 단계에서 대상을 찾지 못하거나 동작을 수행하지 못했고, 자동 적응도 실패했음을 의미합니다.

이것은 주의가 필요한 결과입니다. 다음과 같은 원인이 있을 수 있습니다.

- 지시문에 작성한 대상이 화면에 존재하지 않습니다.
- 웹사이트의 구조가 크게 변경되어 시스템이 적응하지 못했습니다.
- 이전 단계의 결과로 예상과 다른 페이지에 있습니다.
- 페이지 로딩이 완료되지 않았습니다.

FAIL이 발생하면 해당 단계의 캡처 이미지와 오류 메시지를 확인하고, 4.5절의 대처 방법을 참고하십시오.

---

### 적응 방식(heal_stage) 값의 의미

**none (적응 불필요)**

자동 적응이 필요하지 않았습니다. 첫 시도에서 바로 성공했거나, 적응을 시도하지 않고 실패한 경우입니다.

---

**local (자체 적응)**

시스템이 현재 화면을 자체적으로 분석하여, 원래 대상과 가장 유사한 요소를 찾아 대체한 경우입니다. 이 방식은 매우 빠르게 처리되며 (거의 즉시), 경미한 화면 변경에 효과적입니다.

---

**dify (지능형 적응)**

자체 적응이 실패한 후, 시스템의 지능형 분석 기능이 화면 전체를 종합적으로 분석하여 대체 요소를 제안한 경우입니다. 이 방식은 더 정교하지만, 처리에 약간의 시간(2~3초)이 소요됩니다.

---

### 실행 기록 예시와 해석

아래는 실행 기록의 예시와 각 줄의 해석입니다.

---

**예시 1: 모든 단계가 성공한 경우**

```
단계 1 | 동작: 페이지 이동 | 결과: PASS | 적응: 없음
단계 2 | 동작: 텍스트 입력 | 결과: PASS | 적응: 없음
단계 3 | 동작: 키 누르기   | 결과: PASS | 적응: 없음
단계 4 | 동작: 확인       | 결과: PASS | 적응: 없음
```

해석: 4개 단계 모두 처음 시도에서 성공했습니다. 화면 변경이 없었고 모든 동작이 정상적으로 수행되었습니다. 테스트가 완벽하게 성공한 경우입니다.

---

**예시 2: 일부 단계에서 자동 적응이 발생한 경우**

```
단계 1 | 동작: 페이지 이동 | 결과: PASS   | 적응: 없음
단계 2 | 동작: 텍스트 입력 | 결과: PASS   | 적응: 없음
단계 3 | 동작: 클릭       | 결과: HEALED | 적응: 자체 적응
단계 4 | 동작: 확인       | 결과: PASS   | 적응: 없음
```

해석: 3번째 단계의 클릭 동작에서 원래 대상을 찾지 못했지만, 시스템이 자체적으로 유사한 요소를 찾아 성공했습니다. 해당 버튼의 이름이나 위치가 변경되었을 가능성이 있으므로, 3번째 단계의 캡처 이미지를 확인하십시오.

---

**예시 3: 실패가 포함된 경우**

```
단계 1 | 동작: 페이지 이동 | 결과: PASS   | 적응: 없음
단계 2 | 동작: 텍스트 입력 | 결과: PASS   | 적응: 없음
단계 3 | 동작: 클릭       | 결과: HEALED | 적응: 지능형 적응
단계 4 | 동작: 확인       | 결과: FAIL   | 적응: 없음
```

해석: 3번째 단계에서 자체 적응도 실패하여 지능형 적응까지 동원했고, 결국 성공했습니다. 그러나 4번째 단계의 확인에서 실패했습니다. 이는 3번째 단계의 클릭 후 예상한 결과가 화면에 표시되지 않았다는 의미입니다. 화면 변경의 영향이거나, 웹사이트의 동작 자체에 문제가 있을 수 있습니다.

---

## 9. 자주 발생하는 문제와 해결 방법

아래 표는 자율 테스트 시스템을 사용하면서 자주 발생하는 문제와 그 해결 방법을 정리한 것입니다.

---

| 번호 | 증상 | 원인 | 해결 방법 |
| --- | --- | --- | --- |
| 1 | 테스트를 실행했는데 아무 일도 일어나지 않습니다 | 에이전트가 Jenkins에 연결되어 있지 않습니다 | Jenkins 관리 화면에서 mac-local-agent의 연결 상태를 확인하십시오. 'Connected' 상태가 아니라면 에이전트 컴퓨터가 켜져 있는지 확인하고, 에이전트 프로그램을 재시작하십시오. |
| 2 | '서비스 연결 실패' 오류가 표시됩니다 | 테스트 생성 서비스가 중지되었거나, 인증 정보가 올바르지 않습니다 | 시스템 관리 담당자에게 서비스 상태를 확인 요청하십시오. 인증 정보가 변경된 경우 Jenkins의 인증 설정을 업데이트해야 합니다. |
| 3 | 시나리오가 생성되지 않고 오류가 발생합니다 | 지시문이 너무 모호하여 시스템이 테스트 시나리오를 만들지 못했습니다 | 지시문을 4.2절의 작성 요령에 따라 더 구체적으로 수정하십시오. 번호를 매기고 각 단계에 하나의 동작만 포함하십시오. |
| 4 | 브라우저가 열리지 않습니다 | 에이전트 컴퓨터에서 브라우저를 실행할 권한이 없거나, 화면 접근 권한이 부여되지 않았습니다 | 에이전트 컴퓨터의 macOS 시스템 설정에서 '개인정보 보호 및 보안'의 '화면 기록' 및 '접근성' 항목에서 필요한 권한을 부여하십시오. |
| 5 | 화면 캡처 이미지가 검은색으로 나옵니다 | macOS의 화면 기록 권한이 부여되지 않아 화면 내용을 캡처하지 못한 것입니다 | 에이전트 컴퓨터의 macOS 시스템 설정에서 '화면 기록' 권한을 부여하십시오. 권한을 부여한 후 에이전트 프로그램을 재시작하십시오. |
| 6 | 특정 단계에서 '대상을 찾을 수 없습니다' 오류가 반복됩니다 | 지시문에 작성한 대상의 이름이 실제 화면에 표시되는 이름과 다릅니다 | 실패한 단계의 캡처 이미지를 확인하고, 실제 화면에 표시되는 이름으로 지시문을 수정하십시오. |
| 7 | 테스트는 성공했지만 결과가 예상과 다릅니다 | 확인 조건이 너무 넓거나, 의도와 다른 요소를 확인하고 있을 수 있습니다 | 확인 조건을 더 구체적으로 작성하십시오. 예: '페이지에 텍스트가 있는지 확인' 대신 '결과 목록에 DSCORE라는 텍스트가 있는지 확인' |
| 8 | 테스트 실행 중 시간이 너무 오래 걸립니다 | 테스트 생성 서비스가 느리게 응답하거나, 대상 웹사이트의 로딩 속도가 느립니다 | 대상 웹사이트에 직접 접속하여 로딩 속도를 확인하십시오. 정상인데도 느리다면 시스템 관리 담당자에게 서비스 성능을 확인 요청하십시오. |
| 9 | Doc 모드에서 파일 업로드가 되지 않습니다 | Jenkins에 파일 업로드 기능이 설치되어 있지 않습니다 | 5.3절의 안내에 따라 Chat 모드로 대체하십시오. 기획서의 핵심 내용을 복사하여 SRS_TEXT에 붙여넣으면 동일한 효과를 얻을 수 있습니다. |
| 10 | Record 모드에서 동작이 일부만 기록됩니다 | 동작 사이의 간격이 너무 짧거나, 시스템이 지원하지 않는 동작을 수행했습니다 | 각 동작 사이에 1~2초의 간격을 두고 천천히 시연하십시오. 6.3절의 '기록되는 동작' 표를 확인하여 지원되는 동작만 수행하십시오. |
| 11 | 로그인이 필요한 페이지에서 테스트가 실패합니다 | 시스템이 로그인 과정 없이 바로 로그인 이후 페이지에 접근하려 했습니다 | 지시문의 첫 번째 부분에 로그인 절차(아이디 입력, 비밀번호 입력, 로그인 버튼 클릭)를 포함하십시오. |
| 12 | 팝업이나 알림 창 때문에 이후 동작이 실패합니다 | 예상하지 못한 팝업이나 알림이 화면을 가려 다음 동작을 수행하지 못합니다 | 팝업이 나타나는 시점에 팝업을 닫는 단계를 지시문에 추가하십시오. 예: "'닫기' 버튼을 클릭합니다" 또는 "'확인' 버튼을 클릭합니다" |
| 13 | 이전에 성공했던 테스트가 갑자기 실패합니다 | 대상 웹사이트의 화면이 크게 변경되었거나, 서버에 문제가 발생했습니다 | 대상 웹사이트에 직접 접속하여 정상 동작 여부를 먼저 확인하십시오. 화면이 변경된 경우 지시문을 새로운 화면에 맞게 업데이트하십시오. |
| 14 | 같은 이름의 버튼이 여러 개 있어서 엉뚱한 버튼을 클릭합니다 | 지시문에서 대상을 구체적으로 지정하지 않아 시스템이 첫 번째로 발견한 요소를 클릭한 것입니다 | 대상을 더 구체적으로 지정하십시오. 예: "'확인' 버튼을 클릭합니다" 대신 "'주문 확인' 버튼을 클릭합니다" 또는 "페이지 하단의 '확인' 버튼을 클릭합니다" |
| 15 | 적응 결과 파일(scenario.healed.json)을 재사용하고 싶습니다 | 자동 적응이 발생한 후 수정된 시나리오를 다음 실행에 활용하고 싶은 경우입니다 | 본 문서의 "적응 반영 시나리오 파일" 섹션의 활용 방법을 참조하십시오. 적응 결과 파일을 다운로드하여 내용을 확인한 뒤, 올바르다면 다음 실행의 기준 지시문에 반영하면 됩니다. |
| 16 | "최종 복구 실패" 메시지가 표시되며 테스트가 중단됩니다 | 대상 웹사이트의 화면이 완전히 개편되어, 자동 적응만으로는 올바른 요소를 찾을 수 없는 상태입니다 | 화면이 대폭 변경된 경우에는 기존 지시문으로는 한계가 있습니다. 방법 3(직접 시연하여 녹화하기)을 사용하여 변경된 화면에서 새로운 시나리오를 처음부터 만드는 것을 권장합니다. 또는 SRS_TEXT를 새 화면에 맞게 다시 작성하십시오. |
| 17 | AI 서비스 통신 실패 또는 응답 시간 초과 메시지가 나타납니다 | 지시문이 너무 길거나 복잡하여 AI가 처리 시간 내에 시나리오를 생성하지 못한 경우입니다 | 지시문을 짧은 단위로 분할하여 실행하십시오. 예를 들어, 20단계짜리 테스트를 10단계씩 2회로 나누어 실행하면 각각의 처리 시간이 단축됩니다. 그래도 실패하면 시스템 관리 담당자에게 AI 서비스의 상태를 확인해 달라고 요청하십시오. |

## 부록: 시스템이 인식하는 9가지 동작

자율 테스트 시스템은 다음 9가지 동작을 인식하고 수행할 수 있습니다. 지시문에서 아래의 동작에 해당하는 내용을 작성하면, 시스템이 자동으로 적절한 동작을 선택하여 실행합니다.

---

### 동작 1: 페이지 이동

**하는 일:** 지정한 웹사이트 주소로 이동합니다.

**지시문 작성 예시:**
- "네이버 메인 페이지(https://www.naver.com)에 접속합니다."
- "로그인 페이지(https://my-app.com/login)로 이동합니다."

**설명:** 테스트의 첫 번째 단계에서 주로 사용합니다. 접속할 웹사이트의 전체 주소를 포함하여 작성하면 시스템이 해당 주소로 브라우저를 이동시킵니다. 테스트 도중 다른 페이지로 이동해야 할 때도 사용할 수 있습니다.

---

### 동작 2: 클릭

**하는 일:** 화면에서 지정한 요소(버튼, 링크, 메뉴 등)를 클릭합니다.

**지시문 작성 예시:**
- "'로그인' 버튼을 클릭합니다."
- "'회원가입' 링크를 클릭합니다."
- "'상품 목록' 메뉴를 클릭합니다."

**설명:** 가장 많이 사용되는 동작입니다. 클릭할 대상의 이름을 화면에 표시되는 그대로 작성하십시오. 같은 이름의 요소가 여러 개 있는 경우, 위치나 부가 설명을 추가하여 구분하십시오.

---

### 동작 3: 텍스트 입력

**하는 일:** 지정한 입력란에 텍스트를 입력합니다.

**지시문 작성 예시:**
- "'이메일' 입력란에 'test@example.com'을 입력합니다."
- "'검색창'에 'DSCORE'를 입력합니다."
- "'비밀번호' 입력란에 'Password123!'을 입력합니다."

**설명:** 입력란의 이름(레이블)과 입력할 텍스트를 함께 작성합니다. 입력할 텍스트는 작은따옴표로 감싸면 시스템이 더 정확하게 인식합니다. 기존에 입력되어 있는 값이 있으면 지우고 새로 입력합니다.

---

### 동작 4: 키 누르기

**하는 일:** 키보드의 특정 키를 누릅니다.

**지시문 작성 예시:**
- "엔터 키를 누릅니다."
- "탭 키를 누릅니다."
- "ESC 키를 누릅니다."

**설명:** 검색창에 텍스트를 입력한 후 엔터 키를 누르거나, 팝업 창을 ESC 키로 닫는 등의 키보드 조작에 사용합니다.

---

### 동작 5: 선택 목록에서 항목 선택

**하는 일:** 드롭다운 선택 목록에서 지정한 항목을 선택합니다.

**지시문 작성 예시:**
- "'국가' 선택 목록에서 '대한민국'을 선택합니다."
- "'카테고리' 목록에서 '공지사항'을 선택합니다."
- "'정렬 기준' 목록에서 '최신순'을 선택합니다."

**설명:** 드롭다운 형태의 선택 목록에서 특정 항목을 선택할 때 사용합니다. 선택 목록의 이름과 선택할 항목의 이름을 정확하게 작성하십시오.

---

### 동작 6: 체크박스 선택

**하는 일:** 체크박스를 체크하거나 해제합니다.

**지시문 작성 예시:**
- "'이용약관 동의' 체크박스를 체크합니다."
- "'마케팅 수신 동의' 체크박스를 체크합니다."
- "'전체 동의' 체크박스를 해제합니다."

**설명:** 체크박스를 선택(체크)하거나 선택 해제할 때 사용합니다. '체크합니다'라고 쓰면 선택 상태로 만들고, '해제합니다'라고 쓰면 선택을 해제합니다.

---

### 동작 7: 마우스 올리기

**하는 일:** 지정한 요소 위에 마우스 커서를 올립니다 (클릭하지 않습니다).

**지시문 작성 예시:**
- "'메뉴' 항목 위에 마우스를 올립니다."
- "'내 계정' 위에 마우스를 올립니다."

**설명:** 마우스를 올렸을 때 하위 메뉴가 나타나거나, 도움말 풍선이 표시되는 경우에 사용합니다. 클릭과 달리 마우스 커서만 해당 위치로 이동시킵니다.

---

### 동작 8: 대기

**하는 일:** 지정한 시간 동안 아무 동작도 하지 않고 기다립니다.

**지시문 작성 예시:**
- "2초 대기합니다."
- "3초 대기합니다."
- "5초 대기합니다."

**설명:** 페이지 전환 후 로딩이 완료될 때까지 기다리거나, 데이터가 화면에 표시될 때까지 기다릴 때 사용합니다. 페이지 전환이 발생하는 단계 다음에 '2초 대기합니다'를 추가하는 것을 권장합니다. 데이터가 많은 페이지는 더 긴 대기 시간을 설정하십시오.

---

### 동작 9: 확인

**하는 일:** 화면에 특정 요소가 존재하는지, 또는 특정 텍스트가 표시되는지 확인합니다.

**지시문 작성 예시:**
- "화면에 '검색 결과'라는 텍스트가 표시되는지 확인합니다."
- "'환영합니다'라는 메시지가 화면에 나타나는지 확인합니다."
- "상품 목록이 화면에 표시되는지 확인합니다."
- "'로그인 성공'이라는 텍스트가 포함되어 있는지 확인합니다."

**설명:** 테스트의 성공 여부를 판단하는 가장 중요한 동작입니다. 지시문의 마지막 단계에 확인 동작을 포함하면, 시스템이 자동으로 해당 조건을 검증하고 성공 또는 실패를 판정합니다. 확인할 텍스트는 실제 화면에 표시되는 텍스트와 정확하게 일치하도록 작성하십시오.

---

### 동작 요약 표

| 번호 | 동작 | 지시문에서의 표현 | 사용 예시 |
| --- | --- | --- | --- |
| 1 | 페이지 이동 | "~에 접속합니다", "~로 이동합니다" | 웹사이트 주소로 이동할 때 |
| 2 | 클릭 | "~를 클릭합니다" | 버튼, 링크, 메뉴를 누를 때 |
| 3 | 텍스트 입력 | "~에 '값'을 입력합니다" | 입력란에 글자를 쓸 때 |
| 4 | 키 누르기 | "~ 키를 누릅니다" | 엔터, 탭, ESC 등을 누를 때 |
| 5 | 선택 목록에서 항목 선택 | "~ 목록에서 '값'을 선택합니다" | 드롭다운에서 고를 때 |
| 6 | 체크박스 선택 | "~ 체크박스를 체크합니다" | 체크박스를 켜거나 끌 때 |
| 7 | 마우스 올리기 | "~ 위에 마우스를 올립니다" | 하위 메뉴를 표시할 때 |
| 8 | 대기 | "N초 대기합니다" | 페이지 로딩을 기다릴 때 |
| 9 | 확인 | "~가 표시되는지 확인합니다" | 테스트 결과를 검증할 때 |
