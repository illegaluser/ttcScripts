# 자율 E2E 테스트 파이프라인 (Job #6)

---

## 테스트 코드, 아직도 직접 작성하고 계신가요?

소프트웨어를 개발하다 보면, 기능을 만드는 것보다 그 기능이 제대로 동작하는지 확인하는 테스트를 작성하는 데 더 많은 시간이 걸리는 경우가 있습니다. 화면의 버튼 하나가 바뀌면 관련된 테스트가 전부 깨져서 다시 고쳐야 하고, 기획서에 적힌 시나리오를 일일이 테스트 코드로 번역하는 반복 작업이 이어집니다.

QA 담당자분은 테스트해야 할 시나리오를 가장 잘 알고 있지만, 정작 테스트를 만들려면 코드를 작성해야 합니다. 기획자분은 기획서에 모든 시나리오를 꼼꼼히 정리해 두셨는데, 이것을 테스트로 옮기는 것은 또 다른 일입니다.

이 모든 문제를 해결하기 위해 만들어진 것이 자율 E2E 테스트 파이프라인입니다.

---

## 이 기능이 해결하는 문제

자연어로 "네이버에서 DSCORE를 검색해줘"라고 지시하면, AI가 알아서 브라우저를 열고, 검색창에 입력하고, 엔터를 누르고, 결과 화면을 캡처합니다. 테스트 코드를 한 줄도 작성할 필요가 없습니다.

"로그인 페이지에 가서 아이디와 비밀번호를 입력하고 로그인 버튼을 눌러줘"라고 말하면, 시스템이 사람처럼 브라우저를 조작하여 로그인 과정을 수행하고 그 결과를 보고합니다.

더 놀라운 점이 있습니다. 화면 디자인이 변경되어도 AI가 "로그인 버튼이 여기로 옮겨갔구나" 하고 스스로 찾아갑니다. 깨지지 않는 테스트입니다.

코드를 작성할 줄 몰라도, 브라우저 테스트 도구를 다뤄본 적이 없어도, 누구나 바로 사용할 수 있습니다.

---

## 세 가지 방법으로 테스트를 만들 수 있습니다

테스트를 만드는 방법은 세 가지입니다. 상황에 따라 가장 편한 방법을 선택하시면 됩니다.

---

### 방법 1: 말로 지시하기 (Chat 모드)

가장 직관적인 방법입니다. 평소에 동료에게 테스트를 부탁하듯 자연어로 지시하면, AI가 그대로 실행합니다.

예를 들어 이렇게 지시할 수 있습니다.

"검색창에 'DSCORE'를 입력하고 엔터를 눌러주세요. 검색 결과가 화면에 나타나는지 확인해주세요."

이렇게 지시하면 AI가 브라우저를 열고, 검색창을 찾아 클릭하고, 'DSCORE'를 입력하고, 엔터를 누르고, 검색 결과가 나타나는지 확인한 후, 각 단계의 화면을 캡처하여 보고합니다.

더 복잡한 시나리오도 가능합니다.

"로그인 페이지에서 testuser 계정으로 로그인한 후, 대시보드 페이지가 나타나는지 확인해주세요. 대시보드에서 '최근 활동' 영역이 표시되는지도 함께 확인해주세요."

이 한 문장으로 로그인부터 대시보드 확인까지 전체 시나리오가 자동으로 실행됩니다.

#### 좋은 지시문 작성 요령

지시문을 잘 작성하면 AI가 더 정확하게 동작합니다. 다음 세 가지를 기억해 주십시오.

첫째, 번호를 매겨 순서대로 작성합니다. "1번, 검색창을 클릭합니다. 2번, DSCORE를 입력합니다."처럼 단계를 명확히 구분하면 AI가 순서를 놓치지 않습니다.

둘째, 한 단계에 하나의 동작만 적습니다. "검색창에 입력하고 엔터를 누르고 결과를 확인합니다"보다 각각 별도의 단계로 나누는 것이 더 정확합니다.

셋째, 마지막에 확인할 내용을 추가합니다. "~가 나타나는지 확인해주세요" 또는 "~가 표시되는지 확인해주세요"를 마지막 단계에 넣으면 테스트의 성공과 실패를 명확하게 판단할 수 있습니다.

#### Chat 모드 실행 절차

Jenkins에서 DSCORE-ZeroTouch-QA 프로젝트를 선택합니다. RUN_MODE를 chat으로 설정하고, SRS_TEXT 항목에 위와 같은 지시문을 입력한 후 빌드를 실행하면 됩니다.

---

### 방법 2: 기획서 올리기 (Doc 모드)

이미 작성된 기획서가 있다면, 해당 파일을 그대로 올리기만 하면 됩니다. AI가 기획서를 읽고, 그 안에 담긴 요구사항을 분석하여 테스트 시나리오를 자동으로 만들어 실행합니다.

예를 들어, 회원 가입 기능에 대한 기획서를 올리면 AI는 이런 테스트들을 스스로 만들어 냅니다. 정상적인 정보를 입력했을 때 가입이 완료되는지, 필수 항목을 비워 두었을 때 오류 메시지가 나타나는지, 이미 가입된 이메일로 시도했을 때 안내가 표시되는지 등을 자동으로 확인합니다.

기획서 한 장에서 수십 개의 테스트 시나리오가 동시에 만들어지므로, 수동으로 하나씩 작성하는 것보다 훨씬 빠르고 누락이 적습니다.

PDF, Word, 텍스트 파일 등 다양한 형식을 지원합니다.

#### Doc 모드 실행 절차

Jenkins에서 DSCORE-ZeroTouch-QA 프로젝트를 선택합니다. RUN_MODE를 doc으로 설정하고, DOC_FILE 항목에 기획서 파일을 업로드한 후 빌드를 실행하면 됩니다.

---

### 방법 3: 직접 시연하기 (Record 모드)

복잡한 화면 조작을 말로 설명하기 어려울 때가 있습니다. 이럴 때는 직접 브라우저에서 시연하면 됩니다.

브라우저를 평소처럼 사용하면, 시스템이 여러분의 동작을 관찰하고 테스트 시나리오로 변환합니다. 클릭할 때마다 해당 요소에 빨간 테두리가 표시되어 어떤 동작이 기록되고 있는지 실시간으로 확인할 수 있습니다.

예를 들어, 쇼핑몰에서 상품을 검색하고, 장바구니에 담고, 결제 페이지까지 이동하는 과정을 직접 시연합니다. 시연이 끝나면 시스템이 이 과정을 자동으로 정리하여 언제든 다시 실행할 수 있는 테스트 시나리오로 만들어 줍니다.

한 번 녹화해 두면, 이후에는 버튼 하나로 동일한 과정을 반복 검증할 수 있습니다.

#### Record 모드 실행 절차

Jenkins에서 DSCORE-ZeroTouch-QA 프로젝트를 선택합니다. RUN_MODE를 record로 설정하고 빌드를 실행하면, 녹화용 브라우저가 열립니다. 이 브라우저에서 테스트할 동작을 수행한 후 녹화를 종료하면 됩니다.

---

## 화면이 바뀌어도 테스트가 깨지지 않는 이유

기존의 테스트 도구들이 자주 실패하는 가장 큰 이유는 화면 요소를 찾는 방식에 있습니다.

기존 방식은 "3번째 영역 안의 2번째 버튼"처럼 위치를 기준으로 요소를 찾습니다. 디자인이 변경되어 영역의 순서가 바뀌거나 버튼의 위치가 달라지면, 테스트가 요소를 찾지 못해 실패합니다.

자율 테스트 시스템은 전혀 다른 방식으로 동작합니다. "로그인이라고 적힌 버튼"처럼 의미를 기준으로 요소를 찾습니다. 버튼의 위치가 화면 오른쪽에서 왼쪽으로 옮겨져도, 색깔이 파란색에서 초록색으로 바뀌어도, "로그인"이라는 텍스트가 있으면 정확히 찾아갑니다.

만약 텍스트까지 바뀌었다면 어떻게 될까요? 예를 들어 "로그인"이 "Sign In"으로 변경된 경우, 시스템이 화면 전체를 스캔하여 가장 비슷한 역할을 하는 요소를 자동으로 찾아서 테스트를 계속합니다.

이 기능 덕분에 디자인 변경이나 화면 개편이 있을 때마다 테스트를 수정하는 수고가 크게 줄어듭니다.

---

## 실행 결과로 받게 되는 것

테스트가 완료되면 다음과 같은 결과물을 확인할 수 있습니다.

### 각 단계별 스크린샷

테스트가 실제로 어떻게 진행되었는지 눈으로 확인할 수 있습니다. "검색창에 입력하는 장면", "검색 결과가 나타난 장면" 등 각 단계의 화면이 캡처되어 있어, 마치 옆에서 지켜보듯이 테스트 과정을 확인할 수 있습니다.

### 성공과 실패 보고서

어떤 단계에서 성공했고, 어떤 단계에서 문제가 발견되었는지 한눈에 파악할 수 있습니다. 실패한 단계가 있다면, 해당 시점의 스크린샷과 함께 무엇이 잘못되었는지 설명이 포함됩니다.

### 적응 기록

화면이 변경되었을 때 시스템이 어떻게 적응했는지 기록됩니다. 예를 들어, "로그인 버튼이 이전 위치에서 발견되지 않아 새로운 위치에서 찾았습니다"와 같은 기록을 통해 시스템의 판단 과정을 투명하게 확인할 수 있습니다.

---

## 이런 상황에서 활용해 보십시오

### QA 담당자의 회귀 테스트

새로운 기능이 추가되었을 때, 기존 기능이 여전히 잘 동작하는지 확인해야 합니다. 자주 확인하는 시나리오를 한 번 등록해 두면, 이후에는 빌드 버튼 하나로 전체 회귀 테스트를 실행할 수 있습니다. 코드를 작성하거나 수정할 필요가 없습니다.

### 기획자의 기능 검증 요청

기획서를 작성한 후, 해당 기능이 기획서대로 구현되었는지 직접 확인하고 싶을 때 유용합니다. 기획서 PDF를 올리기만 하면 시스템이 기획서의 요구사항을 테스트로 변환하여 자동으로 검증합니다.

### 스프린트 종료 시 주요 시나리오 검증

스프린트가 끝날 때마다 핵심 기능들이 정상 동작하는지 빠르게 확인할 수 있습니다. 로그인, 결제, 검색 등 주요 시나리오를 등록해 두고, 매 스프린트마다 동일한 기준으로 검증하면 품질을 일정하게 유지할 수 있습니다.

### 신규 배포 전 스모크 테스트

새로운 버전을 배포하기 전, 가장 기본적인 기능들이 정상인지 빠르게 확인하는 용도로 활용할 수 있습니다. 핵심 시나리오 몇 가지를 등록해 두면, 배포 전 빠른 검증이 가능합니다.

---

## 참고 사항

테스트 대상 웹사이트의 주소와 필요한 계정 정보는 사전에 준비해 주십시오. 한 번의 테스트 실행에 포함되는 단계가 많을수록 처리 시간이 길어질 수 있습니다. 기획서 기반 테스트의 경우 기획서의 내용이 구체적일수록 더 정확한 테스트가 생성됩니다.
