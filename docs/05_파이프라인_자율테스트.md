# 자율 E2E 테스트 파이프라인

## 1. 이 파이프라인의 목적

E2E(End-to-End) 테스트는 실제 사용자가 웹사이트를 사용하는 것과 동일한 방식으로, 브라우저에서 클릭하고, 입력하고, 결과를 확인하는 테스트입니다. 가장 현실적인 검증 방법이지만, 테스트를 작성하고 유지하는 비용이 높다는 근본적인 문제가 있습니다.

테스트 코드를 작성하려면 개발 역량이 필요하고, 대상 웹사이트의 화면이 조금만 바뀌어도 기존 테스트가 동작하지 않게 됩니다. 버튼의 위치가 옮겨지거나, 텍스트가 변경되거나, 디자인이 개편될 때마다 테스트를 수정해야 하므로, 테스트 유지보수가 개발 자체보다 더 많은 시간을 소모하는 경우도 드물지 않습니다.

자율 E2E 테스트 파이프라인은 이러한 문제를 근본적으로 해결합니다. 사용자가 자연어로 "검색창에 DSCORE를 입력하고 엔터를 눌러주세요"라고 지시하면, AI가 이를 브라우저에서 실행 가능한 동작으로 변환하여 직접 수행합니다. 테스트 코드를 작성할 필요가 없습니다. 또한, 대상 웹사이트의 화면이 변경되더라도 시스템이 변경된 요소를 자동으로 찾아가므로, 테스트가 쉽게 깨지지 않습니다.

---

## 2. 동작 흐름

이 파이프라인은 "지시 → 변환 → 실행 → 적응"의 흐름으로 동작합니다.

### 지시

사용자가 테스트하고 싶은 내용을 전달합니다. 전달 방식은 세 가지가 있습니다.

첫 번째는 자연어로 직접 지시하는 방법입니다. "로그인 페이지에서 아이디와 비밀번호를 입력하고 로그인 버튼을 클릭합니다. 대시보드 화면이 나타나는지 확인합니다."처럼, 테스트하고 싶은 동작을 순서대로 작성합니다.

두 번째는 기획서를 업로드하는 방법입니다. 요구사항 정의서나 기능 명세서(PDF, Word)를 시스템에 전달하면, AI가 문서를 읽고 테스트 시나리오를 자동으로 추출합니다.

세 번째는 브라우저에서 직접 시연하는 방법입니다. 시스템이 제공하는 브라우저에서 사용자가 평소처럼 웹사이트를 조작하면, 클릭, 입력, 선택 등의 동작이 자동으로 기록됩니다.

### 변환

사용자가 전달한 지시(자연어, 기획서, 또는 녹화된 동작)를 AI가 분석하여, 브라우저가 수행할 수 있는 표준화된 동작 목록으로 변환합니다. 이 시스템에서 사용하는 표준 동작은 페이지 이동, 클릭, 텍스트 입력, 키보드 입력, 드롭다운 선택, 체크박스 선택, 마우스 올리기, 대기, 확인 등 9가지입니다. AI가 이 범위를 벗어나는 동작을 생성하면 시스템이 즉시 차단하므로, 예측 불가능한 동작이 발생하지 않습니다.

### 실행

변환된 동작 목록에 따라 시스템이 실제 브라우저를 열고 각 동작을 순서대로 수행합니다. 각 동작이 완료될 때마다 화면을 캡처하여 증적 스크린샷으로 보존합니다.

### 적응 (Self-Healing)

실행 중 특정 요소를 찾지 못하면, 시스템이 자동으로 적응을 시도합니다.

일반적인 테스트 도구는 "페이지의 3번째 영역 안에 있는 2번째 버튼"처럼 위치 기반으로 요소를 찾습니다. 위치가 바뀌면 실패합니다. 이 시스템은 "로그인이라고 적힌 버튼"처럼 의미 기반으로 요소를 찾습니다. 요소의 접근성 역할, 화면 텍스트, 라벨, 힌트 텍스트 등 7가지 기준을 순서대로 시도하며, 이 중 하나라도 일치하면 테스트를 계속합니다.

7가지 기준 모두로 찾지 못한 경우에는, 먼저 현재 화면에서 가장 유사한 요소를 자동 검색합니다. 이 과정은 AI를 호출하지 않으므로 즉시 완료됩니다. 그래도 찾지 못하면 AI에게 화면 구조와 오류 정보를 전달하여 대체 요소를 제안받습니다.

이 적응 과정 덕분에, 대상 웹사이트의 디자인이 일부 변경되어도 기존 테스트 시나리오를 수정하지 않고 계속 사용할 수 있습니다.

## 3. 사용 방법

### 자연어로 지시하기

1. Jenkins에서 "DSCORE-ZeroTouch-QA" 작업을 선택합니다.
2. 테스트 대상 웹사이트의 주소와 자연어 지시문을 입력하고 실행합니다.
3. 실행이 완료되면 산출물에서 각 단계별 스크린샷과 성공/실패 기록을 확인합니다.

지시문은 번호를 매겨 순서대로, 각 단계에 하나의 동작만 포함하여 작성하는 것이 가장 효과적입니다. 마지막에 "~가 화면에 표시되는지 확인합니다"를 추가하면 검증 동작이 자동으로 포함됩니다.

### 기획서로 지시하기

1. Jenkins에서 동일한 작업을 선택합니다.
2. 기획서 파일을 업로드하고 실행합니다.
3. AI가 기획서를 분석하여 테스트 시나리오를 생성하고 실행합니다.

### 직접 시연하여 녹화하기

이 방법은 맥북 터미널에서 별도의 명령어로 실행합니다. 브라우저가 열리면 평소처럼 웹사이트를 사용하고, 창을 닫으면 녹화가 종료됩니다. 녹화된 동작은 AI가 정제하여 재사용 가능한 테스트 시나리오로 변환합니다.

## 4. 이 파이프라인을 통해 가능해지는 것

이 파이프라인의 실행이 완료되면, 각 테스트 단계별로 캡처된 스크린샷과 성공/실패 기록이 산출됩니다. 화면 변경에 시스템이 자동으로 적응한 경우에는 적응 전후의 기록도 함께 제공되므로, 어떤 요소가 변경되었고 시스템이 어떻게 대응했는지 확인할 수 있습니다.

이를 통해 다음과 같은 업무 흐름이 가능해집니다.

QA 담당자가 테스트 코드를 작성할 줄 몰라도, 자연어로 주요 사용자 시나리오를 정의하여 정기적으로 회귀 테스트를 실행할 수 있습니다. 기획자가 요구사항 문서를 업로드하면 해당 기능에 대한 E2E 검증이 자동으로 수행되므로, 개발팀에 별도의 테스트 작성을 요청하지 않아도 됩니다. 신규 배포 전 핵심 시나리오(로그인, 검색, 결제 등)를 자연어로 정의해 두면, 매 배포 시 자동으로 스모크 테스트를 수행하여 치명적 결함을 조기에 발견할 수 있습니다.

또한, 적응 기록이 포함된 시나리오 파일은 다음 실행의 기준으로 재활용할 수 있습니다. 화면 변경 후 적응에 성공한 시나리오를 저장해 두면, 이후 실행에서는 적응 과정을 건너뛰어 실행 속도가 빨라집니다.

---

## 5. 참고 사항

- 이 파이프라인은 Mac 환경의 실제 브라우저에서 실행되며, 테스트가 진행되는 동안 화면에서 브라우저 동작을 직접 관찰할 수 있습니다.
- 지시문이 구체적일수록 AI가 생성하는 시나리오의 정확도가 높아집니다.
- 모든 데이터 처리는 사내 로컬 서버에서 이루어지며, 테스트 대상 웹사이트의 정보가 외부로 전송되지 않습니다.
