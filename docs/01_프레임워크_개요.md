# DSCORE-TTC 프레임워크 개요

## 한 줄 요약

DSCORE-TTC는 **인터넷이 차단된 폐쇄망 환경에서, 단일 로컬 머신 위에 로컬 LLM을 구동하여 코드 품질 진단, 자율 E2E 테스트, AI 평가를 수행하는 온프레미스 QAOps 프레임워크**입니다.

---

## 1. 이 프레임워크의 핵심 특성

### 폐쇄망 완전 대응 (Air-Gapped Environment)

DSCORE-TTC는 **외부 인터넷 연결 없이 동작하도록 설계**되었습니다. 금융, 국방, 공공기관 등 보안 정책상 외부 네트워크 접속이 금지된 환경에서도 모든 기능을 사용할 수 있습니다.

- 모든 서비스(Jenkins, SonarQube, GitLab, Dify, Langfuse)가 로컬 Docker 컨테이너로 구동됩니다.
- AI 추론은 **Ollama**를 통해 로컬 머신의 GPU/CPU에서 직접 수행됩니다. OpenAI, Google 등 외부 AI 서비스에 데이터를 전송하지 않습니다.
- 코드, 문서, 테스트 결과 등 모든 데이터가 로컬 머신의 디스크에만 저장되며, 외부로 유출되지 않습니다.

> 외부 AI 서비스(OpenAI, Gemini 등)와의 연동은 **선택 사항**입니다. 폐쇄망에서는 로컬 LLM만으로 전체 파이프라인이 동작하며, 개방망에서는 외부 AI를 병행하여 성능을 높일 수 있습니다.

### 단일 머신 구동 (Single Machine Deployment)

별도의 서버 클러스터나 클라우드 인프라가 필요 없습니다. **개발자의 맥북 한 대**에서 전체 시스템을 구동할 수 있습니다.

- Docker Compose 하나로 6개 서비스(Jenkins, SonarQube, GitLab, Langfuse, PostgreSQL x2)를 일괄 기동합니다.
- Dify는 별도의 Docker Compose로 같은 머신에서 구동하며, 가상 네트워크로 연결됩니다.
- Ollama는 호스트 머신에서 직접 실행되어, 컨테이너 내부의 Jenkins가 `host.docker.internal`을 통해 접근합니다.
- 권장 사양: macOS, 16GB RAM 이상, 50GB 여유 디스크 공간

### 로컬 LLM 기반 (Local LLM First)

모든 AI 기능의 기본 동작은 **로컬에서 실행되는 LLM**에 의존합니다.

| 기능 | 사용하는 로컬 LLM | 역할 |
| --- | --- | --- |
| 지식 관리 (Job #1~#3, #5) | Dify + Ollama 연동 모델 | 문서 임베딩, 지식 검색, Q&A 생성 |
| 품질 분석 (Job #4) | Dify Workflow 내 LLM 노드 | SonarQube 이슈의 원인 진단 및 수정 방안 제안 |
| 자율 E2E 테스트 (Job #6) | Dify Chatflow (Planner/Healer) | 자연어를 테스트 시나리오로 변환, 에러 복구 판단 |
| AI 에이전트 평가 (Job #7) | Ollama (심판 LLM) | 11대 지표 채점 (Answer Relevancy, Toxicity 등) |

로컬 LLM은 Ollama를 통해 관리되며, 모델을 한 번 다운로드하면 이후 인터넷 없이도 무제한으로 사용할 수 있습니다. 추론 비용은 0원입니다.

## 2. 이 프레임워크를 만든 이유

소프트웨어 개발 현장에서 반복적으로 발생하는 네 가지 병목이 있습니다.

### 병목 1: 지식이 사람 머릿속에만 있습니다

프로젝트 문서, 코드 컨텍스트, 기술 레퍼런스가 개인 PC나 사내 위키에 흩어져 있어 신규 인력이 업무에 투입되기까지 오랜 시간이 소요됩니다. AI에게 "우리 프로젝트에서 로그인 로직이 어떻게 구현되어 있어?"라고 물어볼 수 있는 환경이 갖춰져 있지 않습니다.

### 병목 2: 코드 리뷰와 이슈 등록이 수동입니다

SonarQube가 코드 품질 문제를 발견해도, 그 결과를 해석하고 GitLab Issue로 정리하는 것은 사람의 몫입니다. 수십 개의 이슈를 일일이 분류하고 설명을 작성하는 데 반나절이 소요되는 경우가 흔합니다.

### 병목 3: UI 테스트가 깨지기 쉽습니다

Playwright나 Selenium으로 E2E 테스트를 작성해도, 버튼 텍스트 하나가 바뀌면 테스트가 즉시 실패합니다. 테스트 유지보수 비용이 개발 비용을 초과하는 상황이 발생합니다.

### 병목 4: 보안 규정이 외부 AI 사용을 금지합니다

ChatGPT, Claude 같은 외부 AI 서비스를 활용하면 생산성이 높아지지만, 폐쇄망 환경에서는 사용 자체가 불가능합니다. 코드와 문서를 외부 서버로 전송할 수 없기 때문입니다.

**DSCORE-TTC는 이 네 가지 병목을 로컬 LLM + 자동화 파이프라인으로 해결합니다. 외부 인터넷 연결 없이, 개발자의 로컬 머신 한 대에서 전체 QAOps 체계를 운영할 수 있습니다.**

### 병목 1: 지식이 사람 머릿속에만 있습니다

프로젝트 문서, 코드 컨텍스트, 기술 레퍼런스가 개인 PC나 사내 위키에 흩어져 있어 신규 인력이 업무에 투입되기까지 오랜 시간이 소요됩니다. AI에게 "우리 프로젝트에서 로그인 로직이 어떻게 구현되어 있어?"라고 물어볼 수 있는 환경이 갖춰져 있지 않습니다.

### 병목 2: 코드 리뷰와 이슈 등록이 수동입니다

SonarQube가 코드 품질 문제를 발견해도, 그 결과를 해석하고 GitLab Issue로 정리하는 것은 사람의 몫입니다. 수십 개의 이슈를 일일이 분류하고 설명을 작성하는 데 반나절이 소요되는 경우가 흔합니다.

### 병목 3: UI 테스트가 깨지기 쉽습니다

Playwright나 Selenium으로 E2E 테스트를 작성해도, 버튼 텍스트 하나가 바뀌면 테스트가 즉시 실패합니다. 테스트 유지보수 비용이 개발 비용을 초과하는 상황이 발생합니다.

**DSCORE-TTC는 이 세 가지 병목을 AI + 자동화 파이프라인으로 해결합니다.**

## 3. 5단계 자동화 체계

DSCORE-TTC는 5개의 Phase로 구성되며, 각 Phase는 독립적으로 사용할 수도 있고, 전체를 조합하여 통합 체계로 운영할 수도 있습니다.

| 단계 | 이름 | 하는 일 | 관련 Job |
| --- | --- | --- | --- |
| **Phase 1** | DevOps 인프라 | Jenkins, SonarQube, GitLab, Dify, Langfuse를 Docker로 구동합니다 | (인프라) |
| **Phase 2** | 지식 관리 | 문서, 코드, 웹 콘텐츠를 AI가 학습할 수 있도록 Dify에 자동 업로드합니다 | Job #1~#3, #5 |
| **Phase 3** | 품질 분석 | SonarQube 분석 결과를 LLM이 진단하고 GitLab Issue를 자동 생성합니다 | Job #4 |
| **Phase 4** | 자율 E2E 테스트 | 자연어 요구사항만으로 브라우저 E2E 테스트를 자동 생성 및 실행합니다 | Job #6 |
| **Phase 5** | AI 에이전트 평가 | 외부 AI(챗봇, API)를 11대 지표로 자동 채점합니다 | Job #7 |

---

## 4. 7개 Jenkins 파이프라인

모든 자동화는 Jenkins 파이프라인(Job)을 통해 실행됩니다. 사용자는 Jenkins 웹 UI에서 파라미터를 입력하고 "빌드" 버튼만 누르면 됩니다.

| Job | 이름 | 한 줄 설명 |
| --- | --- | --- |
| **#1** | DSCORE-Knowledge-Sync | 문서(PDF/Word/Excel)를 AI 지식 베이스에 업로드합니다 |
| **#2** | DSCORE-Knowledge-Sync-QA | 문서를 Q&A 형태로 변환하여 업로드합니다 |
| **#3** | DSCORE-Code-Knowledge-Sync | Git 저장소 구조와 핵심 코드를 분석하여 업로드합니다 |
| **#4** | DSCORE-Quality-Issue-Workflow | 정적 분석 → LLM 진단 → GitLab Issue 자동 등록을 수행합니다 |
| **#5** | DSCORE-Web-Knowledge-Sync | 웹사이트를 크롤링하여 텍스트 지식을 업로드합니다 |
| **#6** | DSCORE-ZeroTouch-QA | 자연어 기반 자율 E2E 테스트를 생성하고 실행합니다 |
| **#7** | DSCORE-Agent-Eval | 외부 AI 에이전트를 11대 지표로 자동 평가합니다 |

각 Job의 상세한 사용 방법은 개별 파이프라인 가이드 문서(03~06)를 참조하십시오.

## 5. 대상 사용자

| 역할 | 주로 사용하는 기능 |
| --- | --- |
| **개발자** | Job #1~#3으로 프로젝트 지식을 AI에 학습시키고, Job #4로 코드 품질 이슈를 자동 관리합니다 |
| **QA 엔지니어** | Job #6으로 자연어 요구사항만 입력하면 E2E 테스트가 자동 생성 및 실행됩니다 |
| **AI/ML 엔지니어** | Job #7로 개발한 AI 모델/챗봇의 품질을 11대 지표로 자동 측정합니다 |
| **기획자/PO** | Job #6에 기획서(PDF)를 업로드하면 테스트 시나리오가 자동 생성됩니다 |
| **팀 리드** | Langfuse 대시보드(`http://localhost:3000`)에서 AI 품질 추이를 실시간 모니터링합니다 |

## 6. 기술 스택

| 구분 | 기술 | 역할 |
| --- | --- | --- |
| CI/CD | Jenkins | 파이프라인 실행 및 스케줄링 |
| 코드 품질 | SonarQube | 정적 코드 분석 (버그, 취약점, 코드 스멜 탐지) |
| 소스 관리 | GitLab | Git 저장소 호스팅 및 이슈 트래커 |
| AI 플랫폼 | Dify | LLM 워크플로우 설계, 지식 베이스(RAG), Chatflow |
| AI 관측성 | Langfuse | LLM 호출 추적, 평가 점수 대시보드, 비용 모니터링 |
| 로컬 LLM | Ollama | 온프레미스 LLM 실행 엔진 (qwen3-coder 등) |
| E2E 테스트 | Playwright | 브라우저 자동화 엔진 |
| AI 평가 | DeepEval + Promptfoo | LLM 응답 심층 채점 및 보안 검사 |
| 인프라 | Docker Compose | 전체 서비스의 컨테이너화 및 네트워크 구성 |

## 7. 접속 정보

시스템이 구축된 후 브라우저에서 접속할 수 있는 주소는 다음과 같습니다.

| 서비스 | URL | 용도 |
| --- | --- | --- |
| Jenkins | `http://localhost:8080` | 파이프라인 실행 및 관리 |
| SonarQube | `http://localhost:9000` | 정적 분석 결과 확인 |
| GitLab | `http://localhost:8929` | 소스 코드 및 이슈 관리 |
| Langfuse | `http://localhost:3000` | AI 평가 결과 대시보드 |
| Dify | `http://localhost` (기본) | AI 워크플로우 및 지식 베이스 관리 |
