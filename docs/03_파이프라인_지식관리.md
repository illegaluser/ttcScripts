# 지식 관리 파이프라인 사용 가이드 (Job #1~#3, #5)

## 1. 이 기능으로 할 수 있는 것

이 기능은 프로젝트에서 사용하는 각종 문서, 소스 코드, 웹사이트 내용을 AI에게 학습시키는 기능입니다. 학습이 완료되면, AI가 일반적인 지식이 아니라 실제 프로젝트의 내용을 바탕으로 질문에 답변하게 됩니다.

쉽게 말해, 프로젝트 관련 자료를 AI에게 읽혀 두면, 이후에 AI에게 프로젝트에 대해 질문했을 때 실제 문서와 코드를 참고하여 구체적인 답변을 받을 수 있습니다.

아래 두 가지 사례를 통해 학습 전과 후의 차이를 확인하실 수 있습니다.

### 사례 1: 프로젝트 코드에 대한 질문

**학습 전:**

질문: "로그인 로직이 어떻게 되어 있어?"

AI 답변: "일반적으로 로그인 로직은 사용자 이름과 비밀번호를 서버에 전송하고, 서버에서 인증 토큰을 발급하는 방식으로 구현됩니다..."

이처럼 AI는 일반론만 답변하며, 실제 프로젝트에서 어떤 파일에 어떤 코드가 작성되어 있는지는 알 수 없습니다.

**학습 후:**

질문: "로그인 로직이 어떻게 되어 있어?"

AI 답변: "현재 프로젝트에서는 JwtTokenProvider.java 파일에서 토큰 기반 인증을 사용하고 있습니다. 토큰 만료 시간은 설정 파일에서 3600초로 지정되어 있으며, 로그인 요청은 AuthController.java의 login 메서드에서 처리됩니다."

학습 후에는 AI가 실제 프로젝트의 코드 파일을 참고하여 구체적인 파일명과 설정값을 포함한 답변을 제공합니다.

### 사례 2: 기획 문서에 대한 질문

**학습 전:**

질문: "우리 프로젝트의 인증 방식이 뭐라고 적혀 있어?"

AI 답변: "죄송합니다. 해당 프로젝트의 문서를 확인할 수 없어 정확한 답변을 드리기 어렵습니다."

AI가 프로젝트 문서를 읽은 적이 없기 때문에, 문서에 무엇이 적혀 있는지 답변할 수 없습니다.

**학습 후:**

질문: "우리 프로젝트의 인증 방식이 뭐라고 적혀 있어?"

AI 답변: "프로젝트 기획서 3.2절 '보안 요구사항'에 따르면, 본 프로젝트는 OAuth 2.0 기반의 소셜 로그인과 자체 회원가입을 병행하며, 세션 유지 시간은 30분으로 명시되어 있습니다."

학습 후에는 AI가 실제 기획 문서의 해당 항목을 참조하여, 문서에 기재된 내용을 그대로 인용하며 답변합니다.

## 2. 시작하기 전에 확인할 사항

학습 기능을 사용하기 전에, 아래 세 가지 사항이 모두 준비되어 있는지 확인하십시오. 하나라도 준비되지 않은 상태에서 실행하면 오류가 발생합니다.

### 2.1. Jenkins에 로그인할 수 있는지 확인합니다

Jenkins는 학습 작업을 실행하는 도구입니다. 먼저 Jenkins에 정상적으로 접속할 수 있는지 확인하십시오.

**확인 방법:**

1. 웹 브라우저(Safari 또는 Chrome)를 엽니다.
2. 주소 입력란에 `http://localhost:8080`을 입력하고 Enter 키를 누릅니다.
3. 로그인 화면이 나타나면, 관리자에게 부여받은 아이디와 비밀번호를 입력합니다.
4. 로그인 후 Jenkins 대시보드 화면이 나타나면 정상입니다.

**로그인 화면이 나타나지 않는 경우:**

- 브라우저에 "사이트에 연결할 수 없음" 메시지가 표시되면, Jenkins 서비스가 실행되고 있지 않은 것입니다.
- 이 경우 시스템 관리자에게 "Jenkins 서비스를 시작해 주십시오"라고 요청하십시오.

**아이디 또는 비밀번호를 모르는 경우:**

- 시스템 관리자에게 Jenkins 접속 계정을 요청하십시오.
- 최초 설치 시 기본 아이디는 `admin`이며, 비밀번호는 설치 시 설정한 값입니다.

### 2.2. AI 지식 저장소가 생성되어 있는지 확인합니다

AI가 학습한 내용을 저장하는 공간이 사전에 만들어져 있어야 합니다. 이 저장소는 Dify라는 AI 관리 도구에서 확인할 수 있습니다.

**확인 방법:**

1. 웹 브라우저에서 `http://localhost/apps`를 입력하고 Enter 키를 누릅니다.
2. Dify 로그인 화면이 나타나면, 관리자에게 부여받은 아이디와 비밀번호를 입력합니다.
3. 로그인 후 화면 상단의 메뉴에서 "Knowledge"(또는 "지식")를 클릭합니다.
4. 지식 저장소 목록이 나타납니다. 여기에 최소 2개의 저장소가 있어야 합니다.
   - 하나는 문서 원문을 저장하는 "문서형 저장소"입니다.
   - 다른 하나는 질문-답변 쌍을 저장하는 "질문답변형 저장소"입니다.
5. 2개의 저장소가 모두 표시되면 정상입니다.

**저장소가 보이지 않는 경우:**

- 시스템 관리자에게 "Dify에 지식 저장소(Dataset)를 2개 만들어 주십시오. 하나는 문서형, 하나는 질문답변형으로 필요합니다."라고 요청하십시오.

### 2.3. 연결 설정이 Jenkins에 등록되어 있는지 확인합니다

Jenkins가 AI 지식 저장소에 접근하려면, 접속 키와 저장소 주소가 Jenkins에 미리 등록되어 있어야 합니다.

**확인 방법:**

1. Jenkins에 로그인합니다.
2. 화면 왼쪽 메뉴에서 "Jenkins 관리"를 클릭합니다.
3. "Jenkins 관리" 화면에서 "Credentials"(또는 "인증 정보")를 클릭합니다.
4. "(global)" 영역을 클릭하면 등록된 인증 정보 목록이 나타납니다.
5. 다음 4개의 항목이 모두 등록되어 있는지 확인합니다.

| 번호 | 항목 이름 | 용도 |
| --- | --- | --- |
| 1 | dify-knowledge-key | 문서형 저장소 접속 키 |
| 2 | dify-dataset-id | 문서형 저장소 주소 |
| 3 | dify-knowledge-key-qa | 질문답변형 저장소 접속 키 |
| 4 | dify-dataset-id-qa | 질문답변형 저장소 주소 |

**4개 항목 중 하나라도 없는 경우:**

- 시스템 관리자에게 "Jenkins Credentials에 Dify 연결 정보 4개를 등록해 주십시오."라고 요청하십시오.
- 접속 키는 Dify의 지식 저장소 설정 화면에서 발급받을 수 있습니다.

## 3. Job #1: 프로젝트 문서를 AI에 학습시키기

### 3.1. 언제 사용합니까

이 기능은 프로젝트에서 사용하는 문서 파일을 AI에게 학습시킬 때 사용합니다. 다음과 같은 상황에서 실행하십시오.

**신규 프로젝트를 시작할 때:**

프로젝트 초기에 기획서, 요구사항 정의서, 설계 문서, 회의록 등 프로젝트 관련 문서를 한꺼번에 학습시킵니다. 이후 AI에게 프로젝트에 대한 질문을 하면, 학습된 문서를 기반으로 답변을 받을 수 있습니다.

**기존 문서가 수정되었을 때:**

기획서나 설계 문서의 내용이 변경된 경우, 변경된 문서를 다시 학습시켜야 합니다. 그래야 AI가 최신 내용을 기반으로 답변합니다.

**새로운 참고 자료를 추가할 때:**

외부에서 받은 기술 문서, 표준 가이드, 규격 문서 등 프로젝트에 참고할 자료가 추가된 경우, 해당 문서를 학습시키면 AI가 참고할 수 있는 지식 범위가 넓어집니다.

**팀원이 새로 합류했을 때:**

새로운 팀원이 프로젝트에 대해 질문할 수 있도록, 프로젝트의 주요 문서를 미리 AI에 학습시켜 두면 효과적입니다. 신규 팀원이 AI를 통해 프로젝트에 빠르게 적응할 수 있습니다.

### 3.2. 지원하는 파일 형식

AI에 학습시킬 수 있는 파일 형식은 다음과 같습니다.

| 파일 형식 | 확장자 | 설명 |
| --- | --- | --- |
| PDF 문서 | .pdf | 기획서, 보고서, 매뉴얼 등 PDF 형식의 문서입니다. 페이지별로 내용이 추출됩니다. |
| Word 문서 | .docx | 마이크로소프트 워드로 작성된 문서입니다. 문단별로 내용이 추출됩니다. |
| Excel 파일 | .xlsx, .xls | 마이크로소프트 엑셀로 작성된 스프레드시트입니다. 시트별로 표 형태로 내용이 추출됩니다. |
| PowerPoint 파일 | .pptx | 마이크로소프트 파워포인트로 작성된 프레젠테이션입니다. 슬라이드별로 내용이 추출됩니다. |
| 텍스트 파일 | .txt | 일반 텍스트 파일입니다. 내용이 그대로 학습됩니다. |

**지원하지 않는 파일 형식:**

- 한글(.hwp) 파일은 지원하지 않습니다. 한글 파일은 PDF로 변환한 후 학습시키십시오.
- 이미지 파일(PNG, JPG 등)은 이 기능으로 학습시킬 수 없습니다.
- 압축 파일(.zip, .rar 등)은 압축을 해제한 후 개별 문서를 학습시키십시오.

### 3.3. 학습시킬 문서를 준비합니다

학습시킬 문서를 지정된 폴더에 복사하는 과정입니다. 아래 단계를 순서대로 따라 하십시오.

**Step 1. Finder를 엽니다.**

Mac 화면 하단의 Dock에서 Finder 아이콘(웃는 얼굴 모양)을 클릭합니다. 또는 키보드에서 Command + Space를 누르고 "Finder"를 입력한 후 Enter 키를 누릅니다.

**Step 2. 프로젝트 폴더로 이동합니다.**

Finder 상단 메뉴에서 "이동" > "폴더로 이동..."을 클릭합니다 (또는 키보드에서 Command + Shift + G를 누릅니다). 나타나는 입력란에 다음 경로를 입력합니다.

```text
<프로젝트 루트 폴더>/data/knowledges/docs/org/
```

예를 들어 프로젝트가 홈 폴더의 Developer 디렉터리에 있다면, 다음과 같이 입력합니다.

```text
~/Developer/dscore-ttc/data/knowledges/docs/org/
```

입력 후 "이동" 버튼을 클릭합니다.

**Step 3. 문서 파일을 복사합니다.**

이 폴더가 바로 AI에 학습시킬 문서를 넣어 두는 곳입니다. 학습시키고 싶은 문서 파일을 이 폴더 안으로 복사(또는 이동)하십시오.

복사 방법은 간단합니다. 학습시킬 문서가 있는 폴더를 별도의 Finder 창으로 열고, 해당 문서 파일을 마우스로 드래그하여 이 폴더에 놓으면 됩니다. 또는 문서 파일을 선택한 후 Command + C로 복사하고, 이 폴더에서 Command + V로 붙여 넣으셔도 됩니다.

**Step 4. 복사된 파일을 확인합니다.**

폴더 안에 학습시키려는 문서 파일들이 모두 들어 있는지 확인합니다. 파일명 옆에 표시되는 확장자(.pdf, .docx 등)가 위의 지원 형식 표에 해당하는지도 확인하십시오.

**파일에 관한 주의 사항:**

- 비밀번호가 걸린 PDF 파일은 내용을 추출할 수 없습니다. 비밀번호를 해제한 후 넣어 주십시오.
- 스캔한 이미지만으로 구성된 PDF 파일(글자를 선택할 수 없는 PDF)은 텍스트 추출이 제한될 수 있습니다.
- 파일 크기가 5MB를 초과하는 문서는 처리 시간이 길어질 수 있습니다. 가능하면 문서를 분할하여 넣는 것을 권장합니다.
- 파일명에 한글을 사용해도 됩니다. 다만, 특수문자(#, %, & 등)가 포함된 파일명은 오류를 유발할 수 있으므로 피하십시오.
- 하위 폴더를 만들어서 문서를 정리해도 됩니다. 하위 폴더 안의 문서도 모두 자동으로 인식됩니다.

### 3.4. Jenkins에서 학습을 실행합니다

문서가 준비되었으면, Jenkins에서 학습 작업을 실행합니다. 아래 단계를 순서대로 따라 하십시오.

**Step 1. 웹 브라우저를 엽니다.**

Safari 또는 Chrome 등 평소 사용하시는 웹 브라우저를 엽니다.

**Step 2. Jenkins에 접속합니다.**

브라우저의 주소 입력란에 다음 주소를 입력하고 Enter 키를 누릅니다.

```text
http://localhost:8080
```

로그인 화면이 나타나면, 아이디와 비밀번호를 입력하고 "로그인" 버튼을 클릭합니다.

**Step 3. Jenkins 대시보드에서 학습 작업을 찾습니다.**

로그인 후 나타나는 Jenkins 대시보드 화면에는 여러 개의 작업(Job) 목록이 표시됩니다. 이 목록에서 `DSCORE-Knowledge-Sync`라는 이름을 찾습니다.

목록이 길어서 한눈에 보이지 않을 수 있습니다. 이 경우 화면을 아래로 스크롤하거나, 대시보드 오른쪽 상단의 검색란에 "Knowledge"를 입력하면 해당 작업을 빠르게 찾을 수 있습니다.

**Step 4. 학습 작업을 클릭합니다.**

`DSCORE-Knowledge-Sync` 이름을 클릭합니다. 해당 작업의 상세 화면으로 이동합니다.

**Step 5. 학습을 시작합니다.**

화면 왼쪽 메뉴에서 "지금 빌드" (영문 환경에서는 "Build Now")를 클릭합니다.

이 작업은 별도의 입력값(파라미터)이 필요하지 않습니다. 앞서 폴더에 넣어 둔 모든 문서가 자동으로 처리됩니다.

**Step 6. 학습이 진행되는 것을 확인합니다.**

"지금 빌드"를 클릭하면, 화면 왼쪽 하단의 "Build History"(빌드 이력) 영역에 새로운 항목이 나타납니다.

- 항목 왼쪽에 회전하는 아이콘이 표시되면, 학습이 진행 중인 것입니다.
- 아이콘이 파란색 원으로 바뀌면, 학습이 성공적으로 완료된 것입니다.
- 아이콘이 빨간색 원으로 바뀌면, 학습 중 오류가 발생한 것입니다.

**Step 7. 완료를 기다립니다.**

학습에 걸리는 시간은 문서의 수와 크기에 따라 다릅니다.

- 문서 1~5개 (각 수 페이지): 약 1~2분
- 문서 10~20개: 약 3~5분
- 문서 30개 이상 또는 대용량 PDF: 약 5~10분

학습이 완료될 때까지 브라우저를 닫지 마시고 기다려 주십시오. 브라우저를 닫아도 학습 자체는 중단되지 않지만, 완료 여부를 확인하기 어려워집니다.

### 3.5. 학습 결과를 확인합니다

학습이 완료되면, 결과가 정상적으로 처리되었는지 확인합니다.

#### 3.5.1. Jenkins에서 실행 로그를 확인합니다

**Step 1. 빌드 번호를 클릭합니다.**

"Build History" 영역에서 방금 완료된 빌드 번호(예: `#5`)를 클릭합니다.

**Step 2. 실행 로그를 엽니다.**

해당 빌드의 상세 화면이 나타나면, 왼쪽 메뉴에서 "Console Output"(콘솔 출력)을 클릭합니다.

**Step 3. 로그 내용을 확인합니다.**

화면에 학습 과정의 상세 기록이 텍스트로 표시됩니다.

**정상적으로 완료된 경우의 로그 예시:**

```text
[DocProcessor] === [Hybrid Doc Processor] Convert Start ===
[DocProcessor] [Target] PDF Detected: 프로젝트_기획서.pdf
[DocProcessor] [Hybrid] Processing Start: 프로젝트_기획서.pdf (Total Pages: 12)
[DocProcessor]   Processing Page 1/12...
[DocProcessor]     [Step 1] Detected 2 tables.
[DocProcessor]     [Step 2] Extracted 15 text blocks and 1 images (filtered).
[DocProcessor]     [Step 3] Page 1 reconstruction complete.
...
[DocProcessor] [Success] 프로젝트_기획서.pdf -> 프로젝트_기획서.pdf.md (45.23s)
[DocProcessor] [Target] PDF Detected: API_스펙.pdf
...
[DocProcessor] [Success] API_스펙.pdf -> API_스펙.pdf.md (12.87s)
[DocProcessor] === [Hybrid Doc Processor] Convert Done ===
[DocProcessor] === [Hybrid Doc Processor] Upload Start ===
[DocProcessor] Attempting upload: 프로젝트_기획서.pdf.md (Dataset: abc123...)
[DocProcessor] [Upload:SUCCESS] 프로젝트_기획서.pdf.md | OK (ID: doc_001)
[DocProcessor] Attempting upload: API_스펙.pdf.md (Dataset: abc123...)
[DocProcessor] [Upload:SUCCESS] API_스펙.pdf.md | OK (ID: doc_002)
[DocProcessor] === [Hybrid Doc Processor] Upload Summary: Success=2, Fail=0 ===
[DocProcessor] === [Hybrid Doc Processor] Upload Done ===
Finished: SUCCESS
```

위와 같이 각 문서에 대해 `[Success]`와 `[Upload:SUCCESS]`가 표시되고, 마지막 줄에 `Finished: SUCCESS`가 나타나면 모든 문서가 정상적으로 학습된 것입니다.

**오류가 발생한 경우의 로그 예시:**

```text
[DocProcessor] === [Hybrid Doc Processor] Convert Start ===
[DocProcessor] [Target] PDF Detected: 보고서.pdf
[DocProcessor] [Hybrid] Processing Start: 보고서.pdf (Total Pages: 5)
...
[DocProcessor] [Success] 보고서.pdf -> 보고서.pdf.md (8.45s)
[DocProcessor] === [Hybrid Doc Processor] Convert Done ===
[DocProcessor] === [Hybrid Doc Processor] Upload Start ===
[DocProcessor] Attempting upload: 보고서.pdf.md (Dataset: abc123...)
[DocProcessor] [Upload:FAIL] 보고서.pdf.md | Reason: HTTP 401 - {"code":"unauthorized"}
[DocProcessor] === [Hybrid Doc Processor] Upload Summary: Success=0, Fail=1 ===
[DocProcessor] === [Hybrid Doc Processor] Upload Done ===
Finished: FAILURE
```

위와 같이 `[Upload:FAIL]`이 표시되고 마지막 줄에 `Finished: FAILURE`가 나타나면, 문서 변환은 되었지만 AI 지식 저장소로의 전송에 실패한 것입니다. 이 경우 "3.7. 자주 발생하는 문제와 해결 방법"을 참고하십시오.

#### 3.5.2. Dify에서 학습된 문서를 확인합니다

Jenkins 로그에서 성공을 확인한 후, Dify에서도 문서가 실제로 등록되었는지 확인할 수 있습니다.

**Step 1.** 웹 브라우저에서 `http://localhost/apps`에 접속하여 Dify에 로그인합니다.

**Step 2.** 상단 메뉴에서 "Knowledge"(또는 "지식")를 클릭합니다.

**Step 3.** 문서형 지식 저장소를 클릭합니다.

**Step 4.** 저장소 안에 방금 학습시킨 문서 파일 이름들이 목록에 표시되는지 확인합니다.

**Step 5.** 문서 이름을 클릭하면 AI가 저장한 내용의 미리보기를 확인할 수 있습니다.

#### 3.5.3. AI에게 질문하여 학습 결과를 테스트합니다

가장 확실한 확인 방법은 AI에게 직접 질문해 보는 것입니다.

**Step 1.** Dify에서 프로젝트 AI 챗봇 화면으로 이동합니다.

**Step 2.** 학습시킨 문서에 있는 내용에 대해 구체적인 질문을 입력합니다.

예시: "프로젝트 기획서에서 로그인 기능의 요구사항이 뭐라고 적혀 있어?"

**Step 3.** AI의 답변에 학습시킨 문서의 내용이 반영되어 있으면, 학습이 성공적으로 완료된 것입니다.

### 3.6. 문서가 갱신되었을 때 재학습하는 방법

프로젝트 문서가 수정되었을 때, AI에 최신 내용을 다시 학습시키는 방법입니다.

**Step 1. 기존 문서를 Dify에서 삭제합니다.**

동일한 파일명의 문서를 다시 학습시키면, AI 지식 저장소에 같은 내용이 중복으로 등록됩니다. 이를 방지하려면 먼저 기존 문서를 삭제해야 합니다.

1. Dify에 로그인합니다 (`http://localhost/apps`).
2. "Knowledge" 메뉴에서 문서형 지식 저장소를 클릭합니다.
3. 삭제할 문서 이름 옆의 메뉴(점 세 개 아이콘 또는 우클릭)를 클릭합니다.
4. "Delete"(삭제)를 선택합니다.
5. 확인 메시지가 나타나면 "확인"을 클릭합니다.

**Step 2. 수정된 문서를 폴더에 넣습니다.**

수정된 문서 파일을 `data/knowledges/docs/org/` 폴더에 복사합니다. 기존 파일이 있다면 덮어쓰기 하십시오.

**Step 3. Jenkins에서 학습을 다시 실행합니다.**

3.4절의 과정을 동일하게 반복하여 "지금 빌드"를 클릭합니다.

**Step 4. 학습 결과를 확인합니다.**

3.5절의 과정을 동일하게 반복하여 결과를 확인합니다.

### 3.7. 자주 발생하는 문제와 해결 방법

#### 문제 1: 빌드가 빨간색 원으로 표시되고 "Finished: FAILURE"가 나타남

**로그에 "HTTP 401 - unauthorized"가 포함된 경우:**

이 오류는 AI 지식 저장소의 접속 키가 잘못되었거나 만료된 경우에 발생합니다.

해결 방법:
1. Dify에 로그인합니다.
2. "Knowledge" > 해당 저장소 > "Settings"(설정)에서 새로운 접속 키를 발급받습니다.
3. Jenkins에서 "Jenkins 관리" > "Credentials"로 이동합니다.
4. `dify-knowledge-key` 항목을 클릭하고 "Update"(수정)를 선택합니다.
5. 새로 발급받은 접속 키를 입력하고 "Save"(저장)를 클릭합니다.
6. 학습을 다시 실행합니다.

**로그에 "HTTP 400 - doc_form mismatch"가 포함된 경우:**

이 오류는 저장소 유형과 학습 방식이 일치하지 않을 때 발생합니다. 예를 들어, 문서형 저장소에 질문답변형으로 업로드하려고 하면 이 오류가 발생합니다.

해결 방법:
1. Jenkins에서 "Jenkins 관리" > "Credentials"로 이동합니다.
2. `dify-dataset-id` 항목에 등록된 저장소 주소가 문서형 저장소의 것인지 확인합니다.
3. Dify에서 문서형 저장소의 설정을 열어 저장소 주소를 복사합니다.
4. 일치하지 않으면 올바른 주소로 수정합니다.

**로그에 "변환 결과 디렉터리가 없습니다"가 포함된 경우:**

이 오류는 문서 변환 과정에서 결과 파일을 저장할 폴더가 존재하지 않을 때 발생합니다.

해결 방법:
1. 시스템 관리자에게 "서비스가 정상적으로 구동되고 있는지 확인해 주십시오."라고 요청하십시오.
2. 서비스 재시작 후 다시 실행해 보십시오.

#### 문제 2: 빌드가 성공했지만, Dify에 문서가 보이지 않음

Jenkins 로그에서는 `Finished: SUCCESS`로 표시되었지만 Dify 저장소에 문서가 나타나지 않는 경우입니다.

해결 방법:
1. Dify의 지식 저장소 화면을 새로고침합니다 (브라우저에서 F5 키 또는 Command + R).
2. 문서가 등록 처리 중일 수 있습니다. 1~2분 기다린 후 다시 확인합니다.
3. 여전히 보이지 않으면, Jenkins 로그를 다시 확인하여 `[Upload:SUCCESS]` 메시지가 실제로 있는지 확인합니다.

#### 문제 3: 특정 파일만 변환에 실패함

일부 문서는 성공했지만 특정 파일만 오류가 발생하는 경우입니다.

해결 방법:
1. 해당 파일이 비밀번호로 보호되어 있지는 않은지 확인합니다.
2. 파일이 손상되지 않았는지 확인합니다 (해당 파일을 직접 열어 정상적으로 표시되는지 확인).
3. 파일명에 특수문자(#, %, & 등)가 포함되어 있다면 파일명을 변경합니다.
4. PowerPoint 파일(.pptx)의 변환이 실패하는 경우, 시스템 관리자에게 "변환 도구가 설치되어 있는지 확인해 주십시오."라고 요청하십시오.

#### 문제 4: 학습이 매우 오래 걸림 (30분 이상)

해결 방법:
1. 대용량 PDF 파일(수백 페이지)이 포함되어 있지 않은지 확인합니다.
2. 이미지가 많이 포함된 PDF는 이미지 분석에 시간이 소요됩니다. 텍스트 위주의 문서로 변경하거나, 문서를 분할하여 넣는 것을 권장합니다.
3. 30분이 지나도 완료되지 않으면, 빌드를 중지하고 문서를 줄여서 다시 시도하십시오.

빌드를 중지하는 방법: 빌드 번호 옆의 빨간색 X 아이콘을 클릭합니다.

#### 문제 5: Jenkins 접속 자체가 되지 않음

"http://localhost:8080"에 접속했을 때 화면이 표시되지 않는 경우입니다.

해결 방법:
1. 시스템 관리자에게 "Jenkins 서비스가 실행되고 있는지 확인해 주십시오."라고 요청하십시오.
2. 서비스가 재시작된 후 다시 접속을 시도합니다.

## 4. Job #2: 문서를 질문-답변 형태로 학습시키기

Job #1은 AI가 문서 원문을 그대로 참고하여 답변하는 방식입니다. Job #2는 이와 다르게, AI가 문서에서 미리 질문과 답변 쌍을 만들어 둡니다. 예를 들어, "로그인은 어떻게 하면 되나요?", "시스템 요구사항은 무엇인가요?" 같은 질문에 대한 답변을 사전에 준비해 둡니다.

두 가지의 차이를 비유하자면 다음과 같습니다.

- **Job #1 (문서형 학습):** 참고서 전체를 AI가 읽고, 질문을 받으면 해당 부분을 찾아서 답변합니다. "이 문서에 뭐라고 적혀 있어?" 같은 질문에 적합합니다.
- **Job #2 (질문답변형 학습):** AI가 참고서를 읽고 미리 예상 질문과 답변을 정리해 둡니다. "이건 어떻게 하면 돼?" 같은 질문에 더 정확하게 답변합니다.

**두 가지를 모두 실행하면 AI의 답변 품질이 가장 높아집니다.** 따라서 Job #1을 실행한 후 Job #2도 실행하는 것을 권장합니다.

### 4.1. 언제 사용합니까

이 기능은 Job #1(문서 학습)을 실행한 후에 추가로 실행합니다. 다음과 같은 상황에서 사용하십시오.

**Job #1을 처음 실행한 직후:**

문서형 학습을 완료한 후, 질문답변형 학습을 추가로 실행하면 AI의 답변 정확도가 크게 향상됩니다.

**AI의 답변이 질문과 잘 맞지 않을 때:**

Job #1만 실행한 상태에서 AI에게 "~는 어떻게 하면 돼?", "~의 절차가 뭐야?" 같은 질문을 했을 때 답변이 부정확하다면, Job #2를 추가로 실행하여 개선할 수 있습니다.

**문서를 재학습한 후:**

Job #1에서 문서를 재학습(갱신)한 경우, Job #2도 동일하게 재실행하여 질문-답변 내용을 최신 상태로 유지하는 것이 좋습니다.

### 4.2. Jenkins에서 학습을 실행합니다

문서 준비는 Job #1과 동일한 폴더를 사용하므로, 별도의 파일 준비가 필요하지 않습니다. Job #1에서 넣어 둔 문서가 그대로 사용됩니다.

**Step 1. 웹 브라우저를 엽니다.**

Safari 또는 Chrome을 엽니다.

**Step 2. Jenkins에 접속합니다.**

주소 입력란에 `http://localhost:8080`을 입력하고 Enter 키를 누릅니다. 로그인 화면이 나타나면, 아이디와 비밀번호를 입력하고 "로그인" 버튼을 클릭합니다.

**Step 3. 질문답변형 학습 작업을 찾습니다.**

Jenkins 대시보드에서 `DSCORE-Knowledge-Sync-QA`라는 이름을 찾습니다. 검색란에 "QA"를 입력하면 빠르게 찾을 수 있습니다.

**Step 4. 작업 이름을 클릭합니다.**

`DSCORE-Knowledge-Sync-QA`를 클릭하여 작업 상세 화면으로 이동합니다.

**Step 5. 학습을 시작합니다.**

왼쪽 메뉴에서 "지금 빌드"(Build Now)를 클릭합니다. 이 작업도 Job #1과 마찬가지로 별도의 입력값이 필요하지 않습니다.

**Step 6. 완료를 기다립니다.**

Build History에서 회전 아이콘이 파란색 원으로 바뀔 때까지 기다립니다. 소요 시간은 문서 분량에 따라 1~10분 정도입니다.

### 4.3. 학습 결과를 확인합니다

#### Jenkins 로그 확인

1. Build History에서 빌드 번호를 클릭합니다.
2. "Console Output"을 클릭합니다.
3. 로그 내용을 확인합니다.

**정상 완료 시의 로그 예시:**

```text
[DocProcessor] === [Hybrid Doc Processor] Convert Start ===
[DocProcessor] [Target] PDF Detected: 프로젝트_기획서.pdf
...
[DocProcessor] [Success] 프로젝트_기획서.pdf -> 프로젝트_기획서.pdf.md (45.23s)
[DocProcessor] === [Hybrid Doc Processor] Convert Done ===
[DocProcessor] === [Hybrid Doc Processor] Upload Start ===
[DocProcessor] Attempting upload: 프로젝트_기획서.pdf.md (Dataset: def456...)
[DocProcessor] [Upload:SUCCESS] 프로젝트_기획서.pdf.md | OK (ID: qa_001)
[DocProcessor] === [Hybrid Doc Processor] Upload Summary: Success=1, Fail=0 ===
[DocProcessor] === [Hybrid Doc Processor] Upload Done ===
Finished: SUCCESS
```

`Finished: SUCCESS`가 표시되면 정상입니다.

#### Dify에서 확인

1. Dify에 로그인합니다 (`http://localhost/apps`).
2. "Knowledge" 메뉴를 클릭합니다.
3. 이번에는 **질문답변형 지식 저장소**(문서형 저장소가 아닌 다른 하나)를 클릭합니다.
4. 저장소 안에 문서 이름들이 표시되는지 확인합니다.
5. 문서를 클릭하면 AI가 자동으로 생성한 질문-답변 쌍의 목록을 미리보기로 확인할 수 있습니다.

### 4.4. 자주 발생하는 문제와 해결 방법

#### 문제 1: 빌드가 실패하고 "HTTP 401 - unauthorized"가 나타남

이 경우 질문답변형 저장소의 접속 키가 잘못된 것입니다. 문서형 저장소와 질문답변형 저장소의 접속 키는 서로 다릅니다.

해결 방법:
1. Dify에서 **질문답변형** 지식 저장소의 설정에서 접속 키를 새로 발급받습니다.
2. Jenkins에서 "Jenkins 관리" > "Credentials"로 이동합니다.
3. `dify-knowledge-key-qa` 항목(문서형의 `dify-knowledge-key`가 아닌 점에 주의)을 클릭하고 수정합니다.
4. 새 접속 키를 입력하고 저장한 후, 학습을 다시 실행합니다.

#### 문제 2: 빌드가 실패하고 "HTTP 400 - doc_form mismatch"가 나타남

이 오류는 질문답변형 저장소 주소가 잘못 등록된 경우에 발생합니다.

해결 방법:
1. Jenkins의 "Credentials"에서 `dify-dataset-id-qa` 항목을 확인합니다.
2. Dify에서 **질문답변형** 저장소의 주소를 복사합니다.
3. 값이 다르면 올바른 주소로 수정합니다.

#### 문제 3: 빌드는 성공했지만 질문-답변 쌍이 생성되지 않음

Dify의 질문답변형 저장소에 문서는 등록되었지만, 질문-답변 쌍이 아직 보이지 않는 경우입니다.

해결 방법:
1. Dify에서 질문-답변 생성에는 시간이 소요됩니다. 5~10분 정도 기다린 후 다시 확인합니다.
2. Dify의 해당 문서 항목에서 상태가 "처리 중"으로 표시될 수 있습니다. "완료"로 변경될 때까지 기다립니다.

#### 문제 4: Job #1은 성공했는데 Job #2만 실패함

두 작업은 동일한 문서를 사용하지만, 서로 다른 저장소에 업로드합니다. 따라서 접속 키와 저장소 주소가 각각 별도로 올바르게 등록되어 있어야 합니다.

해결 방법:
1. Jenkins의 "Credentials"에서 `dify-knowledge-key-qa`와 `dify-dataset-id-qa` 두 항목이 모두 존재하는지 확인합니다.
2. 두 항목의 값이 질문답변형 저장소에 대한 것인지 확인합니다 (문서형 저장소의 값과 혼동하지 않도록 주의).

## 5. Job #3: 소스 코드 구조를 AI에 학습시키기

Git 저장소에 있는 프로젝트의 폴더 구조와 주요 설정 파일을 AI가 분석합니다. 이후 "이 프로젝트는 어떤 구조로 되어 있어?", "설정 파일에 뭐가 적혀 있어?", "이 프로젝트는 어떤 프로그래밍 언어를 사용해?" 같은 질문에 AI가 답변할 수 있게 됩니다.

쉽게 말해, 프로젝트의 소스 코드가 저장된 저장소를 AI에게 보여 주면, AI가 프로젝트의 전체적인 구조를 파악하고 기억하는 것입니다.

### 5.1. 언제 사용합니까

이 기능은 다음과 같은 상황에서 사용합니다.

**프로젝트의 코드 구조를 AI에 알려 주고 싶을 때:**

AI가 프로젝트의 폴더 구조, 사용하는 프로그래밍 언어, 주요 설정 내용을 알게 되면, 코드 관련 질문에 훨씬 정확하게 답변할 수 있습니다.

**신규 팀원이 프로젝트 구조를 파악해야 할 때:**

새로운 팀원이 "이 프로젝트는 어떤 폴더에 어떤 코드가 있어?", "메인 설정은 어디서 관리해?" 같은 질문을 AI에게 할 수 있도록 사전에 학습시켜 둡니다.

**코드 구조에 큰 변경이 있었을 때:**

프로젝트 구조를 대폭 변경하거나 모듈을 추가/삭제한 경우, 최신 구조를 AI에 다시 학습시킵니다.

### 5.2. 실행에 필요한 정보를 준비합니다

이 작업을 실행하려면, 분석할 프로젝트의 Git 저장소 주소가 필요합니다. 이 주소는 GitLab에서 확인할 수 있습니다.

**Step 1. GitLab에 접속합니다.**

웹 브라우저에서 사내 GitLab 주소를 입력합니다. 일반적으로 `http://localhost:8929`입니다. 로그인 화면이 나타나면 아이디와 비밀번호를 입력합니다.

**Step 2. 분석할 프로젝트를 찾습니다.**

GitLab 대시보드에서 분석하려는 프로젝트를 클릭합니다. 프로젝트 목록에서 찾을 수 없으면, 상단 검색란에 프로젝트 이름을 입력하여 검색합니다.

**Step 3. 저장소 주소를 복사합니다.**

프로젝트 메인 화면에서 파란색 "Clone" 버튼을 클릭합니다. 드롭다운 메뉴가 나타나면, "Clone with HTTP" 항목의 주소를 복사합니다.

복사된 주소는 대략 다음과 같은 형태입니다.

```text
http://localhost:8929/root/my-project.git
```

**Step 4. 주소의 호스트 부분을 변경합니다.**

복사한 주소에서 `localhost:8929` 부분을 `gitlab:8929`로 변경합니다. 이는 학습 작업이 실행되는 환경에서 GitLab에 접근하기 위한 내부 주소입니다.

변경 전: `http://localhost:8929/root/my-project.git`
변경 후: `http://gitlab:8929/root/my-project.git`

이 주소를 메모장이나 클립보드에 복사해 둡니다.

### 5.3. Jenkins에서 학습을 실행합니다

**Step 1. 웹 브라우저에서 Jenkins에 접속합니다.**

`http://localhost:8080`에 접속하고 로그인합니다.

**Step 2. 코드 학습 작업을 찾습니다.**

Jenkins 대시보드에서 `DSCORE-Code-Knowledge-Sync`라는 이름을 찾습니다. 검색란에 "Code"를 입력하면 빠르게 찾을 수 있습니다.

**Step 3. 작업 이름을 클릭합니다.**

`DSCORE-Code-Knowledge-Sync`를 클릭하여 작업 상세 화면으로 이동합니다.

**Step 4. "Build with Parameters"를 클릭합니다.**

이 작업은 앞서의 Job #1, #2와 다르게 입력값이 필요합니다. 왼쪽 메뉴에서 "Build with Parameters"(파라미터와 함께 빌드)를 클릭합니다.

**Step 5. 저장소 주소를 입력합니다.**

화면에 입력란이 나타납니다. `REPO_URL` 항목에 앞서 준비한 Git 저장소 주소를 입력합니다.

| 입력 항목 | 입력할 내용 |
| --- | --- |
| REPO_URL | `http://gitlab:8929/root/my-project.git` (앞서 복사한 주소) |

주소를 입력할 때 앞뒤에 불필요한 공백이 들어가지 않도록 주의하십시오.

**Step 6. "빌드" 버튼을 클릭합니다.**

주소 입력이 완료되면, 화면 하단의 "빌드" (영문: "Build") 버튼을 클릭합니다.

**Step 7. 완료를 기다립니다.**

Build History에서 회전 아이콘이 파란색 원으로 바뀔 때까지 기다립니다. 소요 시간은 프로젝트 규모에 따라 약 1~3분입니다.

### 5.4. 학습 결과를 확인합니다

#### Jenkins 로그 확인

1. Build History에서 빌드 번호를 클릭합니다.
2. "Console Output"을 클릭합니다.

**정상 완료 시의 로그 예시:**

```text
[Clone] Git 레포지토리 클론 시작: http://gitlab:8929/root/my-project.git
[Clone] 완료
[Build] 레포지토리 컨텍스트 생성 시작
[Build] 완료
[Upload] 코드 컨텍스트 전송 시작
[Upload:SUCCESS] context_my-project.md | OK (ID: code_001)
[Upload] 완료
[Success] 코드 지식화 작업이 성공적으로 완료되었습니다.
Finished: SUCCESS
```

`Finished: SUCCESS`가 표시되면 정상입니다.

#### AI가 분석하는 내용

코드 구조 학습을 실행하면, AI는 다음과 같은 정보를 자동으로 분석하고 기억합니다.

- 프로젝트의 전체 폴더 및 파일 구조 (어떤 폴더에 어떤 파일이 있는지)
- 프로젝트 설명 파일 (README)
- 패키지 및 라이브러리 의존성 정보 (프로젝트가 어떤 외부 도구를 사용하는지)
- 빌드 및 배포 설정 파일
- 환경 설정 파일
- 프로젝트가 사용하는 프로그래밍 언어 및 프레임워크

#### Dify에서 확인

1. Dify에 로그인합니다.
2. "Knowledge" > 문서형 지식 저장소를 클릭합니다.
3. `context_프로젝트명.md` 형태의 문서가 등록되어 있는지 확인합니다.

### 5.5. 자주 발생하는 문제와 해결 방법

#### 문제 1: 빌드가 실패하고 "Could not resolve host: gitlab"이 나타남

학습 작업이 실행되는 환경에서 GitLab 서버에 접근할 수 없을 때 발생하는 오류입니다.

로그에 다음과 유사한 메시지가 표시됩니다.

```text
fatal: unable to access 'http://gitlab:8929/root/my-project.git/':
Could not resolve host: gitlab
```

해결 방법:
1. GitLab 서비스가 실행 중인지 확인합니다. 브라우저에서 `http://localhost:8929`에 접속하여 GitLab이 정상적으로 표시되는지 확인합니다.
2. GitLab이 정상적이라면, 시스템 관리자에게 "Jenkins와 GitLab 간의 네트워크 연결을 확인해 주십시오."라고 요청합니다.

#### 문제 2: 빌드가 실패하고 "repository not found" 또는 "authentication failed"가 나타남

저장소 주소가 잘못되었거나, 해당 저장소에 접근 권한이 없을 때 발생합니다.

해결 방법:
1. 입력한 저장소 주소가 정확한지 다시 확인합니다.
2. 주소의 호스트 부분이 `gitlab:8929`인지 확인합니다 (`localhost:8929`로 입력하면 오류가 발생합니다).
3. 해당 저장소가 비공개(Private) 저장소인 경우, 시스템 관리자에게 접근 권한 설정을 요청합니다.

#### 문제 3: REPO_URL을 비워 둔 채로 빌드함

REPO_URL 입력란을 비워 둔 상태에서 빌드를 실행하면, Git 저장소를 내려받는 단계가 건너뛰어지며 이후 단계에서 오류가 발생할 수 있습니다.

해결 방법:
1. 반드시 REPO_URL에 유효한 저장소 주소를 입력한 후 빌드를 실행하십시오.
2. 빌드를 다시 실행할 때 "Build with Parameters"를 클릭하여 주소를 입력합니다.

#### 문제 4: 분석 결과에 프로젝트 파일이 거의 포함되지 않음

코드 학습이 성공했지만, AI에게 프로젝트 구조를 물었을 때 매우 적은 정보만 답변하는 경우입니다.

해결 방법:
1. 저장소의 기본 브랜치(보통 main 또는 master)에 코드가 정상적으로 포함되어 있는지 확인합니다.
2. 프로젝트에 README 파일이 있으면 AI가 더 풍부한 컨텍스트를 확보할 수 있습니다. README 파일을 작성하는 것을 권장합니다.

## 6. Job #5: 웹사이트 내용을 AI에 학습시키기

사내 위키, 기술 블로그, 외부 참고 문서 등 웹사이트의 텍스트 내용을 AI가 수집하여 학습합니다. 웹 페이지에 있는 유용한 정보를 AI가 읽고 기억하여, 이후 관련 질문에 답변할 수 있게 됩니다.

예를 들어, 사내 위키에 정리된 개발 가이드를 학습시키면 "우리 팀의 코딩 컨벤션이 뭐야?", "배포 절차가 어떻게 돼?" 같은 질문에 위키 내용을 기반으로 AI가 답변합니다.

### 6.1. 언제 사용합니까

이 기능은 다음과 같은 상황에서 사용합니다.

**사내 위키의 내용을 AI에 학습시킬 때:**

사내 위키에 정리된 개발 가이드, 운영 매뉴얼, 업무 프로세스 문서 등을 AI가 학습하면, 팀원들이 위키를 직접 찾아보지 않아도 AI에게 질문하여 빠르게 답변을 받을 수 있습니다.

**기술 블로그의 내용을 참고 자료로 학습시킬 때:**

프로젝트에 참고가 되는 기술 블로그 글이 있다면, 해당 페이지를 AI에 학습시킬 수 있습니다.

**외부 기술 문서를 학습시킬 때:**

프로젝트에서 사용하는 프레임워크나 라이브러리의 공식 문서 페이지를 학습시키면, AI가 해당 기술에 대한 질문에 더 정확하게 답변할 수 있습니다.

### 6.2. Jenkins에서 수집을 실행합니다

**Step 1. 웹 브라우저에서 Jenkins에 접속합니다.**

`http://localhost:8080`에 접속하고 로그인합니다.

**Step 2. 웹 수집 작업을 찾습니다.**

Jenkins 대시보드에서 `DSCORE-Web-Knowledge-Sync`라는 이름을 찾습니다. 검색란에 "Web"을 입력하면 빠르게 찾을 수 있습니다.

**Step 3. 작업 이름을 클릭합니다.**

`DSCORE-Web-Knowledge-Sync`를 클릭하여 작업 상세 화면으로 이동합니다.

**Step 4. "Build with Parameters"를 클릭합니다.**

이 작업은 수집할 웹사이트 주소를 입력해야 합니다. 왼쪽 메뉴에서 "Build with Parameters"를 클릭합니다.

**Step 5. 웹사이트 주소를 입력합니다.**

화면에 입력란이 나타납니다. `ROOT_URL` 항목에 수집하려는 웹사이트의 주소를 입력합니다.

| 입력 항목 | 입력할 내용 |
| --- | --- |
| ROOT_URL | 수집하려는 웹 페이지의 전체 주소 |

입력 예시:
- 사내 위키 페이지: `http://wiki.company.com/development-guide`
- 기술 블로그 글: `https://techblog.example.com/2024/best-practices`
- 공식 문서 페이지: `https://docs.framework.com/getting-started`

**Step 6. "빌드" 버튼을 클릭합니다.**

주소 입력이 완료되면, 화면 하단의 "빌드" 버튼을 클릭합니다.

**Step 7. 승인 절차를 진행합니다.**

이 작업에는 보안을 위한 승인 단계가 포함되어 있습니다. 웹 수집이 완료되면, 수집된 내용을 AI 지식 저장소에 업로드하기 전에 승인을 요청하는 화면이 나타납니다.

빌드 상세 화면에서 "승인 및 업로드" 버튼이 나타나면, 수집된 내용을 확인한 후 해당 버튼을 클릭합니다.

**Step 8. 완료를 기다립니다.**

승인 후 업로드가 진행됩니다. Build History에서 파란색 원이 나타나면 완료입니다. 소요 시간은 수집할 페이지 수에 따라 약 1~5분입니다.

### 6.3. 수집 결과를 확인합니다

#### Jenkins 로그 확인

1. Build History에서 빌드 번호를 클릭합니다.
2. "Console Output"을 클릭합니다.

**정상 완료 시의 로그 예시:**

```text
[Crawl-Universal] Phase 1: 타겟 분석 시작 - https://docs.example.com
[Crawl-Universal] Selected best content area (Length: 8523 chars)
[Crawl-Universal] Removed 12 noise elements.
[Crawl-Universal] Extracted text length: 9834 chars
```

### 6.4. 주의 사항

웹사이트 수집 기능을 사용할 때 다음 사항을 반드시 유의하십시오.

**외부 웹사이트의 이용 규정을 확인하십시오.**

외부 웹사이트의 내용을 수집할 때는 해당 사이트의 이용약관과 수집 허용 정책을 반드시 확인하십시오. 일부 웹사이트는 자동 수집을 금지하고 있을 수 있습니다. 확인이 어려운 경우, 사내 법무 담당자에게 문의하시는 것을 권장합니다.

**대상 서버에 과도한 부하를 주지 마십시오.**

수집 범위가 넓은 웹사이트(수백 페이지 이상)를 한꺼번에 수집하면, 해당 웹사이트 서버에 부하를 줄 수 있습니다. 가능하면 특정 페이지나 섹션 단위로 나누어 수집하십시오.

**로그인이 필요한 사이트는 수집이 제한됩니다.**

사내 시스템 중 아이디와 비밀번호를 입력해야 접근할 수 있는 페이지는 자동 수집이 불가능합니다. 이 경우 해당 페이지의 내용을 수동으로 텍스트 파일(.txt)이나 Word 파일(.docx)로 저장한 후, Job #1(문서 학습)을 통해 학습시키는 방법을 사용하십시오.

**수집된 내용에 민감 정보가 포함될 수 있습니다.**

웹 페이지에 개인정보, 내부 보안 정보 등이 포함되어 있을 수 있습니다. 수집 후 승인 단계에서 내용을 검토하고, 민감 정보가 포함된 경우 해당 빌드를 취소하십시오.

### 6.5. 자주 발생하는 문제와 해결 방법

#### 문제 1: 빌드가 실패하고 웹 페이지 접속 오류가 나타남

입력한 웹사이트 주소에 접근할 수 없을 때 발생합니다.

로그에 다음과 유사한 메시지가 표시됩니다.

```text
[Crawl-Universal] Phase 1: 타겟 분석 시작 - https://example.com
[Crawl-Universal] Error: Navigation failed because page crashed
```

해결 방법:
1. 입력한 주소가 정확한지 확인합니다. 웹 브라우저에서 해당 주소에 직접 접속하여 페이지가 정상적으로 표시되는지 확인합니다.
2. `https://`가 아닌 `http://`로 접속해야 하는 사이트인지 확인합니다.
3. 사내 네트워크에서만 접근 가능한 사이트의 경우, 서비스가 실행되고 있는 환경에서도 해당 사이트에 접근할 수 있는지 확인합니다.

#### 문제 2: 수집은 되었지만 내용이 비어 있거나 매우 짧음

웹 페이지를 수집했지만 의미 있는 텍스트가 추출되지 않은 경우입니다.

로그에 "무시됨: 텍스트가 너무 짧음"이 대부분의 페이지에 표시됩니다.

해결 방법:
1. 해당 웹 페이지가 이미지 위주로 구성되어 있지는 않은지 확인합니다. 이미지 안의 텍스트는 추출되지 않습니다.
2. 웹 페이지가 접속 후 별도의 동작(스크롤, 클릭 등)을 해야 내용이 표시되는 구조인 경우, 수집이 정상적으로 되지 않을 수 있습니다.
3. 이 경우 해당 페이지의 내용을 수동으로 복사하여 텍스트 파일로 저장한 후, Job #1을 통해 학습시키는 방법을 사용하십시오.

#### 문제 3: 수집된 내용에 메뉴, 광고 등 불필요한 텍스트가 포함됨

수집 과정에서 본문이 아닌 메뉴, 광고 문구, 바닥글 등의 내용이 함께 추출되는 경우입니다.

해결 방법:
1. 수집 도구는 자동으로 메뉴, 광고, 바닥글 등을 제거하려고 시도하지만, 모든 웹사이트에서 완벽하게 작동하지는 않습니다.
2. 특정 페이지의 정확한 본문만 필요한 경우, 해당 페이지에서 필요한 부분만 수동으로 복사하여 텍스트 파일로 저장한 후 Job #1을 사용하는 것이 더 정확합니다.

#### 문제 4: 승인 단계에서 멈추어 있음

빌드가 진행 중으로 표시되며, 승인을 기다리는 상태에서 멈추어 있는 경우입니다.

해결 방법:
1. 빌드 상세 화면으로 이동합니다 (Build History에서 빌드 번호를 클릭).
2. 화면에 "승인 및 업로드" 또는 "Abort"(중단) 버튼이 표시됩니다.
3. 수집된 내용이 적절하면 "승인 및 업로드"를 클릭합니다.
4. 수집을 취소하려면 "Abort"를 클릭합니다.

## 7. 전체 학습을 처음부터 진행하는 경우

프로젝트에 처음으로 AI 지식 학습을 진행하는 경우, 다음 순서를 따라 전체 과정을 진행하십시오. 각 단계는 순서대로 실행하는 것을 권장하며, 해당되지 않는 단계는 건너뛰셔도 됩니다.

### Step 1. 사전 준비를 확인합니다 (약 10분)

본 가이드의 "2. 시작하기 전에 확인할 사항"을 참고하여, 4가지 사전 준비 항목을 모두 확인합니다.

확인할 항목:
- Jenkins 로그인 가능 여부
- Dify 지식 저장소 생성 여부
- Jenkins 연결 설정 등록 여부

모든 항목이 확인되면 다음 단계로 진행합니다.

### Step 2. 학습시킬 문서를 폴더에 넣습니다 (약 5~15분)

프로젝트 관련 문서(기획서, 설계서, 매뉴얼, 보고서 등)를 다음 폴더에 복사합니다.

```text
<프로젝트 루트 폴더>/data/knowledges/docs/org/
```

넣을 수 있는 파일: PDF, Word(.docx), Excel(.xlsx, .xls), PowerPoint(.pptx), 텍스트(.txt)

### Step 3. Job #1을 실행합니다 - 문서 학습 (약 1~10분)

Jenkins에서 `DSCORE-Knowledge-Sync` 작업을 실행합니다.

1. Jenkins(`http://localhost:8080`)에 로그인합니다.
2. `DSCORE-Knowledge-Sync` 작업을 클릭합니다.
3. "지금 빌드"를 클릭합니다.
4. 빌드가 파란색 원으로 표시될 때까지 기다립니다.
5. Console Output에서 `Finished: SUCCESS`를 확인합니다.

### Step 4. Job #2를 실행합니다 - 질문답변형 학습 (약 1~10분)

Jenkins에서 `DSCORE-Knowledge-Sync-QA` 작업을 실행합니다.

1. Jenkins 대시보드로 돌아갑니다 (화면 좌측 상단의 "Jenkins" 로고를 클릭).
2. `DSCORE-Knowledge-Sync-QA` 작업을 클릭합니다.
3. "지금 빌드"를 클릭합니다.
4. 빌드가 파란색 원으로 표시될 때까지 기다립니다.
5. Console Output에서 `Finished: SUCCESS`를 확인합니다.

### Step 5. Job #3을 실행합니다 - 소스 코드 구조 학습 (Git 저장소가 있는 경우에만, 약 1~3분)

프로젝트의 소스 코드가 GitLab에 저장되어 있다면, 이 단계를 실행합니다. Git 저장소가 없으면 건너뛰십시오.

1. GitLab에서 저장소 주소를 확인합니다 (5.2절 참조).
2. Jenkins 대시보드로 돌아갑니다.
3. `DSCORE-Code-Knowledge-Sync` 작업을 클릭합니다.
4. "Build with Parameters"를 클릭합니다.
5. REPO_URL에 저장소 주소를 입력합니다.
6. "빌드" 버튼을 클릭합니다.
7. 빌드가 파란색 원으로 표시될 때까지 기다립니다.

### Step 6. Job #5를 실행합니다 - 웹사이트 학습 (참고할 웹 문서가 있는 경우에만, 약 1~5분)

사내 위키나 기술 문서 웹사이트를 학습시키려면, 이 단계를 실행합니다. 해당 사항이 없으면 건너뛰십시오.

1. Jenkins 대시보드로 돌아갑니다.
2. `DSCORE-Web-Knowledge-Sync` 작업을 클릭합니다.
3. "Build with Parameters"를 클릭합니다.
4. ROOT_URL에 웹사이트 주소를 입력합니다.
5. "빌드" 버튼을 클릭합니다.
6. 승인 요청이 나타나면 "승인 및 업로드"를 클릭합니다.
7. 빌드가 파란색 원으로 표시될 때까지 기다립니다.

| 단계 | 작업 | 예상 소요 시간 |
| --- | --- | --- |
| Step 1 | 사전 준비 확인 | 약 10분 |
| Step 2 | 문서 폴더에 복사 | 약 5~15분 |

이후에는 변경된 항목에 해당하는 작업만 재실행하면 됩니다. 예를 들어, 기획서가 수정되었으면 Job #1만 다시 실행하고, 코드 구조가 변경되었으면 Job #3만 다시 실행합니다.

## 8. 학습된 지식을 활용하는 방법

모든 학습이 완료되면, AI에게 프로젝트에 대한 질문을 하여 학습된 지식을 활용할 수 있습니다. 아래 과정을 통해 AI 챗봇을 사용하는 방법을 안내합니다.

### 9.1. AI 챗봇에 접속합니다

**Step 1. 웹 브라우저를 엽니다.**

Safari 또는 Chrome을 엽니다.

**Step 2. Dify에 접속합니다.**

주소 입력란에 `http://localhost/apps`를 입력하고 Enter 키를 누릅니다. 로그인 화면이 나타나면 아이디와 비밀번호를 입력합니다.

**Step 3. AI 챗봇 앱을 찾습니다.**

Dify 대시보드에 생성된 앱 목록이 표시됩니다. 프로젝트용 AI 챗봇 앱을 클릭합니다. 앱 이름은 시스템 관리자가 설정한 이름으로, 프로젝트 이름이 포함되어 있을 수 있습니다.

**Step 4. 채팅 화면을 엽니다.**

앱 화면에서 "미리보기" 또는 화면 오른쪽의 채팅 아이콘을 클릭하면, AI와 대화할 수 있는 채팅 화면이 나타납니다.

### 9.2. AI에게 질문합니다

채팅 화면 하단의 입력란에 질문을 입력하고 Enter 키를 누르거나, 보내기 버튼을 클릭합니다.

**질문 예시와 기대 답변:**

질문 1: "우리 프로젝트의 전체 구조가 어떻게 되어 있어?"

기대 답변: AI가 Job #3에서 학습한 소스 코드 구조를 기반으로, 프로젝트의 폴더 구조와 주요 파일에 대해 설명합니다.

질문 2: "기획서에서 회원가입 기능의 요구사항이 뭐야?"

기대 답변: AI가 Job #1에서 학습한 기획서 문서를 참조하여, 회원가입 관련 요구사항을 답변합니다.

질문 3: "결재 프로세스가 어떤 순서로 진행돼?"

기대 답변: AI가 학습된 문서 내용을 기반으로, 결재 프로세스의 단계를 설명합니다.

질문 4: "설정 파일에서 데이터베이스 접속 정보가 어떻게 되어 있어?"

기대 답변: AI가 Job #3에서 학습한 설정 파일 내용을 참조하여, 데이터베이스 관련 설정을 답변합니다.

질문 5: "개발 가이드에서 코드 작성 규칙이 뭐라고 적혀 있어?"

기대 답변: AI가 Job #5에서 학습한 사내 위키 내용을 참조하여, 코딩 규칙을 답변합니다.

### 9.3. 더 좋은 답변을 받기 위한 요령

AI에게 질문할 때 다음 요령을 활용하면 더 정확하고 유용한 답변을 받을 수 있습니다.

**구체적으로 질문하십시오.**

- 좋지 않은 예: "로그인 어떻게 해?"
- 좋은 예: "우리 프로젝트에서 로그인 기능의 구현 방식이 기획서에 어떻게 정의되어 있어?"

구체적인 질문일수록 AI가 학습된 문서에서 정확한 부분을 찾아 답변합니다.

**문서나 파일을 지정하여 질문하십시오.**

- 좋지 않은 예: "인증 방식이 뭐야?"
- 좋은 예: "프로젝트 기획서에서 인증 방식에 대해 뭐라고 적혀 있어?"

특정 문서를 지정하면 AI가 해당 문서의 내용을 우선적으로 참조합니다.

**후속 질문을 활용하십시오.**

AI의 답변이 충분하지 않다면, 후속 질문을 통해 더 깊은 정보를 얻을 수 있습니다.

- 첫 번째 질문: "프로젝트의 기술 스택이 뭐야?"
- AI 답변: "프로젝트는 Spring Boot, React, PostgreSQL을 사용하고 있습니다."
- 후속 질문: "Spring Boot의 버전은 몇이야? 그리고 어떤 설정 파일에서 확인할 수 있어?"

**기대하는 답변 형식을 알려 주십시오.**

- "간단히 요약해 줘"
- "표로 정리해 줘"
- "단계별로 설명해 줘"

이와 같이 원하는 답변 형식을 함께 요청하면, AI가 읽기 편한 형태로 답변을 구성합니다.

### 9.4. AI 답변이 기대와 다를 때

AI의 답변이 학습시킨 내용과 다르거나 부정확한 경우, 다음 사항을 확인하십시오.

**해당 내용이 실제로 학습되었는지 확인합니다.**

1. Dify의 "Knowledge" 메뉴에서 지식 저장소를 열어, 해당 문서가 등록되어 있는지 확인합니다.
2. 문서가 등록되어 있지 않다면, 해당 문서를 폴더에 넣고 학습을 다시 실행합니다.

**질문을 다시 표현하여 시도합니다.**

같은 내용이라도 질문 방식에 따라 AI의 답변 정확도가 달라질 수 있습니다. 같은 질문을 다른 방식으로 다시 해 보십시오.

- 방법 1: "기획서에 인증 방식이 뭐라고 적혀 있어?"
- 방법 2: "프로젝트의 인증 방식에 대해 설명해 줘"
- 방법 3: "로그인 기능의 보안 요구사항이 뭐야?"

**학습 데이터를 보강합니다.**

AI의 답변이 지속적으로 부정확하다면, 관련 문서를 추가하거나 기존 문서의 내용을 보완한 후 다시 학습시키는 것을 권장합니다.

| Credentials | Jenkins에 저장된 접속 정보(키, 주소 등)입니다. AI 지식 저장소에 접근하기 위해 필요합니다. |
| Git 저장소 | 프로젝트의 소스 코드가 버전별로 관리되는 저장소입니다. GitLab에서 관리됩니다. |
| GitLab | 소스 코드 저장소를 제공하는 도구입니다. 프로젝트의 코드를 팀원들과 공유하고 관리합니다. |
