# 품질 분석 파이프라인 사용 가이드 (Job #4)

---

## 1. 이 기능으로 할 수 있는 것

코드 품질 분석 도구(SonarQube)는 프로젝트의 소스 코드를 자동으로 검사하여, 버그 가능성이 있는 코드나 보안상 위험한 코드를 찾아냅니다. 그러나 분석 결과를 확인하고, 각 문제의 원인을 이해하고, 이슈 관리 시스템(GitLab)에 수정 요청을 등록하는 과정은 지금까지 모두 수작업이었습니다.

이 품질 분석 파이프라인은 그 전체 과정을 자동화합니다. 코드 품질 분석 도구가 발견한 문제들을 AI가 읽고, 무엇이 잘못되었는지 알기 쉬운 말로 설명하고, 어떻게 고쳐야 하는지 구체적인 수정 방안을 제시한 뒤, 그 내용을 이슈 관리 시스템에 자동으로 등록해 줍니다. 더 이상 분석 결과를 한 건씩 읽고 이슈를 직접 만들 필요가 없습니다.

### 이 기능을 사용하기 전과 후의 차이

**사용 전 -- 수동 작업 방식:**

1. 개발자가 코드 품질 분석 도구에 접속합니다.
2. 분석 결과 화면에서 "Method complexity 25"와 같은 문제를 발견합니다.
3. 이 문제가 어떤 의미인지 파악하는 데 약 5분이 걸립니다.
4. 이슈 관리 시스템에 접속하여 이슈 제목, 설명, 수정 방안을 직접 작성합니다.
5. 이 과정을 문제가 발견된 건수만큼 반복합니다.

만약 분석 결과에 문제가 20건 있다면, 이 작업만으로 2시간 이상이 소요될 수 있습니다.

**사용 후 -- 자동화 방식:**

1. Jenkins에서 이 파이프라인을 한 번 실행합니다.
2. 파이프라인이 코드 품질 분석 도구에서 문제 목록을 자동으로 가져옵니다.
3. AI가 각 문제를 분석하여 다음 내용을 작성합니다.
   - 이 문제가 왜 발생했는지에 대한 원인 분석
   - 이 문제를 방치하면 어떤 영향이 있는지에 대한 설명
   - 어떻게 수정해야 하는지에 대한 구체적인 방안
4. 작성된 내용이 이슈 관리 시스템에 자동으로 등록됩니다.

20건의 문제가 있더라도 파이프라인 실행 한 번으로 모든 이슈가 자동 등록됩니다.

### 구체적인 예시

**코드 품질 분석 도구가 발견한 문제:**

```text
SQL Injection in QueryBuilder.java:78
Severity: CRITICAL
```

**AI가 자동 작성하여 이슈 관리 시스템에 등록한 내용:**

```text
[제목] SQL Injection 취약점 - QueryBuilder.java:78

[원인 분석]
QueryBuilder.java 파일의 78번째 줄에서 사용자가 입력한 값을
직접 데이터베이스 조회 명령에 연결하고 있습니다. 이 방식은
외부 공격자가 의도하지 않은 데이터베이스 명령을 실행할 수 있게
만드는 보안 취약점입니다.

[영향 범위]
이 취약점을 악용하면 데이터베이스의 모든 정보가 유출되거나
삭제될 수 있습니다.

[수정 방안]
사용자 입력값을 직접 연결하는 대신, 안전한 매개변수 바인딩
방식으로 변경하십시오. 구체적으로 78번째 줄의 코드를
PreparedStatement를 사용하는 방식으로 수정하면 됩니다.
```

이처럼 AI가 기술적 내용을 이해하기 쉬운 언어로 풀어서 설명해 주므로, 개발팀 전체가 문제의 심각성과 수정 방향을 빠르게 파악할 수 있습니다.

## 2. 시작하기 전에 확인할 사항

이 파이프라인을 실행하기 전에 아래 네 가지 항목이 준비되어 있는지 확인하십시오. 하나라도 누락되면 파이프라인이 정상적으로 동작하지 않습니다.

### 2.1. 코드 품질 분석 도구에 분석 대상 프로젝트가 등록되어 있는지 확인합니다

코드 품질 분석 도구에 프로젝트가 등록되어 있어야 하고, 최소 한 번 이상 코드 분석이 실행된 상태여야 합니다. 분석이 한 번도 실행되지 않은 프로젝트에는 가져올 문제가 없으므로 파이프라인이 아무 결과도 생성하지 않습니다.

**확인 절차:**

1. 웹 브라우저를 엽니다.
2. 주소창에 `http://localhost:9000`을 입력하고 이동합니다.
3. 로그인 화면이 나타나면, 등록된 계정으로 로그인합니다.
4. 화면 상단에 있는 메뉴에서 `Projects`를 클릭합니다.
5. 프로젝트 목록이 표시됩니다.
6. 분석하려는 프로젝트가 목록에 있는지 확인합니다.
7. 해당 프로젝트를 클릭했을 때, 분석 결과(버그 수, 취약점 수 등)가 표시되면 준비가 완료된 것입니다.

만약 프로젝트가 목록에 없거나, 프로젝트를 클릭했을 때 "No analysis has been performed"와 같은 메시지가 표시된다면, 먼저 코드 분석을 한 번 실행해야 합니다. 코드 분석 실행 방법은 프로젝트 관리자에게 문의하십시오.

### 2.2. 이슈 관리 시스템에 이슈를 등록할 프로젝트가 있는지 확인합니다

AI가 작성한 분석 결과가 등록될 프로젝트가 이슈 관리 시스템에 존재해야 합니다.

**확인 절차:**

1. 웹 브라우저를 엽니다.
2. 주소창에 `http://localhost:8929`를 입력하고 이동합니다.
3. 로그인 화면이 나타나면, 등록된 계정으로 로그인합니다.
4. 대시보드 또는 좌측 메뉴에서 프로젝트 목록을 확인합니다.
5. 이슈를 등록할 프로젝트가 목록에 있는지 확인합니다.
6. 해당 프로젝트를 클릭하여 프로젝트 페이지에 정상적으로 접속되는지 확인합니다.
7. 좌측 메뉴에서 `Issues`를 클릭하여 이슈 목록 화면이 정상적으로 표시되는지 확인합니다.

만약 프로젝트가 없다면, 이슈 관리 시스템에서 새 프로젝트를 먼저 생성해야 합니다.

### 2.3. 필요한 연결 설정이 Jenkins에 등록되어 있는지 확인합니다

파이프라인이 코드 품질 분석 도구와 이슈 관리 시스템에 접속하려면, Jenkins에 연결 정보가 미리 등록되어 있어야 합니다.

**확인 절차:**

1. 웹 브라우저에서 `http://localhost:8080`에 접속합니다.
2. Jenkins에 로그인합니다.
3. 좌측 메뉴에서 `Jenkins 관리`(또는 `Manage Jenkins`)를 클릭합니다.
4. `Credentials`(또는 `자격 증명 관리`)를 클릭합니다.
5. 다음 항목들이 등록되어 있는지 확인합니다.

| 확인할 항목 | 용도 |
| --- | --- |
| `sonarqube-token` | 코드 품질 분석 도구에 접속하기 위한 인증 정보입니다 |
| `dify-workflow-key` | AI 분석 기능을 호출하기 위한 인증 정보입니다 |

위 항목 중 하나라도 없다면 시스템 관리자에게 등록을 요청하십시오.

### 2.4. 선행 작업: AI에 프로젝트 지식을 학습시키면 진단 품질이 향상됩니다

이 단계는 필수가 아닌 권장 사항입니다. 하지만 결과의 품질에 큰 차이를 만듭니다.

Job #1부터 #4까지의 지식 관리 파이프라인을 먼저 실행하여 AI에 프로젝트의 문서, 코드, 구조를 학습시키면, AI가 문제를 진단할 때 프로젝트의 실제 맥락을 반영한 구체적인 분석과 수정 방안을 제시할 수 있습니다.

예를 들어, AI가 프로젝트 코드를 학습하지 않은 상태에서는 "이 메서드의 순환 복잡도가 높으므로 리팩터링을 권장합니다"와 같은 일반적인 진단만 가능합니다. 반면, 프로젝트 코드를 학습한 상태에서는 "UserService.java의 validateOrder() 메서드에서 할인 정책 분기가 8개의 조건문으로 중첩되어 있습니다. DiscountPolicy 인터페이스를 도입하여 전략 패턴으로 분리하면 복잡도를 효과적으로 낮출 수 있습니다"와 같이 프로젝트 고유의 맥락을 반영한 진단이 가능해집니다.

지식 관리 파이프라인의 사용 방법은 `03_파이프라인_지식관리.md` 문서를 참조하십시오.

## 3. 분석에 필요한 정보를 준비합니다

파이프라인을 실행할 때 두 가지 정보를 입력해야 합니다. 코드 품질 분석 도구에서 어떤 프로젝트의 문제를 가져올지 지정하는 **프로젝트 식별자**와, 이슈 관리 시스템에서 어떤 프로젝트에 이슈를 등록할지 지정하는 **프로젝트 경로**입니다.

아래 안내에 따라 각 값을 미리 확인해 두십시오.

### 3.1. 코드 품질 분석 도구에서 프로젝트 식별자를 확인하는 방법

프로젝트 식별자는 코드 품질 분석 도구가 각 프로젝트를 구분하기 위해 사용하는 고유한 이름입니다. 이 값을 파이프라인에 입력하면, 파이프라인이 해당 프로젝트의 분석 결과를 자동으로 가져옵니다.

**확인 절차:**

1. 웹 브라우저를 엽니다.
2. 주소창에 `http://localhost:9000`을 입력하고 Enter 키를 누릅니다.
3. 코드 품질 분석 도구의 로그인 화면이 나타납니다. 계정 정보를 입력하고 로그인합니다.
4. 로그인에 성공하면 대시보드 화면이 표시됩니다.
5. 화면 상단의 메뉴 바에서 `Projects`라고 쓰여 있는 항목을 클릭합니다.
6. 등록된 프로젝트의 목록이 나타납니다.
7. 분석하려는 프로젝트의 이름을 찾아 클릭합니다.
8. 프로젝트의 분석 결과 화면(대시보드)이 표시됩니다.
9. 이제 웹 브라우저의 **주소창**을 확인합니다.
10. 주소창에 다음과 같은 형태의 주소가 표시되어 있을 것입니다.

    ```text
    http://localhost:9000/dashboard?id=dscore-ttc-sample
    ```

11. 이 주소에서 `id=` 뒤에 오는 값이 바로 **프로젝트 식별자**입니다.
12. 위 예시에서는 `dscore-ttc-sample`이 프로젝트 식별자입니다.

**이 값을 메모해 두십시오.** 4단계에서 파이프라인을 실행할 때 `SONAR_PROJECT_KEY` 항목에 이 값을 입력하게 됩니다.

> 주소창에 `id=` 부분이 보이지 않는 경우, 프로젝트 목록 화면으로 돌아가서 해당 프로젝트를 다시 클릭해 보십시오. 또는 프로젝트 설정 화면에서도 프로젝트 식별자를 확인할 수 있습니다.

### 3.2. 이슈 관리 시스템에서 프로젝트 경로를 확인하는 방법

프로젝트 경로는 이슈 관리 시스템에서 각 프로젝트를 구분하기 위해 사용하는 주소의 일부입니다. 이 값을 파이프라인에 입력하면, AI가 분석한 결과를 해당 프로젝트의 이슈로 자동 등록합니다.

**확인 절차:**

1. 웹 브라우저를 엽니다.
2. 주소창에 `http://localhost:8929`를 입력하고 Enter 키를 누릅니다.
3. 이슈 관리 시스템의 로그인 화면이 나타납니다. 계정 정보를 입력하고 로그인합니다.
4. 로그인에 성공하면 대시보드 화면이 표시됩니다.
5. 이슈를 등록하려는 프로젝트를 찾아 클릭합니다.
   - 대시보드에 프로젝트가 바로 보이면 클릭합니다.
   - 보이지 않는 경우, 좌측 메뉴에서 `Projects`(또는 `프로젝트`)를 클릭하여 전체 목록에서 찾습니다.
6. 프로젝트 페이지가 표시됩니다.
7. 이제 웹 브라우저의 **주소창**을 확인합니다.
8. 주소창에 다음과 같은 형태의 주소가 표시되어 있을 것입니다.

    ```text
    http://localhost:8929/root/dscore-ttc-sample
    ```

9. 이 주소에서 도메인(`http://localhost:8929`) 뒤에 오는 나머지 경로가 바로 **프로젝트 경로**입니다.
10. 위 예시에서는 `root/dscore-ttc-sample`이 프로젝트 경로입니다.
    - `root`는 프로젝트가 속한 그룹(또는 사용자)의 이름입니다.
    - `dscore-ttc-sample`은 프로젝트의 이름입니다.
    - 이 두 부분을 슬래시(`/`)로 연결한 것이 프로젝트 경로입니다.

**이 값을 메모해 두십시오.** 4단계에서 파이프라인을 실행할 때 `GITLAB_PROJECT` 항목에 이 값을 입력하게 됩니다.

> 프로젝트가 하위 그룹에 속해 있는 경우, 경로가 `group/subgroup/project`처럼 더 길어질 수 있습니다. 주소창에 보이는 경로를 그대로 사용하면 됩니다.

## 4. Jenkins에서 품질 분석을 실행합니다

이제 준비가 완료되었으므로, Jenkins에서 파이프라인을 실행합니다.

### 4.1. Job 화면으로 이동합니다

1. 웹 브라우저를 엽니다.
2. 주소창에 `http://localhost:8080`을 입력하고 Enter 키를 누릅니다.
3. Jenkins 로그인 화면이 나타나면, 계정 정보를 입력하고 로그인합니다.
4. 로그인에 성공하면 Jenkins 대시보드가 표시됩니다.
5. 대시보드에 등록된 작업(Job) 목록이 나타납니다.
6. 목록에서 `DSCORE-TTC 코드 정적분석 결과분석 및 이슈등록`이라는 이름의 작업을 찾습니다.
   - 작업 목록이 많아 찾기 어려운 경우, 대시보드 상단의 검색창에 "정적분석"을 입력하면 빠르게 찾을 수 있습니다.
7. 해당 작업 이름을 클릭합니다.
8. 작업의 상세 화면이 표시됩니다. 좌측 메뉴에 `Build with Parameters`(또는 `파라미터와 함께 빌드`)라는 항목이 보일 것입니다.

### 4.2. 분석 조건을 입력합니다

1. 좌측 메뉴에서 `Build with Parameters`(또는 `파라미터와 함께 빌드`)를 클릭합니다.
2. 화면에 다섯 개의 입력 항목이 표시됩니다. 각 항목에 아래 안내에 따라 값을 입력합니다.

---

#### 프로젝트 식별자 (SONAR_PROJECT_KEY)

#### 분석할 문제의 심각도 (SEVERITIES)

이 항목에는 어떤 수준의 문제를 분석할지 지정합니다. 코드 품질 분석 도구는 발견된 문제를 다섯 가지 심각도 수준으로 분류합니다.

- **입력 위치:** `SEVERITIES`라고 표시된 입력 칸
- **기본값:** `BLOCKER,CRITICAL`이 미리 입력되어 있습니다
- **입력 규칙:** 원하는 심각도를 쉼표(`,`)로 구분하여 입력합니다. 쉼표 앞뒤에 공백을 넣지 마십시오.

**각 심각도 수준의 의미:**

| 심각도 | 쉽게 말하면 | 구체적인 예시 |
| --- | --- | --- |
| `BLOCKER` | 프로그램이 멈추거나 데이터가 손상될 수 있는 치명적인 문제입니다 | 프로그램이 갑자기 중단되는 오류, 무한 반복에 빠지는 코드, 데이터가 사라지는 버그 |
| `CRITICAL` | 보안 취약점이거나 심각한 오류입니다 | 외부 공격에 노출되는 보안 허점, 인증을 우회할 수 있는 취약점, 중요한 데이터가 유출될 수 있는 코드 |
| `MAJOR` | 코드 품질에 큰 영향을 미치는 문제입니다 | 코드가 지나치게 복잡하여 유지보수가 어려운 부분, 사용하지 않는 코드가 남아 있는 경우, 오류 처리가 누락된 부분 |
| `MINOR` | 코딩 규칙 위반이나 스타일 문제입니다 | 변수 이름이 규칙에 맞지 않는 경우, 동일한 코드가 여러 곳에 중복된 경우, 들여쓰기가 일관되지 않은 경우 |
| `INFO` | 참고 수준의 개선 제안입니다 | 주석을 추가하면 좋겠다는 제안, 로그 출력을 추가하면 좋겠다는 권고 |

**권장 입력값:**

- 처음 사용할 때: `BLOCKER,CRITICAL` (가장 심각한 문제만 먼저 처리합니다)
- 결과를 확인한 후: `BLOCKER,CRITICAL,MAJOR` (주요 문제까지 범위를 넓힙니다)
- 전체 분석이 필요할 때: `BLOCKER,CRITICAL,MAJOR,MINOR` (스타일 문제까지 포함합니다)

#### 분석할 문제의 상태 (STATUSES)

이 항목에는 어떤 상태의 문제를 분석할지 지정합니다.

- **입력 위치:** `STATUSES`라고 표시된 입력 칸
- **기본값:** `OPEN,REOPENED,CONFIRMED`이 미리 입력되어 있습니다
- **입력 규칙:** 원하는 상태를 쉼표(`,`)로 구분하여 입력합니다. 쉼표 앞뒤에 공백을 넣지 마십시오.

**각 상태의 의미:**

| 상태 | 의미 |
| --- | --- |
| `OPEN` | 새로 발견된 문제입니다. 아직 아무도 검토하지 않은 상태입니다. |
| `REOPENED` | 한 번 해결했다가 다시 발생한 문제입니다. 이전에 수정을 시도했지만 문제가 재발한 경우입니다. |
| `CONFIRMED` | 담당자가 검토하여 실제 문제임을 확인한 상태입니다. |

**권장 입력값:**

대부분의 경우 기본값인 `OPEN,REOPENED,CONFIRMED`을 그대로 사용하면 됩니다. 이 설정은 아직 해결되지 않은 모든 문제를 분석 대상에 포함합니다.

새로 발견된 문제만 분석하고 싶다면 `OPEN`만 입력하십시오.

#### 최대 처리 건수 (MAX_ISSUES)

이 항목에는 한 번의 실행에서 최대 몇 건의 문제를 분석할지 지정합니다.

- **입력 위치:** `MAX_ISSUES`라고 표시된 입력 칸
- **기본값:** `1`이 미리 입력되어 있습니다
- **입력 규칙:** 숫자를 입력합니다. `0`을 입력하면 발견된 모든 문제를 처리합니다.

**권장 입력값:**

- 처음 사용할 때: `5`를 입력하여 결과를 먼저 확인합니다
- 결과가 만족스러우면: `10` 또는 `15`로 늘려 갑니다
- 전체 문제를 한 번에 처리하고 싶을 때: `0`을 입력합니다

처음에는 반드시 5~10건으로 시작하여 결과를 확인한 후 점차 늘려 가십시오. AI 분석에는 문제 1건당 수 초에서 수십 초가 소요되므로, 처음부터 많은 건수를 지정하면 실행 시간이 길어질 수 있습니다.

기본값이 이슈를 등록하려는 프로젝트의 경로와 일치한다면, 이 항목은 수정하지 않아도 됩니다.

### 4.3. 분석을 시작합니다

1. 모든 입력 항목을 확인한 후, 화면 하단에 있는 `빌드`(또는 `Build`) 버튼을 클릭합니다.
2. 버튼을 클릭하면 파이프라인이 즉시 실행을 시작합니다.
3. 화면 좌측 하단의 `빌드 기록`(또는 `Build History`) 영역에 새로운 빌드 번호(예: `#15`)가 나타납니다.
4. 파란색 원이 깜빡이면 현재 실행 중이라는 의미입니다.

**실행 소요 시간 안내:**

파이프라인은 세 단계로 진행됩니다.

| 단계 | 수행 내용 | 예상 소요 시간 |
| --- | --- | --- |
| 1단계 | 코드 품질 분석 도구에서 문제 목록을 가져옵니다 | 수 초 |
| 2단계 | AI가 각 문제를 분석하고 진단 결과를 작성합니다 | 문제 1건당 10~30초 |
| 3단계 | 진단 결과를 이슈 관리 시스템에 등록합니다 | 이슈 1건당 수 초 |

예를 들어 `MAX_ISSUES`를 `10`으로 설정한 경우, 전체 실행에 약 3~7분이 소요될 수 있습니다.

파이프라인 실행 중에는 브라우저를 닫아도 됩니다. 실행은 Jenkins 서버에서 계속 진행됩니다. 나중에 다시 접속하여 결과를 확인할 수 있습니다.

## 5. 분석 결과를 확인합니다

파이프라인 실행이 완료되면 두 곳에서 결과를 확인할 수 있습니다. Jenkins에서 실행 로그를 확인하고, 이슈 관리 시스템에서 자동 생성된 이슈를 확인합니다.

### 5.1. Jenkins에서 실행 로그를 확인합니다

실행 로그를 통해 파이프라인이 정상적으로 완료되었는지, 몇 건의 이슈가 생성되었는지 확인할 수 있습니다.

**확인 절차:**

1. Jenkins 대시보드(`http://localhost:8080`)에 접속합니다.
2. `DSCORE-TTC 코드 정적분석 결과분석 및 이슈등록` 작업을 클릭합니다.
3. 화면 좌측의 `빌드 기록`(또는 `Build History`)에서 확인하려는 빌드 번호를 클릭합니다.
   - 가장 최근 실행은 목록의 맨 위에 있습니다.
   - 빌드 번호 왼쪽의 아이콘 색상으로 실행 결과를 빠르게 파악할 수 있습니다.
     - 파란색: 정상 완료
     - 빨간색: 오류 발생
     - 회색: 취소됨
4. 빌드 상세 화면이 표시됩니다.
5. 좌측 메뉴에서 `Console Output`(또는 `콘솔 출력`)을 클릭합니다.
6. 실행 로그가 표시됩니다.

**정상적으로 완료된 경우의 로그 예시:**

```text
[SONAR] 프로젝트 'dscore-ttc-sample'에서 이슈 12건 조회 완료
[SONAR] 필터 적용: BLOCKER(2건), CRITICAL(5건), MAJOR(5건)
[SONAR] 이슈 1/10: AYx-abc123 (CRITICAL) - SQL Injection in QueryBuilder.java:78
[SONAR]   코드 스니펫 추출 완료 (Line 73-83)
[DIFY]  이슈 1/10 분석 중...
[DIFY]  진단 완료: "QueryBuilder.java의 78번 라인에서 사용자 입력값을 직접
         SQL 쿼리에 연결하고 있어 SQL Injection 공격에 취약합니다.
         PreparedStatement를 사용하여 파라미터 바인딩으로 수정해야 합니다."
[GITLAB] Issue #15 생성 완료: "SQL Injection 취약점 - QueryBuilder.java:78"
[GITLAB] Issue #16 생성 완료: "NullPointerException 위험 - UserService.java:42"
[GITLAB] Issue #17 건너뜀 (이미 등록된 이슈가 있으므로 중복 방지)
...
[SUMMARY] 처리 완료: 10건 중 8건 생성, 2건 중복 건너뜀
Finished: SUCCESS
```

위와 같이 마지막 줄에 `Finished: SUCCESS`가 표시되면 정상적으로 완료된 것입니다.

**오류가 발생한 경우의 로그 예시:**

```text
[SONAR] 프로젝트 'dscore-ttc-sample'에서 이슈 조회 중...
ERROR: SonarQube API 401 Unauthorized
Finished: FAILURE
```

이 경우 마지막 줄에 `Finished: FAILURE`가 표시됩니다. 오류 메시지를 확인하여 원인을 파악하십시오. 자주 발생하는 오류와 해결 방법은 9절에서 상세히 안내합니다.

### 5.2. 이슈 관리 시스템에서 자동 생성된 이슈를 확인합니다

파이프라인이 정상적으로 완료되면, 이슈 관리 시스템에 AI가 작성한 이슈들이 자동으로 등록되어 있습니다.

**확인 절차:**

1. 웹 브라우저에서 `http://localhost:8929`에 접속합니다.
2. 이슈 관리 시스템에 로그인합니다.
3. 이슈를 등록한 프로젝트로 이동합니다.
   - 예: `root/dscore-ttc-sample` 프로젝트를 클릭합니다.
4. 좌측 메뉴에서 `Issues`(또는 `이슈`)를 클릭합니다.
5. 이슈 목록 화면이 표시됩니다.
6. 파이프라인이 자동으로 생성한 이슈들이 목록에 나타납니다.
7. 이슈 하나를 클릭하여 상세 내용을 확인합니다.

**자동 생성된 이슈의 구성:**

자동 생성된 이슈에는 다음 내용이 포함되어 있습니다.

| 항목 | 내용 |
| --- | --- |
| 제목 | 문제의 유형과 해당 파일명, 줄 번호가 포함된 요약입니다 |
| 본문 - 문제 설명 | 코드 품질 분석 도구가 발견한 문제의 내용입니다 |
| 본문 - AI 진단 결과 | AI가 분석한 문제의 원인, 영향 범위, 수정 방안입니다 |
| 본문 - 코드 위치 | 문제가 발생한 파일과 줄 번호, 해당 코드 내용입니다 |
| 본문 - 분석 도구 링크 | 코드 품질 분석 도구에서 해당 문제를 직접 확인할 수 있는 주소입니다 |
| 라벨 | 코드 품질 분석 도구의 문제 고유 식별자가 라벨로 부착되어 있습니다 |

**자동 생성된 이슈의 예시:**

```text

[본문]

데이터를 삭제하거나, 시스템 권한을 탈취할 수 있습니다.

### 수정 방안

수정 전:
  String query = "SELECT * FROM users WHERE name = '" + userName + "'";

수정 후:
  String query = "SELECT * FROM users WHERE name = ?";
  PreparedStatement stmt = conn.prepareStatement(query);
```

### 5.3. AI 진단 결과를 읽고 활용하는 방법

이슈를 클릭하여 상세 내용을 열면, AI가 작성한 진단 결과가 본문에 포함되어 있습니다. 이 진단 결과는 단순히 "어떤 규칙을 위반했다"는 분석 도구의 원문을 넘어, **문제의 근본 원인, 실제 영향 범위, 구체적인 수정 방안**을 프로젝트 맥락에 맞게 설명합니다.

**진단 결과의 각 항목을 읽는 방법:**

| 항목 | 확인 포인트 | 활용 방법 |
| --- | --- | --- |
| **원인 분석** | AI가 문제의 근본적인 원인을 설명합니다. "왜 이 코드가 문제인가?"에 대한 답변입니다. | 코드를 수정하기 전에 원인을 먼저 이해하십시오. 원인을 모르고 수정하면 같은 문제가 반복될 수 있습니다. |
| **영향 범위** | 이 문제를 방치했을 때 발생할 수 있는 구체적인 위험을 설명합니다. | 수정 우선순위를 판단하는 데 활용하십시오. 영향 범위가 "데이터 유출" 수준이면 즉시 수정이 필요하고, "코드 가독성 저하" 수준이면 일정에 맞춰 수정해도 됩니다. |
| **수정 방안** | AI가 제안하는 구체적인 코드 수정 방법입니다. 수정 전/후 코드 예시가 포함되는 경우도 있습니다. | 수정 방안은 참고 자료로 활용하십시오. AI의 제안이 항상 최적의 방법은 아닐 수 있으므로, 담당 개발자가 프로젝트 상황에 맞게 최종 판단하여야 합니다. |

**AI 진단 품질을 높이려면:**

AI는 프로젝트에 대해 학습한 지식이 많을수록 더 구체적이고 정확한 진단을 제공합니다. 예를 들어, Job #1~#3으로 프로젝트 문서와 소스 코드를 사전에 학습시키면 다음과 같은 차이가 발생합니다.

- **프로젝트 지식이 없을 때:** "이 메서드는 순환 복잡도가 높으므로 리팩터링을 권장합니다." (일반적인 조언)
- **프로젝트 지식이 있을 때:** "이 메서드는 인증 검증과 권한 확인을 하나의 함수에서 처리하고 있습니다. `AuthService.validateToken()`과 `PermissionChecker.check()`로 분리하면 복잡도를 각각 10 이하로 줄일 수 있습니다." (프로젝트 맥락에 맞는 구체적 조언)

따라서 이 파이프라인을 처음 사용하시기 전에, 지식 관리 파이프라인(Job #1~#3)을 먼저 실행하여 프로젝트 지식을 AI에 학습시키는 것을 강력히 권장합니다. 자세한 방법은 [03_파이프라인_지식관리.md](03_파이프라인_지식관리.md)를 참조하십시오.

### 5.4. 코드 품질 분석 도구(SonarQube)에서 원본 분석 결과를 확인합니다

이슈 관리 시스템에 등록된 이슈뿐 아니라, 코드 품질 분석 도구에서 원본 분석 결과를 직접 확인할 수도 있습니다. 원본 화면에서는 문제가 발생한 코드 라인을 직접 보거나, 전체적인 코드 품질 현황을 파악할 수 있습니다.

**확인 절차:**

1. 웹 브라우저에서 `http://localhost:9000`에 접속하여 로그인합니다.
2. 상단의 `Projects` 메뉴를 클릭합니다.
3. 분석 대상 프로젝트를 클릭합니다.
4. `Overview` 화면이 표시됩니다. 이 화면의 최상단에서 **Quality Gate 상태**를 확인합니다.
   - **Passed (초록색):** 프로젝트가 설정된 품질 기준을 통과했습니다.
   - **Failed (빨간색):** 품질 기준을 충족하지 못한 항목이 있습니다. 아래 상세 내역을 확인하십시오.

5. 상세 분석 결과는 화면 상단의 탭을 클릭하여 확인합니다.

| 탭 이름 | 확인할 수 있는 내용 |
| --- | --- |
| `Issues` | 발견된 버그(Bugs), 보안 취약점(Vulnerabilities), 코드 스멜(Code Smells)의 목록입니다. 각 항목을 클릭하면 문제가 발생한 소스 코드의 정확한 라인을 확인할 수 있습니다. |
| `Security Hotspots` | 보안 검토가 필요한 코드 영역입니다. 반드시 취약점은 아니지만, 보안 담당자의 확인이 권장되는 코드입니다. |
| `Measures` | 코드 품질의 수치 지표입니다. 테스트 커버리지(Coverage), 중복 코드 비율(Duplications), 코드 라인 수(Lines of Code) 등을 확인할 수 있습니다. |

> 이 파이프라인(Job #4)은 `Issues` 탭에 표시되는 문제들을 AI가 자동으로 진단하고 이슈 관리 시스템에 등록하는 역할을 합니다. 따라서 파이프라인 실행 후 `Issues` 탭의 문제와 이슈 관리 시스템의 이슈를 비교하면, AI가 어떤 문제를 진단했는지 교차 확인할 수 있습니다.

### 5.5. 동일한 문제가 중복으로 등록되지 않는 이유

파이프라인을 여러 번 실행하더라도, 같은 문제에 대한 이슈가 중복으로 생성되지 않습니다.

**중복 방지가 동작하는 방식:**

코드 품질 분석 도구에서 발견된 각 문제에는 고유한 식별 번호가 있습니다(예: `AYx-abc123`). 파이프라인은 이슈를 등록할 때 이 식별 번호를 이슈의 라벨로 함께 부착합니다.

다음 실행에서 같은 문제를 다시 발견했을 때, 파이프라인은 이슈 관리 시스템에서 해당 식별 번호가 라벨로 부착된 이슈가 이미 있는지 먼저 확인합니다. 이미 있다면 해당 문제는 건너뛰고, 없는 경우에만 새 이슈를 생성합니다.

따라서 아래와 같은 상황에서도 안심하고 파이프라인을 반복 실행할 수 있습니다.

- 매일 정기적으로 실행하는 경우: 이미 등록된 문제는 건너뛰고, 새로 발견된 문제만 이슈로 등록됩니다.
- 실수로 같은 조건으로 두 번 실행한 경우: 두 번째 실행에서는 모든 문제가 이미 등록되어 있으므로 "건너뜀"으로 처리됩니다.
- 분석 범위를 넓혀서 다시 실행한 경우: 이전에 이미 등록된 문제는 건너뛰고, 새로 추가된 범위의 문제만 이슈로 등록됩니다.

Jenkins 실행 로그에서 "건너뜀" 또는 "중복 방지"라는 메시지가 표시된 항목이 이미 등록되어 있어 생략된 문제입니다.

## 6. 정기적으로 자동 실행하는 방법

파이프라인을 매번 수동으로 실행하는 대신, 정해진 시간에 자동으로 실행되도록 설정할 수 있습니다. 예를 들어 "매주 월요일 오전 9시"에 자동으로 품질 분석을 실행하도록 설정하면, 매주 월요일 아침 출근했을 때 이슈 관리 시스템에 새로운 품질 이슈가 정리되어 있는 것을 확인할 수 있습니다.

### 6.1. 자동 실행 설정 절차

1. Jenkins 대시보드(`http://localhost:8080`)에 접속합니다.
2. `DSCORE-TTC 코드 정적분석 결과분석 및 이슈등록` 작업을 클릭합니다.
3. 좌측 메뉴에서 `구성`(또는 `Configure`)을 클릭합니다.
4. 작업 설정 화면이 표시됩니다. 화면을 아래로 스크롤합니다.
5. `빌드 트리거`(또는 `Build Triggers`)라는 섹션을 찾습니다.
6. `빌드를 주기적으로 실행`(또는 `Build periodically`)이라는 항목에 체크 표시를 합니다.
7. 체크 표시를 하면 아래에 입력 칸이 나타납니다.
8. 입력 칸에 실행 주기를 지정하는 값을 입력합니다.

### 6.2. 실행 주기를 지정하는 방법

실행 주기는 다섯 개의 숫자를 공백으로 구분하여 입력합니다. 각 위치가 의미하는 바는 다음과 같습니다.

```text
분  시  일  월  요일
```

자주 사용하는 설정 예시를 아래에 안내합니다. 원하는 주기에 해당하는 값을 그대로 복사하여 입력 칸에 붙여넣으면 됩니다.

**매주 월요일 오전 9시에 실행:**

```text
H 9 * * 1
```

이 설정에서 `H`는 Jenkins가 서버 부하를 분산하기 위해 정확히 9시 0분이 아닌 9시대의 임의의 시각에 실행한다는 의미입니다. 예를 들어 9시 23분이나 9시 47분에 실행될 수 있습니다.

**평일(월~금) 매일 오전 9시에 실행:**

```text
H 9 * * 1-5
```

요일 위치에 `1-5`를 입력하면 월요일부터 금요일까지를 의미합니다.

**매일 오전 9시에 실행 (주말 포함):**

```text
H 9 * * *
```

요일 위치에 `*`를 입력하면 모든 요일을 의미합니다.

**매주 월요일과 목요일 오전 9시에 실행:**

```text
H 9 * * 1,4
```

요일 위치에 `1,4`를 입력하면 월요일과 목요일을 의미합니다.

**2주에 한 번 월요일 오전 9시에 실행:**

```text
H 9 1-7,15-21 * 1
```

### 6.3. 설정을 저장합니다

1. 원하는 실행 주기를 입력한 후, 화면 하단의 `저장`(또는 `Save`) 버튼을 클릭합니다.
2. 설정이 저장되면 작업 상세 화면으로 돌아갑니다.
3. 이제부터 지정한 주기에 맞춰 파이프라인이 자동으로 실행됩니다.

> 자동 실행 시에는 마지막으로 수동 실행할 때 사용한 파라미터 값(심각도, 상태, 최대 처리 건수 등)이 그대로 사용됩니다. 자동 실행에 사용할 파라미터 값을 변경하고 싶다면, 원하는 값으로 한 번 수동 실행한 후 자동 실행이 이어지도록 하십시오.

### 6.4. 자동 실행을 중단하는 방법

1. 작업 상세 화면에서 좌측 메뉴의 `구성`(또는 `Configure`)을 클릭합니다.
2. `빌드 트리거` 섹션에서 `빌드를 주기적으로 실행`의 체크 표시를 해제합니다.
3. `저장` 버튼을 클릭합니다.
4. 이후로는 수동으로 실행할 때만 파이프라인이 동작합니다.

## 7. 분석 범위를 단계적으로 넓히는 방법

처음부터 모든 문제를 한꺼번에 분석하는 것보다, 단계적으로 범위를 넓혀 가는 것을 권장합니다. 이렇게 하면 AI 진단 결과의 품질을 확인하면서 점진적으로 활용 범위를 확장할 수 있습니다.

### 1단계: 치명적인 문제만 우선 처리합니다

가장 먼저 프로그램의 동작에 직접적인 영향을 미치는 치명적 문제만 분석합니다.

| 항목 | 설정값 |
| --- | --- |
| SEVERITIES | `BLOCKER,CRITICAL` |
| STATUSES | `OPEN,REOPENED,CONFIRMED` |
| MAX_ISSUES | `5` |

이 설정으로 파이프라인을 실행한 후, 이슈 관리 시스템에 등록된 이슈를 확인합니다. AI가 작성한 진단 내용이 정확한지, 수정 방안이 실제로 도움이 되는지 검토합니다.

**확인할 사항:**

- AI의 원인 분석이 실제 코드 문제와 일치하는지 확인합니다.
- 제시된 수정 방안이 실제로 적용 가능한 내용인지 확인합니다.
- 이슈의 제목과 본문이 팀원들이 이해하기 쉬운 형태인지 확인합니다.

### 2단계: 주요 문제까지 범위를 넓힙니다

1단계의 결과가 만족스러우면, 코드 품질에 큰 영향을 미치는 `MAJOR` 수준의 문제까지 분석 범위를 넓힙니다.

| 항목 | 설정값 |
| --- | --- |
| SEVERITIES | `BLOCKER,CRITICAL,MAJOR` |
| STATUSES | `OPEN,REOPENED,CONFIRMED` |
| MAX_ISSUES | `15` |

### 3단계: 전체 문제를 분석합니다

2단계까지 결과가 안정적이면, 코딩 규칙 위반 수준의 문제까지 포함하여 전체 분석을 실행합니다.

| 항목 | 설정값 |
| --- | --- |
| SEVERITIES | `BLOCKER,CRITICAL,MAJOR,MINOR` |
| STATUSES | `OPEN,REOPENED,CONFIRMED` |
| MAX_ISSUES | `30` 또는 `0` (전체 처리) |

> `INFO` 수준은 참고 사항에 해당하므로, 팀의 필요에 따라 포함 여부를 결정하십시오. 일반적으로 `INFO`까지 포함하면 이슈의 양이 크게 늘어나므로, 꼭 필요한 경우에만 추가하는 것을 권장합니다.

### 단계별 진행 시 확인 체크리스트

각 단계를 진행할 때마다 아래 항목을 확인하십시오.

- [ ] AI 진단 결과가 실제 코드 문제와 일치합니까?
- [ ] 수정 방안이 프로젝트에 실제로 적용 가능합니까?
- [ ] 이슈의 제목만 읽어도 문제가 무엇인지 파악할 수 있습니까?
- [ ] 중복 이슈가 생성되지 않았습니까?
- [ ] 실행 시간이 허용 가능한 범위 내입니까?

모든 항목을 확인한 후에 다음 단계로 진행하십시오.

## 8. AI 진단 품질을 높이는 방법

AI가 제공하는 진단 결과의 품질은 AI가 프로젝트에 대해 얼마나 많은 맥락 정보를 가지고 있느냐에 따라 크게 달라집니다. 아래 방법을 활용하면 AI 진단의 정확도와 구체성을 향상시킬 수 있습니다.

### 8.1. Job #1~#3을 먼저 실행하여 프로젝트 지식을 학습시킵니다

품질 분석 파이프라인(이 문서에서 설명하는 Job #4)을 실행하기 전에, 지식 관리 파이프라인(Job #1~#3)을 먼저 실행하면 AI 진단 품질이 크게 향상됩니다.

지식 관리 파이프라인은 프로젝트의 문서, 코드, 웹 자료를 AI가 참조할 수 있는 형태로 변환하여 저장합니다. AI는 이 저장된 지식을 참조하여 문제를 진단합니다.

**각 Job이 AI 진단에 미치는 영향:**

| Job | 학습시키는 내용 | AI 진단에 미치는 영향 |
| --- | --- | --- |
| Job #1 | 프로젝트 기획서, 설계 문서, 기술 스펙 등 | AI가 프로젝트의 목적과 설계 의도를 이해하고 진단에 반영합니다 |
| Job #2 | 위와 동일한 문서를 질문-답변 형태로 변환 | AI가 "이 코드는 왜 이렇게 작성되었는가?"와 같은 맥락을 파악합니다 |
| Job #3 | 프로젝트의 코드 구조와 설정 파일 | AI가 실제 코드를 참조하여 파일명, 클래스명, 메서드명을 정확하게 언급합니다 |

### 8.2. 프로젝트 지식 유무에 따른 진단 품질 비교

**프로젝트 지식 없이 진단한 경우:**

```text
[문제] UserService.java:142 - Method complexity is 25

[AI 진단]
이 메서드의 순환 복잡도가 25로, 권장 기준인 15를 초과합니다.
복잡도가 높은 메서드는 유지보수가 어렵고 버그 발생 확률이
높아집니다. 메서드를 더 작은 단위로 분리하는 것을 권장합니다.
```

위 진단은 일반적인 내용만 담고 있어, 실제로 어떻게 수정해야 하는지 파악하기 어렵습니다.

**프로젝트 지식을 학습한 후 진단한 경우:**

```text
[문제] UserService.java:142 - Method complexity is 25

[AI 진단]
UserService.java의 processOrder() 메서드가 순환 복잡도 25로
측정되었습니다. 이 메서드는 주문 처리 과정에서 회원 등급별
할인 정책(일반/실버/골드/플래티넘/VIP), 쿠폰 적용 여부,
배송비 계산 로직이 하나의 메서드에 모두 포함되어 있습니다.

프로젝트의 기존 설계 문서(order-flow.md)에 따르면 할인 정책은
향후 추가될 예정이므로, 지금 분리하는 것이 효율적입니다.

수정 방안:
1. 할인 정책 분기를 DiscountCalculator 클래스로 분리합니다.
2. 쿠폰 적용 로직을 CouponApplier 클래스로 분리합니다.
3. 배송비 계산은 기존 ShippingService.calculateFee()를
   활용합니다.

이렇게 분리하면 processOrder()의 복잡도가 약 8로 감소하며,
향후 할인 정책 추가 시에도 processOrder()를 수정할 필요가
없어집니다.
```

위 진단은 프로젝트의 실제 코드와 문서를 참조하여, 구체적인 클래스명과 파일명을 포함한 실행 가능한 수정 방안을 제시하고 있습니다.

### 8.3. 최적의 AI 진단 품질을 위한 권장 순서

1. **Job #1 실행:** 프로젝트 관련 문서(기획서, 설계서, 스펙 등)를 학습시킵니다.
2. **Job #3 실행:** 프로젝트의 코드 구조와 설정 파일을 학습시킵니다.
3. **Job #2 실행:** 문서를 질문-답변 형태로 변환하여 추가 학습시킵니다.
4. **Job #4 실행:** 이 문서에서 설명하는 품질 분석 파이프라인을 실행합니다.

Job #1과 #3은 AI 진단 품질에 가장 큰 영향을 미치므로, 최소한 이 두 가지는 반드시 먼저 실행할 것을 권장합니다.

### 8.4. 진단 품질이 기대에 미치지 못할 때

AI 진단 결과가 너무 일반적이거나 부정확하다면, 다음 사항을 확인하십시오.

1. **지식 관리 파이프라인이 정상적으로 실행되었는지 확인합니다.**
   - Jenkins에서 Job #1~#3의 빌드 기록을 확인하고, 모두 성공(`SUCCESS`)으로 완료되었는지 확인합니다.

2. **학습시킨 문서의 내용이 충분한지 확인합니다.**
   - 프로젝트 설계 문서, 아키텍처 문서가 포함되어 있는지 확인합니다.
   - 코드 관련 규칙이나 가이드라인 문서가 있다면 함께 학습시킵니다.

3. **코드 컨텍스트(Job #3)가 최신 상태인지 확인합니다.**
   - 코드가 변경된 후 Job #3을 다시 실행하지 않았다면, AI가 이전 버전의 코드 구조를 참조하고 있을 수 있습니다.
   - Job #3을 다시 실행하여 최신 코드 구조를 반영합니다.

## 9. 자주 발생하는 문제와 해결 방법

파이프라인 실행 중 오류가 발생하면 Jenkins의 `Console Output`에 오류 메시지가 표시됩니다. 아래에서 자주 발생하는 오류와 그 해결 방법을 안내합니다.

### 9.1. 코드 품질 분석 도구 접속 오류

**오류 메시지:**

```text
SonarQube API 401 Unauthorized
```

**원인:**

코드 품질 분석 도구에 접속하기 위한 인증 정보가 잘못되었거나 만료되었습니다.

**해결 절차:**

1. Jenkins 대시보드(`http://localhost:8080`)에 접속합니다.
2. 좌측 메뉴에서 `Jenkins 관리`를 클릭합니다.
3. `Credentials`를 클릭합니다.
4. `sonarqube-token` 항목을 찾아 클릭합니다.
5. `Update`(또는 `수정`)를 클릭합니다.
6. 코드 품질 분석 도구에서 새로 발급받은 인증 정보를 입력합니다.
7. `Save`를 클릭합니다.

인증 정보를 새로 발급받는 방법은 다음과 같습니다.

1. 코드 품질 분석 도구(`http://localhost:9000`)에 접속합니다.
2. 우측 상단의 사용자 아이콘을 클릭합니다.
3. `My Account`(또는 `내 계정`)를 클릭합니다.
4. `Security`(또는 `보안`) 탭을 클릭합니다.
5. `Generate Tokens`(또는 `토큰 생성`) 섹션에서 새 인증 정보를 생성합니다.
6. 생성된 값을 복사하여 Jenkins에 등록합니다.

---

### 9.2. 조회된 문제가 0건인 경우

**오류 메시지:**

```text
이슈가 0건 조회됨
```

또는 로그에 문제 목록이 전혀 출력되지 않는 경우입니다.

**원인:**

입력한 심각도나 상태 조건에 해당하는 문제가 코드 품질 분석 도구에 없습니다.

**해결 절차:**

1. 먼저 코드 품질 분석 도구에서 직접 확인합니다.
   - `http://localhost:9000`에 접속합니다.
   - 해당 프로젝트를 클릭합니다.
   - `Issues` 탭을 클릭하여 문제 목록을 확인합니다.
2. 문제가 있지만 조건에 맞지 않는 경우:
   - `SEVERITIES` 값을 `BLOCKER,CRITICAL,MAJOR,MINOR`로 넓혀서 다시 실행합니다.
   - `STATUSES` 값을 `OPEN,REOPENED,CONFIRMED`으로 넓혀서 다시 실행합니다.
3. 문제 자체가 없는 경우:
   - 코드 분석이 최소 한 번 이상 실행되었는지 확인합니다.
   - 프로젝트 식별자(`SONAR_PROJECT_KEY`)가 정확한지 확인합니다.

---

### 9.3. 이슈 관리 시스템에 이슈 생성 실패

**오류 메시지:**

```text
GitLab Issue 생성 실패 (403)
```

**원인:**

이슈 관리 시스템에 이슈를 등록할 권한이 없습니다.

**해결 절차:**

1. 이슈 관리 시스템(`http://localhost:8929`)에 접속합니다.
2. 이슈를 등록하려는 프로젝트로 이동합니다.
3. 좌측 메뉴에서 `Settings`(또는 `설정`) > `Members`(또는 `구성원`)를 클릭합니다.
4. 파이프라인이 사용하는 계정이 해당 프로젝트의 구성원으로 등록되어 있는지 확인합니다.
5. 구성원으로 등록되어 있지 않다면, `Developer` 이상의 권한으로 추가합니다.

프로젝트 경로(`GITLAB_PROJECT`)가 정확한지도 함께 확인하십시오. 경로가 잘못되어 존재하지 않는 프로젝트에 이슈를 등록하려고 시도하면 같은 오류가 발생할 수 있습니다.

---

### 9.4. AI 분석 단계 실패

**오류 메시지:**

```text
Dify Workflow 실행 실패
```

**원인:**

AI 분석 기능이 정상적으로 동작하지 않고 있습니다.

**해결 절차:**

1. AI 분석 기능의 상태를 확인합니다.
   - Dify 웹 화면에 접속합니다.
   - 좌측 메뉴에서 품질 분석용으로 설정된 Workflow를 찾습니다.
   - Workflow가 활성 상태인지 확인합니다.
2. AI 분석 기능의 인증 정보를 확인합니다.
   - Jenkins의 `Credentials`에서 `dify-workflow-key` 값이 올바른지 확인합니다.
3. AI 서비스가 정상적으로 실행 중인지 확인합니다.
   - 시스템 관리자에게 Dify 서비스의 상태를 확인 요청합니다.

---

### 9.5. AI 진단 결과가 너무 일반적인 경우

**증상:**

이슈가 정상적으로 생성되기는 하지만, AI 진단 내용이 "복잡도를 줄이십시오", "리팩터링을 권장합니다"와 같은 일반적인 내용만 포함되어 있습니다.

**원인:**

AI가 프로젝트에 대한 구체적인 정보를 가지고 있지 않습니다.

**해결 절차:**

1. 8절에서 안내한 대로, Job #1~#3을 먼저 실행하여 AI에 프로젝트 지식을 학습시킵니다.
2. 특히 Job #1(프로젝트 문서)과 Job #3(코드 구조)를 우선 실행합니다.
3. 지식 학습이 완료된 후, 품질 분석 파이프라인을 다시 실행합니다.
4. 이전 실행에서 생성된 이슈와 새로 생성된 이슈의 진단 내용을 비교하여 품질이 향상되었는지 확인합니다.

### 9.6. 프로젝트 식별자를 찾을 수 없는 경우

**오류 메시지:**

```text
Component key 'my-project' not found
```

**원인:**

입력한 프로젝트 식별자(`SONAR_PROJECT_KEY`)가 코드 품질 분석 도구에 등록된 값과 일치하지 않습니다.

**해결 절차:**

1. 3.1절의 안내에 따라 프로젝트 식별자를 다시 확인합니다.
2. 주소창의 `id=` 뒤에 오는 값을 정확하게 복사합니다.
3. 대소문자, 하이픈(`-`), 밑줄(`_`) 등이 정확한지 확인합니다.
4. 확인한 값으로 파이프라인을 다시 실행합니다.

---

### 9.7. 파이프라인 실행이 오래 걸리는 경우

**증상:**

파이프라인이 30분 이상 실행 중이며 완료되지 않습니다.

**원인:**

`MAX_ISSUES`를 크게 설정했거나, AI 분석 서비스의 응답이 느린 상태입니다.

**해결 절차:**

1. Jenkins에서 현재 실행 중인 빌드의 `Console Output`을 확인하여 어느 단계에서 멈춰 있는지 파악합니다.
2. AI 분석 단계에서 시간이 오래 걸리는 경우:
   - `MAX_ISSUES` 값을 줄여서 다시 실행합니다 (예: `5`로 설정).
   - AI 서비스의 상태를 시스템 관리자에게 확인합니다.
3. 현재 실행을 중단하려면:
   - 빌드 상세 화면에서 좌측의 빨간색 `X` 아이콘(또는 `빌드 중단`)을 클릭합니다.
   - 실행이 중단됩니다.
   - `MAX_ISSUES`를 낮은 값으로 조정한 후 다시 실행합니다.

### 9.8. 이슈 관리 시스템에서 프로젝트를 찾을 수 없는 경우

**오류 메시지:**

```text
Project not found: root/my-project
```

**원인:**

입력한 프로젝트 경로(`GITLAB_PROJECT`)가 이슈 관리 시스템에 존재하지 않습니다.

**해결 절차:**

1. 3.2절의 안내에 따라 프로젝트 경로를 다시 확인합니다.
2. 이슈 관리 시스템에 직접 접속하여 해당 프로젝트가 존재하는지 확인합니다.
3. 프로젝트 경로에 앞뒤 슬래시가 포함되어 있지 않은지 확인합니다.
   - 올바른 예: `root/dscore-ttc-sample`
   - 잘못된 예: `/root/dscore-ttc-sample/`
4. 확인한 경로로 파이프라인을 다시 실행합니다.

## 부록 A. 파이프라인 실행 흐름 요약

파이프라인은 실행 시 다음 세 단계를 순서대로 수행합니다.

```text
[1단계]                    [2단계]                    [3단계]
코드 품질 분석 도구    -->  AI 분석 엔진          -->  이슈 관리 시스템
에서 문제 목록을            이 각 문제를 읽고          에 진단 결과를
가져옵니다                  원인과 수정 방안을         이슈로 자동
                            작성합니다                 등록합니다
```

**1단계 -- 문제 수집:**

파이프라인이 코드 품질 분석 도구에 접속하여, 입력한 조건(심각도, 상태)에 맞는 문제 목록을 가져옵니다. 각 문제에 대해 해당 코드 내용(문제가 있는 줄과 그 주변 코드)도 함께 수집합니다.

**2단계 -- AI 분석:**

수집된 각 문제를 AI에게 전달합니다. AI는 문제의 내용과 코드를 읽고, 다음 세 가지를 작성합니다.
- 이 문제가 왜 발생했는지 (원인 분석)
- 이 문제를 방치하면 어떤 영향이 있는지 (영향 범위)
- 어떻게 고쳐야 하는지 (수정 방안)

이때 AI가 프로젝트 지식(Job #1~#3으로 학습시킨 내용)을 보유하고 있다면, 프로젝트의 실제 맥락을 반영한 더 구체적인 진단을 제공합니다.

**3단계 -- 이슈 등록:**

AI가 작성한 진단 결과를 이슈 관리 시스템에 이슈로 등록합니다. 이때 중복 확인을 수행하여, 이미 등록된 문제는 건너뜁니다.
