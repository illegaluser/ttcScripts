# 품질 분석 파이프라인 (Job #4)

---

## 코드 리뷰, 이렇게 하고 계시지 않나요?

정적 분석 도구가 "순환 복잡도 25"라고 알려줍니다.
화면을 한참 들여다봅니다.
그래서... 어쩌라는 건지 잘 모르겠습니다.

결국 검색을 시작합니다. 순환 복잡도가 무엇인지, 25라는 숫자가 얼마나 심각한 건지, 어떤 부분을 고쳐야 하는 건지.
원인을 파악하고 나면 이제 GitLab에 이슈를 작성해야 합니다.
제목을 쓰고, 원인을 설명하고, 수정 방안을 정리합니다.

이슈 하나를 등록하는 데 10분.
분석 결과에 이슈가 20개라면, 이 작업만으로 반나절이 지나갑니다.

그리고 다음 날, 동료가 같은 이슈를 또 등록합니다.
"아, 이거 어제 제가 이미 올렸는데요."

익숙한 상황이시라면, 이 파이프라인이 도움이 될 수 있습니다.

---

## 이 기능이 해결하는 문제

품질 분석 파이프라인은 정적 분석 결과를 읽고, 해석하고, 이슈로 등록하는 전체 과정을 자동화합니다.

### 도입 전

SonarQube에서 분석 결과가 나오면, 사람이 직접 한 건씩 읽고 해석합니다.
문제의 원인을 파악한 뒤, GitLab에 접속하여 이슈 제목과 설명을 직접 작성합니다.
이슈 하나를 등록하는 데 평균 10분이 걸립니다.
같은 문제를 다른 팀원이 중복 등록하는 일도 종종 발생합니다.

### 도입 후

SonarQube의 분석 결과를 AI가 자동으로 읽고, 각 문제의 원인과 수정 방안을 정리하여
GitLab에 이슈를 자동으로 등록합니다.
이슈 하나를 등록하는 데 약 10초입니다.
이미 등록된 이슈와 같은 문제는 자동으로 건너뜁니다.

사람이 하던 반복 작업을 AI가 대신 수행하고,
개발자는 실제 코드 수정에 집중할 수 있습니다.

---

## AI가 제공하는 진단의 수준

이 파이프라인의 핵심은 단순한 자동화가 아닙니다.
AI가 제공하는 진단의 품질이 기존 도구와 근본적으로 다릅니다.

### 일반적인 정적 분석 도구의 결과

정적 분석 도구는 규칙 위반 사실만 알려줍니다.

"Method has a complexity of 25 which is greater than 15 authorized."

이 메시지를 받은 개발자는 스스로 다음 질문에 답해야 합니다.
복잡도 25가 왜 문제인지, 어떤 부분이 복잡한 건지, 어떻게 나눠야 하는지.
이 과정에서 시간이 소요되고, 경험에 따라 해석의 품질이 달라집니다.

### DSCORE-TTC AI의 진단

AI는 해당 프로젝트의 코드를 이해한 상태에서 진단합니다.

"이 메서드는 인증 검증과 권한 확인을 하나의 함수에서 모두 처리하고 있어
순환 복잡도가 25입니다.
AuthService.validateToken()과 PermissionChecker.check()로 분리하면
각각의 복잡도를 10 이하로 줄일 수 있습니다."

규칙 위반을 알려주는 데 그치지 않고,
왜 그런 문제가 발생했는지 근본 원인을 설명하고,
어떻게 수정해야 하는지 구체적인 방안을 제시합니다.

여기서 한 가지 더 알아두시면 좋은 점이 있습니다.
DSCORE-TTC의 지식 관리 파이프라인(Job #1 ~ #3)을 통해 프로젝트의 설계 문서, 아키텍처 정보,
코딩 컨벤션 등을 미리 학습시켜 두면, AI의 진단이 더욱 구체적이고 정확해집니다.
프로젝트의 맥락을 아는 AI와, 범용적인 규칙만 아는 도구의 차이는
실제로 사용해 보시면 바로 체감하실 수 있습니다.

## 사용 방법

품질 분석 파이프라인의 실행은 네 단계로 완료됩니다.

### 1단계. SonarQube에서 프로젝트 식별자 확인

SonarQube에 접속하여 분석 대상 프로젝트의 프로젝트 키를 확인합니다.
프로젝트 정보 페이지에서 확인할 수 있으며, 일반적으로 프로젝트명과 유사한 형태입니다.

### 2단계. GitLab에서 이슈 등록 대상 프로젝트 경로 확인

분석 결과를 이슈로 등록할 GitLab 프로젝트의 경로를 확인합니다.
GitLab 프로젝트 페이지 상단에 표시되는 "그룹명/프로젝트명" 형태의 경로입니다.

### 3단계. Jenkins에서 파라미터 입력

Jenkins에서 품질 분석 파이프라인을 선택하고 다음 다섯 가지 항목을 입력합니다.

- **SonarQube 프로젝트 키**: 1단계에서 확인한 프로젝트 식별자
- **GitLab 프로젝트 경로**: 2단계에서 확인한 이슈 등록 대상 경로
- **분석 대상 심각도**: 어느 수준의 문제부터 분석할 것인지 선택 (예: BLOCKER, CRITICAL)
- **분석할 이슈 수**: 한 번에 처리할 이슈의 최대 건수
- **지식 저장소 이름**: 프로젝트 학습 데이터가 저장된 저장소 (지식 관리 파이프라인을 사용한 경우)

### 4단계. 빌드 실행

"Build" 버튼을 누르면 파이프라인이 시작됩니다.
진행 상황은 Jenkins의 콘솔 화면에서 실시간으로 확인할 수 있습니다.

### 결과 확인

파이프라인이 완료되면 GitLab 프로젝트의 Issues 탭에
AI가 작성한 이슈들이 자동으로 등록되어 있습니다.
별도의 추가 작업 없이, 등록된 이슈를 검토하고 담당자를 지정하시면 됩니다.

## 자동 생성된 이슈에 포함되는 내용

AI가 GitLab에 자동으로 등록하는 이슈에는 다음과 같은 내용이 포함됩니다.

### 제목

문제의 유형과 해당 파일명이 한눈에 파악되도록 구성됩니다.
어떤 종류의 문제가 어디에서 발견되었는지 제목만으로 알 수 있습니다.

### 원인 분석

AI가 분석한 문제의 근본 원인이 서술됩니다.
단순히 "규칙 위반"이라고 표기하는 것이 아니라,
해당 코드가 왜 문제를 일으키는지 맥락을 포함하여 설명합니다.

### 영향 범위

이 문제를 방치했을 때 발생할 수 있는 위험을 안내합니다.
보안 취약점인 경우 어떤 공격에 노출될 수 있는지,
성능 문제인 경우 어떤 상황에서 장애가 발생할 수 있는지 구체적으로 기술합니다.

### 수정 방안

문제를 해결하기 위한 구체적인 코드 수정 방향을 제안합니다.
어떤 부분을 어떻게 변경하면 되는지, 실제 개발 작업에 바로 참고할 수 있는 수준으로 작성됩니다.

### 중복 방지

이미 GitLab에 동일한 문제에 대한 이슈가 등록되어 있는 경우,
해당 이슈는 자동으로 건너뜁니다.
같은 이슈가 두 번 등록되는 실수를 시스템 차원에서 방지합니다.

---

## 이런 상황에서 활용해 보십시오

### 스프린트 단위 기술 부채 관리

스프린트가 끝날 때마다 정기적으로 실행하면,
새로 발생한 기술 부채를 빠짐없이 이슈로 등록하고 추적할 수 있습니다.
기술 부채가 눈에 보이지 않는 곳에서 쌓이는 것을 방지합니다.

### 신규 프로젝트의 품질 기준선 확인

새로 시작하는 프로젝트나 인수받은 프로젝트의 현재 코드 품질 상태를
빠르게 파악하고 싶을 때 유용합니다.
전체적인 품질 수준과 주요 개선 포인트를 한 번의 실행으로 확인할 수 있습니다.

### 보안 감사 전 사전 점검

보안 감사나 외부 점검이 예정되어 있을 때,
사전에 보안 관련 이슈를 자동으로 수집하고 정리하는 데 활용할 수 있습니다.
취약점을 미리 파악하고 대응 계획을 수립하는 시간을 확보할 수 있습니다.

### 단계적 품질 개선

처음에는 가장 심각한 수준의 이슈만 분석하고,
팀이 이 이슈들을 해결한 뒤 점차 분석 범위를 넓혀 가는 방식으로 운영하면
부담 없이 코드 품질을 지속적으로 개선해 나갈 수 있습니다.

---

## 참고 사항

처음 사용하실 때는 심각도를 BLOCKER와 CRITICAL로 설정하고,
분석 건수도 5건 정도로 시작하시는 것을 권장합니다.
결과를 확인하신 후 팀의 상황에 맞게 범위를 조정하시면 됩니다.

정기 실행을 설정해 두시면 매일 또는 원하시는 주기마다
자동으로 분석이 수행되어, 별도로 실행하지 않아도 이슈가 관리됩니다.

AI가 제안하는 수정 방안은 참고 자료입니다.
최종적인 코드 수정 판단은 해당 코드를 가장 잘 아는 담당 개발자가 하십시오.
AI의 제안을 출발점으로 삼되, 프로젝트의 상황에 맞게 조정하시는 것이 가장 좋습니다.
