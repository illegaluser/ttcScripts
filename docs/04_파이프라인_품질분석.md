# 품질 분석 파이프라인 (Job #4)

## 1. 개요

품질 분석 파이프라인은 SonarQube의 정적 분석 결과를 AI가 자동으로 진단하고, 그 결과를 GitLab Issue로 등록하는 기능입니다.

기존에는 SonarQube가 발견한 코드 문제를 사람이 하나씩 읽고, 원인을 파악하고, GitLab에 이슈를 수동으로 등록해야 했습니다. 이 파이프라인은 해당 과정 전체를 자동화하여, 이슈당 수 분이 소요되던 작업을 수 초 내에 처리합니다.

---

## 2. 기능 설명

### 2.1. 처리 흐름

| 단계 | 수행 주체 | 내용 |
| --- | --- | --- |
| 1 | SonarQube | 소스 코드를 분석하여 버그, 취약점, 코드 스멜을 발견합니다 |
| 2 | Python 스크립트 | SonarQube에서 이슈 목록을 추출하고, 해당 코드 라인의 스니펫을 함께 가져옵니다 |
| 3 | Dify Workflow (AI) | 추출된 이슈를 AI가 분석하여 문제의 원인, 영향 범위, 수정 방안을 자연어로 작성합니다 |
| 4 | Python 스크립트 | AI의 진단 결과를 GitLab Issue 형식으로 정리하여 자동 등록합니다 |

### 2.2. AI 진단의 특징

이 파이프라인의 AI는 단순히 SonarQube의 규칙 설명을 번역하는 것이 아니라, 프로젝트 맥락을 반영한 구체적인 진단을 제공합니다.

| 비교 항목 | 일반 정적 분석 도구의 출력 | DSCORE-TTC AI의 진단 |
| --- | --- | --- |
| 진단 내용 | "Method complexity is 25, exceeding the authorized limit of 15" | "이 메서드는 인증 검증과 권한 확인을 하나의 함수에서 처리하고 있어 복잡도가 25입니다" |
| 수정 방안 | "Refactor this method to reduce its complexity" | "AuthService.validateToken()과 PermissionChecker.check()로 분리하면 각각 10 이하로 줄일 수 있습니다" |
| 근거 | 규칙 문서 링크 | 프로젝트 코드와 설계 문서를 참조한 분석 |

지식 관리 파이프라인(Job #1~#3)으로 프로젝트 문서와 코드를 사전에 학습시켜 두면, AI가 프로젝트 구조를 이해한 상태에서 진단하므로 위와 같은 수준의 구체적인 수정 방안을 제안할 수 있습니다.

### 2.3. 자동 생성되는 GitLab 이슈의 구성

파이프라인이 GitLab에 등록하는 각 이슈에는 다음 항목이 포함됩니다.

| 항목 | 설명 |
| --- | --- |
| 제목 | 심각도, 문제 유형, 해당 파일명과 라인 번호가 포함된 요약입니다 |
| 원인 분석 | AI가 분석한 문제의 근본 원인입니다 |
| 영향 범위 | 해당 문제를 방치할 경우 발생할 수 있는 구체적인 위험입니다 |
| 수정 방안 | AI가 제안하는 코드 수정 방법입니다. 수정 전/후 코드 예시가 포함되는 경우도 있습니다 |
| 코드 위치 | 문제가 발생한 파일명, 라인 번호, 해당 코드 스니펫입니다 |
| SonarQube 링크 | 코드 품질 분석 도구에서 해당 문제를 직접 확인할 수 있는 URL입니다 |
| 라벨 | SonarQube 이슈 고유 식별자가 부착됩니다. 이를 통해 중복 등록이 자동으로 방지됩니다 |

### 2.4. 중복 방지

파이프라인을 반복 실행해도 동일한 문제에 대한 이슈가 중복으로 생성되지 않습니다. SonarQube 이슈의 고유 식별자를 GitLab 라벨로 부착하여, 해당 라벨이 이미 존재하는 이슈는 자동으로 건너뜁니다.

## 3. 사용 절차

### 3.1. 사전 준비

| 항목 | 확인 방법 |
| --- | --- |
| SonarQube에 대상 프로젝트가 등록되어 있고 최소 1회 분석이 수행됨 | `http://localhost:9000` → Projects 메뉴에서 프로젝트 확인 |
| GitLab에 이슈를 등록할 대상 프로젝트가 존재함 | `http://localhost:8929` → 해당 프로젝트 페이지 접속 확인 |
| Dify에 품질 분석용 Workflow가 구성됨 | Dify 웹 UI에서 해당 Workflow 존재 확인 |
| Jenkins Credentials 등록 완료 | `sonarqube-token`, `gitlab-access-token`, `dify-knowledge-key` 등록 확인 |

### 3.2. 실행 절차

1. Jenkins(`http://localhost:8080`)에서 `DSCORE-Quality-Issue-Workflow` Job을 선택합니다.
2. `Build with Parameters`를 클릭합니다.
3. 다음 파라미터를 입력합니다.

| 파라미터 | 설명 | 입력 예시 |
| --- | --- | --- |
| `SONAR_PROJECT_KEY` | SonarQube 프로젝트 식별자. SonarQube 웹 UI에서 프로젝트를 클릭한 후 브라우저 주소창의 `id=` 뒤의 값입니다 | `dscore-ttc-sample` |
| `GITLAB_PROJECT_PATH` | GitLab 프로젝트 경로. GitLab 웹 UI에서 프로젝트 페이지 URL의 도메인 뒤 경로입니다 | `root/dscore-ttc-sample` |
| `SEVERITIES` | 처리할 이슈의 심각도 수준 (쉼표 구분) | `BLOCKER,CRITICAL,MAJOR` |
| `STATUSES` | 처리할 이슈의 상태 (쉼표 구분) | `OPEN,REOPENED` |
| `MAX_ISSUES` | 한 번에 처리할 최대 이슈 건수. 0 입력 시 전체 처리 | `10` |

4. `빌드` 버튼을 클릭합니다.

심각도 기준은 다음과 같습니다.

| 심각도 | 의미 |
| --- | --- |
| BLOCKER | 운영 장애를 유발할 수 있는 치명적 결함 |
| CRITICAL | 보안 취약점 또는 심각한 버그 |
| MAJOR | 코드 품질에 큰 영향을 미치는 문제 |
| MINOR | 코딩 규칙 위반, 스타일 문제 |
| INFO | 참고 수준의 개선 제안 |

처음 실행할 때는 `BLOCKER,CRITICAL` 수준만 5건 정도로 시작하여, 결과를 확인한 후 범위를 넓히는 것을 권장합니다.

### 3.3. 결과 확인

빌드 완료 후 GitLab(`http://localhost:8929`) → 해당 프로젝트 → 좌측 메뉴 `Issues`에서 자동 생성된 이슈를 확인할 수 있습니다.

SonarQube(`http://localhost:9000`)에서는 원본 분석 결과를 직접 확인할 수 있습니다. 프로젝트를 선택하면 `Overview` 화면에서 Quality Gate 상태(Passed/Failed)를, `Issues` 탭에서 발견된 문제의 상세 코드 라인을 확인할 수 있습니다.

## 4. 활용 범위

| 활용 시나리오 | 설명 |
| --- | --- |
| 스프린트별 기술 부채 관리 | 정기적으로 실행하여 신규 발생 이슈를 자동 등록하고, 스프린트 단위로 해소할 수 있습니다 |
| 신규 프로젝트 품질 기준선 확인 | 프로젝트 초기에 전체 이슈를 한 번에 추출하여 현재 코드 품질 수준을 파악할 수 있습니다 |
| 보안 감사 사전 점검 | CRITICAL 이상의 보안 취약점만 추출하여 감사 전에 선제적으로 대응할 수 있습니다 |
| 정기 자동 실행 | Jenkins의 빌드 트리거에 주기를 설정하면 매일 또는 매주 자동으로 분석을 수행합니다 |

---

## 5. 참고 사항

- AI의 수정 제안은 참고 자료입니다. 최종 수정 판단은 담당 개발자가 프로젝트 상황에 맞게 결정하십시오.
- 지식 관리 파이프라인(Job #1~#3)을 사전에 실행하여 프로젝트를 학습시키면 진단 품질이 크게 향상됩니다.
- 모든 데이터 처리는 로컬 서버에서 이루어지며, 외부로 전송되지 않습니다.
