# Product Requirements Document  
## On-Premise AI DevOps·QA Co-pilot Platform  

- **문서명**: Product Requirements Document  
- **적용 범위**: Phase 1 요구사항을 기준으로 정의하며, Phase 2 이후 확장 방향을 포함한다.  
- **운영 전제**: 내부망(폐쇄망) 온프레미스 환경에서 전 과정이 완결되어야 한다.  

---

## 1. 제품 정의  

### 1.1 제품 한 줄 정의  
본 문서는 내부망에서 문서·코드·웹·이미지 지식을 통합하고, 근거 기반 Q&A, 정적 분석(Triage) 자동화, 반복 QA 산출물 초안 생성, 폐쇄망 E2E 테스트 자동화를 제공하는 DevOps·QA 보조 플랫폼에 대한 내용을 담고 있다.  

### 1.2 제품이 제공하는 핵심 가치  
아래 가치는 “기능 제공”이 아니라 “운영 성과” 관점으로 정의한다.  

- **지식 접근 비용 절감**  
  - 산재된 문서/코드/웹/이미지를 단일 지식베이스로 통합한다.  
  - 질문 1회로 관련 근거를 검색·요약·정리하여 답변한다.  
  - 답변에 근거 출처를 포함하여 검증 가능한 형태를 유지한다.  

- **품질 이슈 처리 리드타임 단축**  
  - SonarQube 이슈를 AI가 해석하여 영향·수정 가이드·추가 점검 항목을 구조화한다.  
  - GitLab 이슈 등록까지 자동 연결하여 “수집→해석→등록”을 단일 흐름으로 만든다.  
  - 중복 등록을 방지하여 이슈 큐의 신뢰성과 추적성을 유지한다.  

- **반복 QA 문서 작성 부담 완화**  
  - 테스트 계획/시나리오/결과 보고 등 반복 산출물을 표준 템플릿으로 초안 생성한다.  
  - 누락 정보는 “확인 필요 항목”으로 분리하여 추가 확인을 유도한다.  
  - 초안 작성 시간을 줄이고, 검토·합의·개선에 시간을 재배분한다.  

- **폐쇄망 테스트 자동화 실행 기반 확보**  
  - 외부 서비스 의존 없이 내부 웹앱 대상으로 E2E 테스트를 생성·실행·리포팅한다.  
  - 실행 증적(로그/스크린샷/리포트)을 축적하고 지식베이스에 재적재한다.  
  - 테스트 산출물이 다음 테스트/분석/문서 작성의 근거로 재사용되도록 한다.

## 2. 문제 정의  

문제는 “현상”이 아니라 “반복 비용과 리스크”로 정의한다.  

### 2.1 정보 분산과 탐색 비용  
- 지식이 저장소별로 분리되어 단일 검색으로 파악되지 않는다.  
  - 예: 정책 문서는 문서 저장소에 존재하고, 구현 근거는 코드 저장소에 존재하며, 운영 이력은 내부 웹에 존재한다.  
- 동일한 질문을 해결하기 위해 “찾기→열람→비교→요약→공유” 과정이 반복된다.  
- 결과적으로 업무 처리 시간이 증가하고, 의사결정 근거가 개인에게 종속된다.  

### 2.2 반복 산출물 작성 비용과 품질 편차  
- 테스트 계획/시나리오/결과 보고 등은 매 릴리즈, 매 프로젝트마다 반복된다.  
- 초안 작성 방식이 개인별로 달라 산출물 품질이 일정하지 않다.  
- 문서 작성 시간이 늘수록 테스트 설계/리스크 검토에 투자되는 시간이 감소한다.  

### 2.3 SonarQube 이슈 해석·등록 업무의 누적  
- 이슈는 자동 생성되나, “의미 해석”과 “조치 계획화”는 수작업으로 남는다.  
- 해석 지연은 방치 이슈 증가로 이어지며, 기술부채가 운영 리스크로 전환된다.  
- 동일 유형 이슈가 반복되나, 조직 차원의 표준 수정 가이드가 누적되지 않는다.  

### 2.4 보안 제약 환경에서의 AI 활용 공백  
- 외부 클라우드 AI 사용이 제한되거나 금지되는 환경이 존재한다.  
- 내부망에서 완결되는 AI 활용 체계가 부재하면 자동화 도입이 지연된다.  

### 2.5 테스트 자동화의 도입 장벽  
- E2E 자동화는 작성/유지보수/실행/리포팅까지 운영 체계가 필요하다.  
- 폐쇄망에서는 외부 분석 도구, 외부 저장소, 외부 실행 인프라에 의존하기 어렵다.  
- 따라서 “폐쇄망에서 완결되는 자동화 실행 기반”이 먼저 확보되어야 한다.

## 3. 제품 원칙  

원칙은 설계/구현 시 충돌을 줄이기 위한 “우선순위 규칙”이다.  

### 3.1 보안 원칙  
- 데이터(문서/코드/이미지/로그/산출물)는 외부로 전송되지 않는다.  
- 토큰/비밀 값은 안전 저장소로 관리되며, 평문 노출을 금지한다.  
- 로그에는 민감정보가 최소화되어야 하며, 필요 시 마스킹 정책을 적용한다.  

### 3.2 AI 적용 원칙  
- AI 결과물은 “초안”으로 정의한다.  
- 결과물에는 근거 출처 또는 근거 범위를 포함한다.  
- 근거가 부족하거나 불확실한 내용은 “확인 필요”로 분리 표기한다.  

### 3.3 운영 원칙  
- 자동화는 Jenkins 파이프라인 단위로 운영 가능해야 한다.  
- 스케줄 실행과 수동 실행이 모두 가능해야 한다.  
- 실패는 재실행 가능한 형태로 기록되고, 재처리를 위한 산출물을 남겨야 한다.  

### 3.4 재사용 원칙  
- 문서/테스트/리포트 산출물은 지식베이스에 재적재되어 재사용 가능해야 한다.  
- “생성→사용→축적”이 반복되도록 설계한다.

## 4. 목표와 단계  

### 4.1 Phase 1 목표  
Phase 1은 “지식 통합”, “근거 기반 응답”, “품질 이슈 자동 triage”, “반복 문서 초안”, “폐쇄망 테스트 자동화 기반”을 최소 기능 집합으로 완성한다.  

- **지식베이스 구축(문서·코드·웹·이미지)**  
  - 지정된 입력 소스를 자동 수집한다.  
  - 불필요 요소를 제거하고(정제), 검색 가능한 형태로 변환한다(표준화).  
  - 적재 결과는 검색/응답에 활용 가능해야 한다(인덱싱).  
  - 운영 환경에서 정기 업데이트가 가능해야 한다(스케줄).  

- **Q&A 에이전트 제공(근거·출처 포함)**  
  - 질문을 입력하면 관련 근거를 검색하여 답변을 생성한다.  
  - 답변은 단일 문서 요약이 아니라 다수 근거 종합을 기본값으로 한다.  
  - 답변은 출처(문서명/경로/URL 등)를 포함하여 검증 가능해야 한다.  
  - 답변은 QA/운영 문맥에서 “절차/체크리스트/재현 단계” 형태를 우선 제공한다.  

- **정적 분석 Triage 자동화(SonarQube → AI 분석 → GitLab 이슈)**  
  - SonarQube 이슈를 자동 수집한다(필터링 포함).  
  - 이슈별로 요약/영향/수정 가이드/예시/추가 점검 항목을 생성한다.  
  - GitLab 이슈 생성까지 자동 연결한다(제목 규칙/라벨 포함).  
  - 중복 등록을 방지한다(동일 이슈 또는 동일 원인군 기준).  
  - 분석 입력·출력은 산출물로 보존되어야 한다(재처리/감사 목적).  

- **QA 문서 초안 자동 생성(표준 템플릿 기반)**  
  - 테스트 계획/시나리오/결과 보고 등 문서 초안을 구조화된 마크다운으로 생성한다.  
  - 지식베이스 근거를 반영하여 “문서 빈칸”을 최소화한다.  
  - 누락 정보는 확인 질문 또는 확인 항목으로 분리 제시한다.  
  - 책임 한계 고지 문구를 포함한다(초안/검토 필요 명시).  

- **테스트 자동화 실행 기반 구축(폐쇄망 E2E)**  
  - 내부 웹앱을 대상으로 E2E 테스트를 생성한다.  
  - 생성된 테스트를 실행하고, 결과 리포트를 산출한다.  
  - 실패 증적(스크린샷/로그/실행 기록)을 남긴다.  
  - 테스트 코드/리포트는 지식베이스에 재적재되어 Q&A와 분석 근거로 활용된다.  
  - 자동 생성 테스트는 “초안”으로 취급되며, 운영 반영 전 검토·수정이 전제된다.  

### 4.2 Phase 2 이후 확장 방향  
Phase 2 이후는 “개발 생산성 보조”와 “품질 운영 분석 고도화”를 확장한다.  

- **개발 생산성 보조**  
  - 코드 리뷰 보조(리뷰 포인트 추천, 리스크 요약, 체크리스트 제시)  
  - 리팩터링 가이드(대안 비교, 영향 범위, 점검 항목 제공)  
  - 단위 테스트 초안 생성(핵심 경로, 경계값, 예외 케이스 포함)  

- **품질 운영 분석 고도화**  
  - UI 테스트 결과 해설(실패 원인 후보, 재현 조건, 개선 제안)  
  - 성능/SEO 등 운영 리포트 자동 생성(지표 요약, 추세, 조치 권고)
  - 품질 기준 표준화(템플릿/가이드/운영 정책의 조직 내 내재화)

## 5. 시스템 개요  

### 5.1 운영 환경  
- 온프레미스 환경에서 동작한다.  
- 외부망 차단을 전제로 한다.  
- 서비스 간 통신은 내부 네트워크(컨테이너 기준)로 정의한다.  

### 5.2 구성 요소와 역할  
- **Jenkins**  
  - 파이프라인 실행과 스케줄링을 담당한다.  
  - 배치 처리(정기 지식 업데이트, triage 실행)를 담당한다.  
  - 실행 이력과 산출물 아카이빙을 담당한다.  

- **GitLab**  
  - 소스 저장소 역할을 담당한다.  
  - 이슈 트래킹 저장소 역할을 담당한다.  
  - triage 결과의 최종 기록 지점이 된다.  

- **SonarQube**  
  - 정적 분석 결과(이슈/룰/코드 스니펫)를 제공한다.  
  - triage 자동화의 입력 소스가 된다.  

- **Dify**  
  - 지식베이스(검색 인덱스)와 RAG 기반 응답을 제공한다.  
  - 워크플로 실행(예: Sonar 이슈 분석)을 제공한다.  
  - 챗봇 인터페이스를 제공한다.  

- **Ollama(로컬 LLM 엔진)**  
  - 코드/추론, 비전 분석 등 로컬 모델 실행을 담당한다.  
  - 외부 AI 호출 없이 내부망에서 생성 기능을 제공한다.  

### 5.3 데이터 분류(운영 관점)  
- **입력 데이터**  
  - 문서 파일, 이미지 파일, 코드 저장소, 내부 웹 콘텐츠, Sonar 이슈 데이터  
- **중간 산출물**  
  - 정제된 마크다운, 추출 텍스트, 컨텍스트 요약, triage 분석 결과, 테스트 실행 로그  
- **최종 산출물**  
  - Q&A 응답, GitLab 이슈, QA 문서 초안, E2E 리포트

## 6. 기능 요구사항  

아래 항목은 “무엇을 제공해야 하는지”를 상세 요구사항으로 정의한다.  
목록화가 유효한 항목은 목록으로 구조화하고, 맥락이 필요한 항목은 서술형을 병행한다.  

### 6.1 지식베이스 구축  

#### 6.1.1 공통 요구사항  
- 수집 대상은 문서/코드/웹/이미지를 포함한다.  
- 수집 과정은 “수집→정제→표준화→적재” 순서를 따른다.  
- 적재 결과는 검색 가능한 형태로 유지되어야 한다.  
- 운영 중 정기 업데이트(스케줄 실행)를 지원해야 한다.  
- 실패 시 재처리를 위해 실패 원인과 대상 항목이 식별 가능해야 한다.  

#### 6.1.2 문서 지식화  
- **입력 범위**  
  - TXT, DOCX, XLS/XLSX, PDF, PPT/PPTX를 지원한다.  
- **정제·표준화 정책**  
  - 텍스트 추출이 가능한 경우 텍스트를 우선 보존한다.  
  - 텍스트 추출이 불가능하거나 이미지 기반 문서는 OCR/비전 분석을 선택 적용한다.  
  - 표/목록/제목 구조는 가능한 한 마크다운 구조로 보존한다.  
- **적재 정책**  
  - 문서형 지식(원문 중심)과 Q&A형 지식(질의응답 최적화)은 목적에 따라 구분 운영할 수 있어야 한다.  
  - 문서별 메타정보(문서명/경로/작성일 또는 스캔일)를 함께 관리해야 한다.  

#### 6.1.3 웹 지식화  
- **대상 범위**  
  - 내부 웹, 내부 블로그, 내부 위키 등 내부망 접근 가능한 웹 콘텐츠를 포함한다.  
- **수집 방식**  
  - 렌더링 기반 수집을 지원한다.  
  - SPA는 브라우저 렌더링 후 본문을 수집한다.  
- **정제 정책**  
  - 메뉴/광고/사이드바/푸터 등 반복 UI 요소를 제거한다.  
  - 본문 중심으로 구조화하여 마크다운으로 변환한다.  
- **적재 정책**  
  - URL과 수집 시점을 메타정보로 저장한다.  
  - 동일 URL 재수집 시 변경 여부를 추적할 수 있어야 한다(변경 감지 정책은 운영 정책으로 정의).  

#### 6.1.4 코드 컨텍스트 지식화  
- **입력 범위**  
  - GitLab 저장소를 기준으로 리포지토리 단위로 처리한다.  
- **표준화 정책**  
  - 저장소 트리 구조를 요약하여 “프로젝트 지형도”를 제공한다.  
  - 핵심 파일(예: README, 설정 파일, 주요 엔트리포인트) 내용을 기반으로 시스템 개요를 요약한다.  
  - 경로/파일명은 원본을 보존하여 추적 가능해야 한다.  
- **적재 정책**  
  - 컨텍스트 요약은 Q&A 및 triage 분석의 공통 근거로 재사용 가능해야 한다.  

#### 6.1.5 이미지 지식화  
- **입력 범위**  
  - JPG/PNG/BMP/WEBP를 지원한다.  
- **표준화 정책**  
  - 비전 모델 분석으로 이미지 설명(무엇이 있는지, 어떤 의미인지)을 마크다운으로 생성한다.  
  - 문서/화면 캡처의 경우 핵심 텍스트 요소와 구조를 가능한 범위에서 설명한다.  
- **적재 정책**  
  - 이미지 파일명/경로를 출처로 제공할 수 있어야 한다.  
  - 이미지 설명은 Q&A와 문서 초안 생성의 근거로 활용된다.

### 6.2 Q&A 에이전트  

#### 6.2.1 기본 동작  
- 임베딩 기반 검색 후 답변 생성(RAG)을 기본 동작으로 한다.  
- 단일 문서 요약이 아니라 다수 근거 종합을 기본값으로 한다.  
- 응답에는 출처를 포함하여 사용자가 근거를 검증할 수 있어야 한다.  

#### 6.2.2 출력 품질 요구사항  
- **구조화**  
  - 답변은 가능한 경우 “요약→근거→절차/체크리스트→주의사항” 구조를 따른다.  
- **검증 가능성**  
  - 출처는 문서명/경로/URL 등으로 표기되며, 재탐색 가능한 수준이어야 한다.  
- **불확실성 처리**  
  - 근거가 부족한 내용은 단정하지 않고 “확인 필요 항목”으로 분리한다.  
- **도메인 특화**  
  - QA/운영 문맥에서는 재현 절차, 조건 분기, 예외 케이스를 우선 제시한다.  

#### 6.2.3 기대 사용 예(형식 예시)  
- 질문: “이번 릴리즈에서 결제 실패 케이스 테스트 시나리오를 작성해 달라.”  
- 시스템 동작: 관련 정책/코드/장애 이력/기존 시나리오 템플릿을 검색하고, 초안 시나리오와 출처를 제공한다.  

---

### 6.3 정적 분석 Triage 자동화  

#### 6.3.1 입력(수집) 요구사항  
- SonarQube API 기반으로 이슈를 수집해야 한다.  
- 필터링 기준을 지원해야 한다.  
  - Severity(예: CRITICAL/MAJOR 등).  
  - Status(예: OPEN 등).  
  - Type(예: BUG/CODE_SMELL/VULNERABILITY 등).  
- 이슈별로 분석에 필요한 근거를 함께 구성해야 한다.  
  - 룰 설명(규칙 의미).  
  - 코드 스니펫(발생 지점).  
  - 이슈 URL/키/프로젝트 키.  

#### 6.3.2 분석(생성) 요구사항  
- 시스템은 이슈별 분석 결과를 구조화하여 생성해야 한다.  
  - 문제 요약(무엇이 문제인지).  
  - 영향(안정성/보안/유지보수 관점).  
  - 수정 가이드(권장 수정 방향과 주의점).  
  - 예시(가능한 경우 대체 코드/패턴).  
  - 추가 점검 항목(유사 이슈 탐색, 회귀 위험).  

#### 6.3.3 등록(GitLab) 요구사항  
- 분석 결과는 GitLab 이슈로 등록되어야 한다.  
- 등록 시 운영 규칙을 적용해야 한다.  
  - 제목 규칙(이슈의 식별성과 검색성을 보장하는 형식).  
  - 라벨 정책(예: auto-triage 등 자동 생성 식별).  
  - 본문 템플릿(요약/영향/가이드/근거/링크).  
- 중복 방지 정책을 적용해야 한다.  
  - 동일 Sonar 이슈 키 또는 동일 근거 URL 기준으로 중복 생성이 방지되어야 한다.  
  - 중복 판단 결과(스킵 사유)는 산출물로 기록되어야 한다.  

#### 6.3.4 산출물 보존 요구사항  
- 재처리와 감사 목적을 위해 아래 산출물을 보존해야 한다.  
  - 수집 결과(이슈 목록과 근거 포함).  
  - 분석 결과(이슈별 LLM 출력).  
  - 등록 결과(생성/스킵/실패 목록).  
- 산출물은 실행 시점과 상관관계를 유지해야 한다(이슈 키 기준).  

---

### 6.4 QA 문서 초안 생성  

#### 6.4.1 지원 문서 범위  
- 테스트 계획서.  
- 테스트 시나리오/테스트 케이스 초안.  
- 테스트 결과 보고서.  
- 품질 계획서(품질 목표/기준/리스크 관리).  
- 주간·월간 품질 리포트(이슈 추세/조치 현황/리스크).  

#### 6.4.2 생성 정책  
- 출력은 마크다운 구조를 기본으로 한다.  
- 조직 표준 템플릿을 적용할 수 있도록 섹션 구조를 유지한다.  
- 지식베이스 근거를 반영하여 “근거 없는 일반론”을 최소화한다.  
- 누락 정보는 다음 방식으로 처리한다.  
  - 추가 질문 목록 제시(사용자 입력 유도).  
  - 확인 필요 항목 분리 표기(근거 부족 명시).  

#### 6.4.3 책임 한계 표기  
- 문서 하단 또는 상단에 다음 취지의 고지 문구를 포함한다.  
  - 본 문서는 초안이며, 최종 검토와 승인 책임은 사용자에게 있다.  
  - 조직 정책/보안/릴리즈 기준에 따라 수정이 필요하다.  

---

### 6.5 테스트 자동화(폐쇄망 E2E)  

#### 6.5.1 목표 정의  
- 폐쇄망에서 E2E 테스트를 “생성–실행–리포트”까지 단일 운영 흐름으로 제공한다.  
- 자동 생성 코드는 운영 반영 전 검토·수정이 전제된다.  
- 산출물은 지식베이스에 재적재되어 다음 질문/분석/문서화의 근거가 된다.  

#### 6.5.2 기능 요구사항  
- **테스트 코드 자동 생성**  
  - 내부 웹앱의 사용자 흐름을 기반으로 테스트 스크립트 초안을 생성한다.  
  - 생성 결과는 저장·버전관리 가능한 형태여야 한다(코드 파일 단위).  
  - 생성 시나리오는 “핵심 사용자 경로”를 우선 대상으로 한다(로그인/주요 CRUD/결제 등).  

- **테스트 실행**  
  - 폐쇄망 환경에서 브라우저 기반 테스트를 실행한다.  
  - 실행은 Jenkins 파이프라인에서 트리거 가능해야 한다(수동/스케줄).  

- **리포트 및 증적 산출**  
  - 성공/실패 결과를 요약한 리포트를 산출한다.  
  - 실패 시 아래 증적을 기본 산출물로 남긴다.  
    - 실패 단계 로그.  
    - 스크린샷 또는 화면 증적.  
    - 실행 환경 정보(실행 시각, 대상 URL 등).  
  - 리포트는 사람 검토가 가능한 형태(마크다운 또는 HTML 등)로 제공한다.  

- **지식베이스 재적재**  
  - 테스트 코드, 리포트, 주요 실패 증적 설명을 지식베이스에 적재한다.  
  - 이후 Q&A에서 “최근 테스트 결과 기반” 답변이 가능하도록 한다.  

#### 6.5.3 운영 정책(초안 생성의 한계 관리)  
- 자동 생성 테스트는 “기본 경로 커버리지 확보”를 1차 목표로 한다.  
- 안정화 후 예외/경계/권한 분기 등 고난도 케이스를 점진 확장한다.  
- UI 변경 시 유지보수 비용을 낮추기 위해 선택자 전략과 리포트 기준을 표준화한다(상세 전략은 운영 정책에서 정의).

## 7. 비기능 요구사항  

### 7.1 보안  
- 외부 전송을 금지한다.  
- 비밀 값은 안전 저장소로 관리한다.  
- 로그는 민감정보 최소화를 기본값으로 한다.  
- 권한은 최소 권한 원칙을 따른다.  

### 7.2 성능 목표  
- Q&A 응답 목표는 30초 이내다.  
- 복합 질의 응답 목표는 60초 이내다.  
- 문서 초안 생성 목표는 180초 이내다.  
- 모델 자원 요구사항은 운영 정책에서 별도 정의한다(모델 크기/동시성/배치 전략).  

### 7.3 가용성 및 복구  
- Jenkins/GitLab/SonarQube/Dify는 데이터 볼륨 백업을 지원해야 한다.  
- 복구 절차는 구성요소 단위로 정의되어야 한다(서비스 복구, 데이터 복구 분리).  
- 파이프라인 실패는 재실행 가능해야 하며, 실패 지점이 식별 가능해야 한다.  

### 7.4 관측성과 감사  
- 파이프라인 실행 이력은 Job 단위로 추적 가능해야 한다.  
- 이슈 생성/스킵/실패 결과는 산출물로 남아야 한다.  
- triage 분석 입력과 출력은 이슈 키 기준으로 상관관계를 유지해야 한다.

## 8. 운영 전략  

### 8.1 도입 단계  
- **PoC**  
  - 폐쇄망 완결성, 핵심 기능 동작, 보안 준수 가능성을 검증한다.  
- **Pilot**  
  - 실프로젝트 1개에 적용하여 지표(시간 단축/리드타임 감소)를 측정한다.  
- **확산**  
  - 템플릿/운영 규칙/교육 자료를 표준화하여 프로젝트 적용 범위를 확장한다.  

### 8.2 운영 방식  
- 지식 동기화는 스케줄 기반 정기 실행을 기본값으로 한다.  
- triage는 일 단위 배치 또는 특정 트리거 기준 실행을 지원한다.  
- 프롬프트와 지식베이스 품질은 사용자 피드백과 운영 데이터 기반으로 개선한다.  
- 운영 가이드는 “AI는 초안” 원칙과 검토 책임을 명확히 포함한다.  

### 8.3 자원 대응  
- 모델 크기 조정 옵션을 제공한다(대형 모델/소형 모델 선택).  
- 비핵심 작업은 배치 처리로 전환 가능해야 한다(예: 야간 대량 지식화).

## 9. 리스크와 대응  

### 9.1 환각 및 근거 불충분 리스크  
- RAG 우선 전략을 기본값으로 한다.  
- 출처 표기를 강제하여 검증 가능성을 확보한다.  
- 불확실한 내용은 확인 필요 항목으로 분리한다.  

### 9.2 수용성 리스크  
- 제품 메시지는 “대체”가 아니라 “보조”로 정의한다.  
- 반복 작업 감소와 표준화 효과를 중심으로 커뮤니케이션한다.  

### 9.3 운영 복잡도 리스크  
- 파이프라인 표준화와 산출물 표준화를 통해 운영 난이도를 통제한다.  
- 실패 유형을 분류하고 재실행/재처리 가능한 형태로 운영한다.  

### 9.4 자원 부족 리스크  
- 동시 실행 제한, 배치 처리, 모델 크기 조정으로 자원 사용을 통제한다.

## 10. 성공 지표  

성공 지표는 “측정 가능”해야 하며, Pilot에서 기준선을 확보한다.  

- **문서 초안 작성 시간**  
  - 기준선 대비 30% 이상 단축을 목표로 한다.  
  - 측정 방식: 동일 문서 유형의 작성 착수~초안 완성까지 소요 시간 비교.  

- **Sonar 이슈 triage 리드타임**  
  - CRITICAL/MAJOR 이슈의 “발생→이슈 등록” 리드타임 단축을 목표로 한다.  
  - 측정 방식: Sonar 이슈 생성 시각과 GitLab 이슈 생성 시각 비교.  

- **방치 이슈 비율**  
  - 일정 기간 이상 미처리 이슈 비율 감소를 목표로 한다.  
  - 측정 방식: 라벨/상태 기준의 체류 시간 분석.  

- **문서 표준화 적용률**  
  - 템플릿 기반 산출물 생성·사용 비율 증가를 목표로 한다.  
  - 측정 방식: 표준 섹션/필수 항목 충족률.  

- **E2E 테스트 자동화 적용 범위**  
  - 자동 생성/실행되는 핵심 시나리오 수 증가를 목표로 한다.  
  - 측정 방식: 파이프라인에 등록된 시나리오 수, 실행 성공률, 유지보수 발생률.
