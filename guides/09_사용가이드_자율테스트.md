# 사용 가이드: 자율 E2E 테스트 (Job #7)

## 사전 준비

- Jenkins에 Mac Local Agent(`mac-ui-tester`)가 연결되어 있어야 한다.
- Dify에 Zero-Touch QA용 Chatflow가 구성되어 있어야 한다.
- Jenkins Credentials에 `dify-qa-api-token`이 등록되어 있어야 한다.
- 테스트 대상 웹사이트에 접근 가능해야 한다.

---

## Chat 모드: 자연어로 테스트 실행

가장 빠르고 간편한 방법이다. 자연어로 "무엇을 테스트할지" 지시하면 AI가 즉시 실행한다.

### 실행 방법

1. Jenkins에서 `DSCORE-ZeroTouch-QA` Job을 클릭한다.
2. `Build with Parameters`를 클릭한다.
3. 파라미터를 입력한다:

| 파라미터 | 입력 예시 | 설명 |
| --- | --- | --- |
| `RUN_MODE` | `chat` | 드롭다운에서 `chat` 선택 |
| `TARGET_URL` | `https://www.naver.com` | 테스트 대상 웹사이트 URL |
| `SRS_TEXT` | (아래 예시 참고) | 자연어 테스트 요구사항 |

4. `빌드`를 클릭한다.

### SRS_TEXT 작성 예시

**간단한 검색 테스트:**
```
네이버에 접속해서 검색창에 'DSCORE'를 입력하고 엔터를 누른다.
검색 결과 페이지가 표시되는지 확인한다.
```

**로그인 테스트:**
```
1. 로그인 페이지로 이동한다.
2. 아이디 입력란에 'testuser'를 입력한다.
3. 비밀번호 입력란에 'password123'을 입력한다.
4. 로그인 버튼을 클릭한다.
5. 대시보드 페이지가 표시되는지 확인한다.
```

**쇼핑몰 장바구니 테스트:**
```
1. 메인 페이지에서 첫 번째 상품을 클릭한다.
2. 수량을 2로 변경한다.
3. '장바구니 담기' 버튼을 클릭한다.
4. 장바구니 페이지로 이동한다.
5. 상품이 2개로 표시되는지 확인한다.
```

### 작성 팁

- **구체적으로 쓴다:** "테스트해줘" 보다 "검색창에 DSCORE 입력 후 엔터" 가 낫다.
- **순서대로 나열한다:** 번호를 매겨서 단계별로 작성하면 AI가 더 정확하게 변환한다.
- **확인 조건을 명시한다:** "~가 표시되는지 확인한다"를 마지막에 추가하면 verify 액션이 생성된다.

## Doc 모드: 기획서로 테스트 실행

기획서(PDF/DOCX)에 기재된 요구사항을 기반으로 대량의 테스트 시나리오를 자동 생성한다.

### 실행 방법

1. Jenkins에서 `DSCORE-ZeroTouch-QA` Job을 클릭한다.
2. `Build with Parameters`를 클릭한다.
3. 파라미터를 입력한다:

| 파라미터 | 입력 예시 | 설명 |
| --- | --- | --- |
| `RUN_MODE` | `doc` | 드롭다운에서 `doc` 선택 |
| `TARGET_URL` | `https://my-app.com` | 테스트 대상 URL |
| `DOC_FILE` | (파일 선택) | 기획서 PDF/DOCX 업로드 |

4. `빌드`를 클릭한다.

> `DOC_FILE` 파라미터는 Jenkins의 File Parameter 플러그인이 필요하다. 플러그인이 없으면 `SRS_TEXT`에 기획서 내용을 직접 붙여넣어 Chat 모드로 대체할 수 있다.

## Record 모드: 브라우저 녹화 (로컬 전용)

브라우저를 직접 조작하면 시스템이 동작을 캡처하여 테스트 시나리오로 변환한다. 이 모드는 **맥북 터미널에서 직접 실행**한다 (Jenkins Job이 아님).

### 실행 방법

```bash
cd /Users/<사용자>/Developer/automation/local_qa
source venv/bin/activate
export DIFY_API_KEY="app-xxxxxxxxx"

python3 mac_local_executor.py --mode record --url https://target-app.com
```

### 녹화 과정

1. Chromium 브라우저가 자동으로 열린다.
2. 평소처럼 웹사이트를 조작한다 (클릭, 입력, 선택 등).
3. 클릭/입력할 때마다 대상 요소에 **붉은 테두리**가 표시되고 스크린샷이 캡처된다.
4. 작업이 끝나면 **브라우저 창을 닫는다**.
5. 시스템이 캡처된 데이터를 Dify Vision LLM으로 전송하여 정제된 DSL 시나리오를 생성한다.
6. 결과가 `artifacts/scenario.json`에 저장된다.

## 결과 확인

### Jenkins 아티팩트

빌드 완료 후 해당 빌드 페이지의 `Artifacts` 링크를 클릭하면 다음 파일을 확인할 수 있다:

| 파일 | 확인 내용 |
| --- | --- |
| `scenario.json` | AI가 생성한 테스트 시나리오 (어떤 액션을 수행했는지) |
| `scenario.healed.json` | Self-Healing이 적용된 최종 시나리오 |
| `run_log.jsonl` | 각 스텝의 PASS/FAIL/HEALED 상태 |
| `step_1_pass.png` ~ `step_N_pass.png` | 각 스텝 성공 시 스크린샷 |
| `error_final.png` | 실패 시 마지막 화면 상태 |

### 스크린샷으로 결과 검증

`step_N_pass.png` 파일들을 순서대로 열어보면, 테스트가 실제 화면에서 어떻게 진행되었는지 시각적으로 확인할 수 있다. 예상과 다른 화면이 캡처되었다면 `SRS_TEXT`를 더 구체적으로 수정하여 재실행한다.

### run_log.jsonl 해석

```json
{"step": 1, "action": "navigate", "status": "PASS", "heal_stage": "none"}
{"step": 2, "action": "fill", "status": "PASS", "heal_stage": "none"}
{"step": 3, "action": "click", "status": "HEALED", "heal_stage": "local"}
{"step": 4, "action": "verify", "status": "FAIL", "heal_stage": "none"}
```

- `PASS` — 성공
- `HEALED` — 원래 셀렉터는 실패했지만 Self-Healing으로 복구 후 성공
- `FAIL` — 모든 복구 시도가 실패

## 자주 묻는 질문

**Q: 테스트가 중간에 실패하면 나머지 스텝은 어떻게 되나요?**

Self-Healing(Candidate Search → Dify Healer)을 시도합니다. 2회 재시도 후에도 실패하면 해당 스텝에서 중단되고, 에러 스크린샷이 저장됩니다.

**Q: scenario.healed.json은 어떻게 활용하나요?**

Self-Healing으로 수정된 셀렉터가 포함된 시나리오입니다. UI가 변경된 후에도 healed.json의 셀렉터가 유효하다면, 이를 다음 실행의 기준 시나리오로 재사용할 수 있습니다.

**Q: Chat 모드에서 SRS_TEXT를 영어로 작성해도 되나요?**

네, 한국어와 영어 모두 지원합니다. Dify Planner LLM이 언어를 자동으로 이해하고 DSL로 변환합니다.
