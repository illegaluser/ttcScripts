## 2. 서비스별 역할

| 서비스 | 포트 | 역할 | 데이터 저장 위치 |
| --- | --- | --- | --- |
| **Jenkins** | 8080 | 8개 파이프라인의 실행 엔진. 모든 자동화 작업의 시작점 | `data/jenkins/` |
| **SonarQube** | 9000 | 소스 코드 정적 분석. 버그, 취약점, 코드 스멜을 탐지 | `data/sonarqube/` |
| **GitLab** | 8929 | Git 저장소 호스팅 + 이슈 트래커. Job #5가 이슈를 자동 생성하는 대상 | `data/gitlab/` |
| **Dify** | 80 | AI 워크플로우 설계 플랫폼. Knowledge Base(RAG)와 Chatflow를 제공 | `dify/docker/volumes/` |
| **Langfuse** | 3000 | LLM 호출의 관측성 대시보드. 평가 점수, 지연 시간, 토큰 사용량 추적 | `data/postgres-langfuse/` |
| **Ollama** | 11434 | 로컬 LLM 실행 엔진 (호스트 머신에서 직접 구동) | 호스트 `~/.ollama/` |

## 3. 데이터 흐름

### Phase 2: 지식 관리 흐름

```
[원본 문서]                    [Jenkins Job #1~#4, #6]              [Dify Knowledge Base]
PDF, DOCX, XLSX  ─────────▶  Python 스크립트가 변환  ─────────▶  LLM이 검색 가능한
이미지, 코드, 웹                (Markdown/텍스트)                  벡터 저장소로 보관
```

### Phase 3: 품질 분석 흐름

```
[SonarQube]         [Jenkins Job #5]          [Dify Workflow]       [GitLab]
정적 분석 결과  ──▶  이슈 추출 (API) ──▶  LLM이 원인 진단  ──▶  Issue 자동 생성
(버그/취약점)                               + 수정 방안 제안        (중복 방지 포함)
```

### Phase 4: 자율 E2E 테스트 흐름

```
[사용자 입력]           [Dify Brain]              [Python Executor]         [산출물]
자연어 / 기획서  ──▶  9대 DSL JSON 생성  ──▶  브라우저 자동 실행  ──▶  scenario.json
/ 브라우저 녹화         (Planner LLM)          + Self-Healing            스크린샷, 로그
```

### Phase 5: AI 에이전트 평가 흐름

```
[시험지 CSV]       [Jenkins Job #8]           [DeepEval + Ollama]       [Langfuse]
golden.csv  ──▶  대상 AI에 질문 전송  ──▶  11대 지표 채점  ──▶  점수 대시보드
(질문+기대답변)     (API 또는 웹 UI)          (심판 LLM)              + Jenkins 아티팩트
```

## 4. 네트워크 구성

모든 컨테이너 서비스는 `devops-net`이라는 Docker 가상 네트워크로 연결된다. 이를 통해 서비스 간에 컨테이너 이름(예: `sonarqube`, `gitlab`)으로 직접 통신할 수 있다.

| 통신 경로 | 사용되는 URL | 설명 |
| --- | --- | --- |
| Jenkins → SonarQube | `http://sonarqube:9000` | Job #5에서 정적 분석 결과를 API로 조회 |
| Jenkins → GitLab | `http://gitlab:8929` | Job #5에서 이슈를 API로 생성 |
| Jenkins → Dify | `http://api:5001/v1` | Job #1~#6에서 지식 업로드 및 워크플로우 실행 |
| Jenkins → Langfuse | `http://langfuse-server:3000` | Job #8에서 평가 결과를 전송 |
| Jenkins → Ollama | `http://host.docker.internal:11434` | 로컬 LLM 호출 (호스트 머신 경유) |

Dify는 별도의 Docker Compose로 관리되며, `docker-compose.override.yaml`을 통해 `devops-net`에 연결한다.
