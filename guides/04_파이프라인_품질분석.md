# 파이프라인 소개: 품질 분석 자동화 (Job #5)

## 1. 개요

품질 분석 파이프라인은 **SonarQube가 발견한 코드 문제를 LLM이 진단하고, 그 결과를 GitLab Issue로 자동 등록**하는 자동화 체계이다.

기존에는 SonarQube 분석 결과를 사람이 읽고, 원인을 파악하고, GitLab에 이슈를 수동으로 등록해야 했다. 이 파이프라인은 그 전체 과정을 자동화한다.

---

## 2. 동작 흐름

```
[SonarQube]              [Jenkins Job #5]              [Dify Workflow]           [GitLab]
정적 분석 수행   ──▶    이슈 목록 추출    ──▶    LLM이 원인 진단    ──▶    Issue 자동 등록
(버그/취약점/               (API 호출)           + 수정 방안 제안         (라벨로 중복 방지)
 코드 스멜 발견)
```

### 단계별 상세

**1단계: SonarQube 이슈 추출** (`sonar_issue_exporter.py`)
- SonarQube API를 호출하여 지정된 프로젝트의 이슈를 가져온다.
- 심각도(BLOCKER, CRITICAL, MAJOR 등)와 상태(OPEN, REOPENED 등)로 필터링한다.
- 각 이슈에 대해 해당 코드 라인의 스니펫(앞뒤 5줄)도 함께 추출한다.

**2단계: LLM 진단** (`dify_sonar_issue_analyzer.py`)
- 추출된 이슈를 Dify Workflow에 전달한다.
- Dify의 LLM이 Knowledge Base(Phase 2에서 학습한 프로젝트 지식)를 참조하여 진단한다.
- 진단 내용: 문제의 근본 원인, 영향 범위, 수정 방안을 자연어로 작성한다.

**3단계: GitLab Issue 생성** (`gitlab_issue_creator.py`)
- LLM의 진단 결과를 GitLab Issue 형식으로 정리하여 등록한다.
- **중복 방지:** SonarQube 이슈 키를 GitLab 라벨로 부착하여, 동일 이슈가 이미 등록되어 있으면 건너뛴다.

---

## 3. 주요 특징

### 지능형 진단

단순히 SonarQube의 규칙 설명을 복사하는 것이 아니라, Dify Knowledge Base에 학습된 **프로젝트 코드와 문서를 참조**하여 진단한다. 예를 들어:

- SonarQube 원문: "Method has a complexity of 25 which is greater than 15 authorized."
- LLM 진단: "이 메서드는 인증 로직과 권한 검사를 하나의 함수에서 모두 처리하고 있어 순환 복잡도가 25로 높습니다. `AuthService.validateToken()`과 `PermissionChecker.check()`로 분리하면 복잡도를 각각 10 이하로 줄일 수 있습니다."

### 중복 방지 메커니즘

GitLab Issue를 생성할 때 SonarQube 이슈 키(예: `AYx-...`)를 라벨로 부착한다. 파이프라인을 반복 실행해도 동일한 이슈가 중복 생성되지 않는다.

### 필터링 제어

Jenkins 파라미터로 어떤 이슈를 처리할지 세밀하게 제어할 수 있다:

| 파라미터 | 설명 | 기본값 |
| --- | --- | --- |
| `SONAR_PROJECT_KEY` | SonarQube 프로젝트 식별자 | `dscore-ttc-sample` |
| `GITLAB_PROJECT_PATH` | GitLab 프로젝트 경로 | `root/dscore-ttc-sample` |
| `SEVERITIES` | 처리할 심각도 수준 | `BLOCKER,CRITICAL,MAJOR` |
| `STATUSES` | 처리할 이슈 상태 | `OPEN,REOPENED` |
| `MAX_ISSUES` | 최대 처리 건수 (0=전체) | `10` |

---

## 4. 전제 조건

이 파이프라인을 사용하려면 다음이 선행되어야 한다:

1. SonarQube에 대상 프로젝트가 등록되어 있고, 최소 1회 이상 분석이 수행되어야 한다.
2. GitLab에 대상 프로젝트가 생성되어 있어야 한다.
3. Dify에 품질 분석용 Workflow가 구성되어 있어야 한다.
4. Phase 2(Job #1~#4)로 프로젝트 지식이 학습되어 있으면 진단 품질이 향상된다.
