# 사용 가이드: 품질 분석 파이프라인 (Job #5)

## 사전 준비

- SonarQube에 대상 프로젝트가 등록되어 있고, **최소 1회 이상 정적 분석이 수행**되어야 한다.
- GitLab에 이슈를 등록할 대상 프로젝트가 생성되어 있어야 한다.
- Dify에 품질 분석용 Workflow가 구성되어 있어야 한다.
- Jenkins Credentials에 `sonarqube-token`, `gitlab-access-token`, `dify-knowledge-key`가 등록되어 있어야 한다.

> 사전에 Job #1~#4로 프로젝트 지식을 Dify에 학습시켜두면, LLM의 진단 품질이 크게 향상된다. 프로젝트 코드와 문서를 참조하여 맥락에 맞는 수정 방안을 제안할 수 있기 때문이다.

---

## 실행 방법

1. Jenkins에서 `DSCORE-Quality-Issue-Workflow` Job을 클릭한다.
2. 좌측 메뉴에서 `Build with Parameters`를 클릭한다.
3. 파라미터를 입력한다:

| 파라미터 | 입력 예시 | 설명 |
| --- | --- | --- |
| `SONAR_PROJECT_KEY` | `dscore-ttc-sample` | SonarQube에 등록된 프로젝트 식별자. SonarQube 웹 UI(`http://localhost:9000`) → Projects 메뉴에서 확인 가능 |
| `GITLAB_PROJECT_PATH` | `root/dscore-ttc-sample` | GitLab 프로젝트 경로. GitLab 웹 UI(`http://localhost:8929`)에서 프로젝트 URL의 뒷부분 |
| `SEVERITIES` | `BLOCKER,CRITICAL,MAJOR` | 처리할 이슈의 심각도. 쉼표로 구분. 선택지: `BLOCKER`, `CRITICAL`, `MAJOR`, `MINOR`, `INFO` |
| `STATUSES` | `OPEN,REOPENED` | 처리할 이슈의 상태. 선택지: `OPEN`, `REOPENED`, `CONFIRMED`, `RESOLVED`, `CLOSED` |
| `MAX_ISSUES` | `10` | 한 번에 처리할 최대 이슈 수. 처음에는 5~10으로 시작하여 결과를 확인한 후 늘리는 것을 권장 |

4. `빌드`를 클릭한다.

---

## 실행 결과 확인

### Jenkins Console Output

빌드 완료 후 `Console Output`을 클릭하면 다음 정보를 확인할 수 있다:

```
[SONAR] 이슈 12건 추출 완료 (BLOCKER: 2, CRITICAL: 5, MAJOR: 5)
[DIFY]  이슈 1/10 진단 완료: AYx-abc123 → "NullPointerException 위험..."
[DIFY]  이슈 2/10 진단 완료: AYx-def456 → "SQL Injection 취약점..."
...
[GITLAB] Issue #15 생성 완료: "NullPointerException 위험 - UserService.java:42"
[GITLAB] Issue #16 생성 완료: "SQL Injection 취약점 - QueryBuilder.java:78"
[GITLAB] Issue #17 건너뜀: 이미 존재하는 이슈 (중복 방지)
```

### GitLab에서 이슈 확인

1. GitLab 웹 UI(`http://localhost:8929`)에 접속한다.
2. 해당 프로젝트로 이동한다.
3. 좌측 메뉴에서 `Issues`를 클릭한다.
4. 자동 생성된 이슈를 확인한다. 각 이슈에는 다음이 포함된다:
   - **제목:** SonarQube 이슈의 요약
   - **본문:** LLM이 작성한 문제 원인 분석 + 수정 방안
   - **라벨:** SonarQube 이슈 키 (중복 방지용)

### SonarQube에서 원본 이슈 확인

1. SonarQube 웹 UI(`http://localhost:9000`)에 접속한다.
2. 해당 프로젝트의 `Issues` 탭에서 원본 분석 결과를 확인할 수 있다.

## 활용 팁

### 심각도별 단계적 처리

처음에는 가장 심각한 이슈만 처리하고, 점차 범위를 넓히는 것을 권장한다:

```
1차: SEVERITIES=BLOCKER,CRITICAL / MAX_ISSUES=5
2차: SEVERITIES=MAJOR / MAX_ISSUES=10
3차: SEVERITIES=MINOR / MAX_ISSUES=20
```

### 정기 실행

Jenkins의 `빌드 트리거(Build Triggers)` → `빌드를 주기적으로 실행(Build periodically)`에 cron 표현식을 설정하면 매일 자동으로 분석할 수 있다:

```
# 매일 오전 9시에 실행
H 9 * * *
```

### LLM 진단 품질 개선

LLM의 진단이 너무 일반적이라면:
1. Job #1~#4로 프로젝트 문서와 코드를 더 많이 학습시킨다.
2. Dify Workflow의 LLM 프롬프트를 프로젝트 특성에 맞게 튜닝한다.
