# 파이프라인 소개: AI 에이전트 평가 (Job #8)

## 1. 개요

AI 에이전트 평가 파이프라인은 **외부 AI(챗봇 API, 웹 UI)에 시험 문제를 자동으로 출제하고, 응답을 11대 지표로 채점하는** 자동화 체계이다.

새로운 AI 모델을 도입하거나, 기존 챗봇을 업데이트할 때 "이 AI가 정말 잘 작동하는가?"를 객관적으로 검증할 수 있다. 사람이 일일이 질문하고 답변을 평가하는 대신, 시험지(CSV)를 준비하면 시스템이 자동으로 시험을 치르고 성적표를 발행한다.

---

## 2. 11대 측정 지표

평가는 **4단계 필터링** 구조로 진행된다. 상위 단계에서 실패하면 하위 단계를 건너뛰어 리소스를 절약한다.

### 1단계: 즉시 차단 (Fail-Fast)

| 지표 | 검사 내용 | 도구 |
| --- | --- | --- |
| **Policy Violation** | 주민등록번호, 카드번호, 이메일 등 개인정보가 응답에 노출되는지 검사 | Promptfoo + 정규식 |
| **Format Compliance** | API 응답이 약속된 JSON 형식(예: `answer` 키 포함)을 준수하는지 검사 | jsonschema |

이 단계에서 실패하면 이후 평가를 수행하지 않는다 (비용 절약).

### 2단계: 과업 검사

| 지표 | 검사 내용 | 도구 |
| --- | --- | --- |
| **Task Completion** | 시험지에 명시된 성공 기준을 달성했는지 판정 | 규칙 기반 → DeepEval GEval |

성공 기준 예시: "응답에 '행복상사'가 포함되어야 함", "HTTP 상태 코드가 200이어야 함"

### 3단계: 심층 평가

| 지표 | 검사 내용 | 도구 |
| --- | --- | --- |
| **Answer Relevancy** | 질문의 의도에 맞는 답변인지 (동문서답 여부) | DeepEval + Ollama |
| **Toxicity** | 혐오/차별 발언이 포함되어 있는지 | DeepEval + Ollama |
| **Faithfulness** | 검색된 원문에 근거한 답변인지, AI가 지어낸 말인지 (환각 검사) | DeepEval + Ollama |
| **Contextual Recall** | AI가 답변을 위해 필요한 정보를 올바르게 검색했는지 | DeepEval + Ollama |
| **Contextual Precision** | 검색된 문서에 불필요한 정보(노이즈)가 얼마나 섞여 있는지 | DeepEval + Ollama |

Faithfulness, Contextual Recall, Contextual Precision은 **RAG(검색 증강 생성) API에서만** 측정 가능하다.

### 4단계: 다중 턴 평가

| 지표 | 검사 내용 | 도구 |
| --- | --- | --- |
| **Multi-turn Consistency** | 여러 차례 대화를 나눌 때 일관성과 기억력을 유지하는지 | DeepEval GEval |

### 운영 관제 (상시 기록)

| 지표 | 검사 내용 | 도구 |
| --- | --- | --- |
| **Latency** | 질문 전송부터 답변 수신까지 걸린 시간(밀리초) | Python 타이머 + Langfuse |
| **Token Usage** | API 호출 시 소비된 프롬프트/완성 토큰 수 | API 응답 필드 + Langfuse |

## 3. 평가 대상 유형

| 유형 | 설정값 | 설명 |
| --- | --- | --- |
| **HTTP API** | `TARGET_TYPE=http` | REST API로 통신하는 AI. JSON 요청/응답 |
| **웹 UI 챗봇** | `TARGET_TYPE=ui_chat` | 브라우저에서 접속하는 채팅 UI. Playwright로 조작 |

### 지원하는 심판 모델

채점을 수행하는 심판 LLM은 다음 중 선택할 수 있다:

| 모드 | 설명 |
| --- | --- |
| `local_ollama_wrapper` | 로컬 Ollama 모델을 심판으로 사용 (기본값) |
| `openai_wrapper` | OpenAI API(GPT 계열)를 심판으로 사용 |
| `gemini_wrapper` | Google Gemini API를 심판으로 사용 |
| `direct` | TARGET_URL을 직접 호출 (래퍼 없이) |

## 4. 시험지 (golden.csv) 형식

시험지는 CSV 파일로, 각 행이 하나의 테스트 케이스이다.

| 컬럼 | 필수 | 설명 |
| --- | --- | --- |
| `case_id` | O | 테스트 케이스 고유 ID |
| `input` | O | AI에게 던질 질문 |
| `expected_output` | - | 기대하는 정답 (Answer Relevancy 등에 사용) |
| `success_criteria` | - | 성공 판정 기준 (규칙 기반 또는 자연어) |
| `conversation_id` | - | 다중 턴 대화의 그룹 ID (같은 ID = 같은 대화) |
| `turn_id` | - | 대화 내 순서 (숫자 오름차순) |

**예시:**

```csv
case_id,input,expected_output,success_criteria
basic-1,대한민국의 수도는?,서울입니다,응답에 '서울'이 포함되어야 함
basic-2,1+1은?,2입니다,응답에 '2'가 포함되어야 함
```

**다중 턴 예시:**

```csv
case_id,conversation_id,turn_id,input,expected_output,success_criteria
multi-1,conv-001,1,우리 회사 이름은 '행복상사'야,,
multi-2,conv-001,2,그럼 우리 회사 이름이 뭐야?,행복상사입니다,응답에 '행복상사'가 포함되어야 함
```

## 5. 결과 확인

### Jenkins 아티팩트

빌드 완료 후 Jenkins의 `Artifacts` 탭에서 다음 파일을 확인할 수 있다:

| 파일 | 설명 |
| --- | --- |
| `summary.html` | 시각적 평가 리포트 (브라우저에서 열기) |
| `summary.json` | 프로그래밍 방식으로 처리 가능한 구조화된 결과 |
| `results.xml` | JUnit 형식 테스트 결과 (Jenkins 테스트 트렌드에 연동) |

### Langfuse 대시보드

`http://localhost:3000`에 접속하면 다음을 실시간으로 모니터링할 수 있다:

- 각 테스트 케이스별 점수 추이
- 지표별 합격/불합격 비율
- 응답 지연 시간 분포
- 토큰 사용량 및 비용 추정
